## 1. 引き継ぎフローの実装詳細：仕様からの逸脱・追加判断

### transactionの構成とレースガードの実際

`approveTakeover(uid)`は以下の順で実行する：

1. `seats/{seat}`に対して`.transaction()`を実行。コールバック内で以下いずれかに該当すれば
   `return`（undefined）で中止する：
   - `cur`が存在しない（空席・想定外）
   - `cur.uid !== oldUid`（別の申請が既に承認され、席のUIDが変わっている）
   - `cur.online !== false`（元のプレイヤーが復帰し、オンラインに戻っている）
2. transactionが`committed:true`で成功したら、`privateRecipes/{oldUid}`を読み取り、
   `ref.update()`で以下3パスを**1回のリクエストで原子的に**書き換える：
   `privateRecipes/{newUid}`＝移行データ、`privateRecipes/{oldUid}`＝null、
   `takeoverRequests/{uid}`＝null。

**Realtime Databaseの`.transaction()`は単一パスにしか使えない**ため、「席の奪い合い防止」と
「好みの移行＋後片付け」を1つのtransactionにまとめることはできない。役割を分け、**競合が起きうる
部分（席の奪い合い）だけをtransactionで守り、既に安全に確保した後の処理（好み移行）は複数パスの
`update()`でまとめる**、という設計にした。これは指示書の「transactionで原子的に」という表現を、
RTDBの技術的制約に合わせて解釈し直したものであり、逸脱ではあるが機能的には要求を満たす
（好み移行の直前でホストのプロセスが落ちる、という極めて狭い window は残るが、その場合も
「新プレイヤーが着席済みだが好みが届かない」という表示上の不整合に留まり、秘匿漏洩や
二重着席には至らない）。

### その他の追加判断

- **申請の自動失効（TTL）は実装しなかった**。指示書は「取り下げ」「一定時間無応答」への言及は
  あったが、自動失効までは明示的に求めていなかったため、対症療法の積み増しを避けた（§0-4）。
  ホストが申請の`ts`を見て古いものを目視で却下できることで最小限の対応とした。
- **却下は`takeoverRequests/{uid}`を削除せず`status:"rejected"`に留める**設計にした。削除だと
  申請者側のリスナーが「消えた＝却下」なのか「消えた＝何か別の異常」なのかを区別できないため。
  削除は申請者自身が「別の席で申請し直す」「やめる」を押した時点で自UID書込権限の範囲内で行う。
- ホストの承認処理のレースガード比較対象（`oldUid`）は、ホストのクライアント側購読キャッシュ
  （`ST.seats`）から取得している。UIの表示が多少古くても、`.transaction()`のコールバックは
  常に最新のDB値で再評価されるため、実際の安全性はtransactionの再試行機構が担保する
  （UIの古さは「不要な却下」を稀に招く可能性はあるが、「誤った上書き」は起きない）。

## 2. Firebaseルール差分・コンソール適用手順・リスク評価

### 差分全文

```diff
--- a/firebase/witch_rules.json
+++ b/firebase/witch_rules.json
@@ -108,6 +108,20 @@
           }
         },

+        "takeoverRequests": {
+          ".read": "auth != null && root.child('roomsV2').child($code).child('meta').child('host').val() === auth.uid",
+          "$uid": {
+            ".read": "auth != null && ($uid === auth.uid || root.child('roomsV2').child($code).child('meta').child('host').val() === auth.uid)",
+            ".write": "auth != null && ($uid === auth.uid || root.child('roomsV2').child($code).child('meta').child('host').val() === auth.uid)",
+            ".validate": "newData.hasChildren(['seat','name','ts'])",
+            "seat": { ".validate": "newData.isNumber() && newData.val() % 1 === 0 && newData.val() >= 0 && newData.val() <= 3" },
+            "name": { ".validate": "newData.isString() && newData.val().length <= 24" },
+            "ts": { ".validate": "newData.isNumber()" },
+            "status": { ".validate": "newData.isString() && (newData.val() === 'pending' || newData.val() === 'rejected')" },
+            "$other": { ".validate": false }
+          }
+        },
+
         "$other": { ".validate": false }
       }
     },
```

**既存の`seats`・`privateRecipes`ノードのルールは1文字も変更していない。** 承認時に必要な
「席UIDの差し替え（ホストが任意uidを書ける）」「好みの移行（ホストが`privateRecipes`親を
書ける）」は、**いずれも既存ルールで元々許可されていた**（ホストは`seats`・`privateRecipes`の
親レベルで書込権限を持つため）。今回追加が必要だったのは、新規ノード`takeoverRequests`の
権限定義だけである。

### コンソール適用手順・推奨順序

1. Firebase Console → Realtime Database → ルール タブで、上記差分適用後の
   `firebase/witch_rules.json`全文を貼り付けて公開する。
2. 適用後、`witch.html`のv1.7を`main`へpush（本報告のpush承認は指示書冒頭で取得済み）。

**推奨順序は「ルール適用→push」**（指示書の推奨順序どおり）。理由：

- **逆順（push→ルール未適用）の場合の挙動を実際に確認した**：ルール未適用の状態で
  v1.7のクライアントが`takeoverRequests`へ書き込もうとすると、DB側は**旧ルール
  （`takeoverRequests`ノード自体が未定義＝`$other:{.validate:false}`により拒否）**により
  `permission_denied`で弾かれる。この場合、**申請の送信自体が失敗し`showError()`で
  「引き継ぎ申請の送信に失敗：permission_denied」等が表示される**（無言failはしない。
  `withRetry`のtry/catchが機能する）。**引き継ぎ機能だけが使えず、それ以外の機能
  （通常のプレイ・観戦・ソロ等）は無影響であることを、Rules Playgroundの評価ロジック
  （既存ノードのルールが一切変更されていないこと）から確認した**（実機のRules未適用状態を
  本番Firebase上で意図的に作ることはしていないため、ローカルEmulatorでルール差分適用前の
  旧ルールファイルを使って同様の書込みを試み、`permission_denied`になることを確認する形で
  検証した）。
- 正順（ルール適用→push）なら、この一時的な機能欠落自体が発生しない。

### リスク評価

- 追加ノードのみで、既存の読取・書込制限は一切緩めていないため、**ルール適用単体のリスクは低い**。
- ルール適用とpushの間にタイムラグがあっても、**旧clientはtakeoverRequestsの存在自体を知らない
  ため影響を受けない**（旧v1.6クライアントはtakeoverRequestsを読み書きするコードを持たない）。
- push後にルール未適用のまま放置した場合、**引き継ぎ機能だけが恒久的に使えない状態になる**
  （P0-1の根本原因は解消されないまま）。適用を忘れないよう、本報告に明記する。

## 4. 元UIDが引き継ぎ後に接続してきた場合の実際の挙動

Emulator受入試験で実際に確認した：

- 引き継ぎ承認前の元UID（`tk-player1-uid`）で、引き継ぎ後に`restoreSavedRoom()`が実行されると、
  `findSeatByUid()`が`null`を返す（席のUIDが新UIDに変わっているため）。
- この時、保存していた席番号（`saved.seat`）の現在のUIDが自分と異なることを検知し、
  **「あなたの席は、進行役の承認により別の参加者に引き継がれました。観戦するか、空いている席が
  あれば引き継ぎを申請できます。」という具体的な文言が表示される**ことを確認した
  （旧来の汎用文言「あなたの席がないため再接続できませんでした」ではない）。
- 無言でホーム画面に落ちることはなく、赤バナーで理由が表示される。

## 6. 実装中に気づいた懸念・逸脱・未解決事項

### 未解決・今後の課題

- **申請の自動失効（TTL）が未実装**（§1参照）。放置された申請はホストが手動で却下するまで
  残り続ける。
- **ホストのUID消失はこの導線でも救済できない**（承認者がいなくなるため）。既知の制限として
  引き継ぎ資料§10・CLAUDE.md§7-wに明記した。実装していない。

### 新たに生まれた攻撃面・悪用経路（正直に。楽観的フレーミングなし）

- **申請の乱発によるホストへの通知スパム**：技術的には、悪意あるユーザーが大量の新規UIDで
  同じ部屋へ次々と引き継ぎ申請を送りつけ、ホストの承認パネルを埋め尽くすことが可能。
  レート制限は未実装。実害が出た場合はクライアント側クールダウンやFirebase側のレート制限
  （Cloud Functionsトリガー等）を検討する必要がある。
- **`ts`（申請時刻）の自己申告**：クライアントの`now()`（サーバオフセット込みの近似値）であり
  サーバ側で強制されていない。ホストが「古さ」で判断材料にする際に多少惑わされうるが、
  悪用してもホストの承認判断を歪める程度で、データ漏洩や権限昇格には繋がらない。
- **却下後の申請者による`status`の書き戻し**：ルール上、申請者は自UID子ノードに`.validate`を
  満たす値を書けるため、却下後に`status:"pending"`へ書き戻して同一内容の申請を再送できる。
  これは「再申請」そのものであり意図された挙動の範囲内（ホストが再度却下すればよい）。
  好みの漏洩や不正な席奪取には繋がらない。
- **好みの移行はホスト権限内で完結**しており、申請者や第三者が他人の`privateRecipes`を
  直接読める新しい経路は作っていない（既存の「ホストのみ`privateRecipes`親読取可」を維持）。

### テスト環境固有の制約（実害ではないが正直に記載）

- Emulator受入試験中、**同一ブラウザプロファイルの複数タブでは匿名認証セッション
  （`Persistence.LOCAL`）が共有され、一方のタブでの再サインインがもう一方のタブの
  実効認証状態を巻き込む**ことが判明した（片方のタブがサインアウト/サインインすると、
  もう片方のタブの以降のDB書込みがpermission_deniedになった：ハートビート更新の失敗として
  観測）。この制約のため、ホストの承認クリックをブラウザUIから直接再現する代わりに、
  同一のtransaction/update操作をAdmin SDK経由のスクリプトで実行し、申請者側クライアントが
  正しく反応する（好みが正しく移行され、自動的に着席プレイヤーとして復帰する）ことを確認した。
  **これは本番の実プレイヤー（別デバイス・別ブラウザ）には影響しない、テスト手法固有の制約**
  であり、witch.html本体のコードやルールの問題ではない。
- 却下フロー・「旧UIDの再接続」フローは、上記の制約の影響を受けない形（DBの直接操作＋
  該当タブでの観察）で問題なく実地確認できた。
- 「承認直前に元UIDが復帰する」レースは、コードレビューでレースガード条件
  （`cur.online!==false`での中止）の存在を確認し、静的テストで該当条件式の存在を
  検証したが、**3者同時進行（ホスト・新規申請者・復帰する元プレイヤー）をブラウザで
  ライブ再現することはできなかった**（同一ブラウザでの認証共有制約のため）。
