## 1. Firebase セットアップ（初回のみ・10分）

1. https://console.firebase.google.com/ で無料プロジェクトを新規作成
2. 左メニュー「Realtime Database」を作成（**テストモード**でOK・ロケーションは近い地域）
3. **「Authentication」→「Sign-in method」→「匿名」を有効化**（v1.2以降は必須）
   - 無効のままだと、オンラインの全入口（作る／参加する／観戦する）で「匿名認証に失敗しました」の赤バナーが出る
4. Realtime Database の「ルール」タブに **`firebase/witch_rules.json`** の中身を貼って公開：
   - **公開前セキュリティ完成版（v1.3）**：`roomsV2/{6文字コード}` を使い、共有ノードは認証済みユーザーだけが読む。好みは `privateRecipes/{uid}` に分離し、本人とホスト以外は読めない。
   - `game` は着席者だけ、`reveal` はゲーム終了後のホストだけが書ける。席は「空席への本人着席／自席更新／開始前のホストによる席詰め」に限定する。
   - 旧 `rooms/{4文字}` は最終ルールでは読書きとも拒否される。
   - 簡易設定（非推奨・旧）：`{ "rules": { ".read": true, ".write": true } }` はDB全体が無制限開放になるため使わない。
5. 「プロジェクト設定 → 全般 → マイアプリ」で **ウェブアプリを追加**
6. SDK設定で **「Config」を選択**（npm/CDNではない）。`const firebaseConfig = {...}` の中身が出る
7. その中身を、`witch.html` 冒頭の **`FIREBASE_CONFIG`** に貼る
   - **重要**：変数名は `FIREBASE_CONFIG` のまま。`firebaseConfig` に変えると動かない
8. ファイルを GitHub Pages か Netlify Drop に置く
   - Netlify は **ファイル名を `index.html`** にリネームしてからアップロード
   - 更新時は Netlify の「Deploys」から再アップロード（新規Dropは別URLになる）

未設定のまま開くと、画面に「Firebase未設定」と出る（ローカルで開いても同じ）。
**ソロモードはFirebase不要**で動く（ホーム画面から直接遊べる）。

### 1-1. ローカルEmulator受入試験

本番Firebaseへ接続せず実ゲームHTMLを試験する場合は、リポジトリ直下で `npm run emulators:witch` を実行する。Authentication Emulatorは9099、Realtime Database Emulatorは9150で、クライアント設定と同じ `balance-simulator-firebase` プロジェクトIDを使う。ルール単体試験の `witch-rules-test` とは用途を分ける。別ターミナルからローカルHTTPサーバーを起動し、`http://127.0.0.1:<port>/witch.html?emulator=1` を開く。`file://`では試験しない。

Emulator接続は `?emulator=1` と、hostnameが `localhost` または `127.0.0.1` であることの両方を満たす場合だけ有効になる。接続時は画面右上に `LOCAL EMULATOR` を表示する。本番ドメインへ `?emulator=1` を付けてもEmulatorへは接続しない。

## 2. 遊び方（プレイヤー視点）

ホーム画面に5つの入口がある：

- **🚪 部屋を作る（ホスト）**：名前を入れて「部屋を作る」→ 合言コード（6文字）が出る → 仲間に伝える → 3〜4人揃ったら儀式数・上級モードの有無を選んで「開始」
- **参加する**：名前を入れて合言コードを入力
- **🎭 ひとりで遊ぶ（NPC対戦）**：Firebase不要。人間1人＋NPC2体（3人固定・6儀式）。NPCの性格タイプ
  （世話焼き/欲張り/気分屋）は対戦前から常時表示。本番と同じくチーム合計のみ公開（B方式）。詳細は§12
- **🧪 ソロで試す**：Firebase不要。1人で全席を操作する検証モード。全員の好み・全個人点を常時表示（本番のB方式とは別に、確認用に全部見える）
- **👁 観戦する**：合言コードで席に着かず入室。プレイ中は共有盤面だけを見られ、好み・個人点・発火状況は見えない。ゲーム終了後に `reveal` が作成されると全好み・個人点・名場面を見られる。書き込み権限はない

対戦中は、手番が来たら操作（切り役＝分ける、手番＝坩堝を選ぶ）。それ以外は待機。**無言で**遊ぶのが本体。
右上の **❓ ボタン**でルール説明（用語ツールチップ付き）がいつでも開ける。

## 3. 状態スキーマ（Firebase `roomsV2/{CODE}/`）

- `meta`：`{host:uid, phase:"lobby"|"playing"|"ended"}`
- `seats/{0..3}`：`{uid, name, lastSeen, online}`（生存報告）
- `privateRecipes/{uid}`：`[好みID×3]`（本人とホストだけが読める。通常プレイヤーは常に自分のUID配下だけを購読。上級モードは通常枠2＋秘伝枠1）
- `game`：`{n, rounds, round, cutter, teamTotal, phase, band, cuts, assign, chooseIdx, roundEval, history}`
  - `game.phase`：`cut → choose → await_score → score →`（次の儀式）`cut` または `end`
  - `roundEval`：`{roundScore, omni, rational, pct}` — **個人点(totals)は含まない**（B方式・下記§5参照）
  - `history[]` の各儀式：`{round, cutter, contents, roundScore, omni, rational, pct, cutPct, band, cuts, piles, owners}`
    - `cutPct`：**切り方そのものの質**（後述§7）。個人点は含まない
    - `band`/`cuts`：名場面の再現用に、その儀式の帯と切れ目を保存
- `reveal/{席}`：`game.phase === "end"` 後にホストだけが `privateRecipes` を席番号へ対応させて公開する。終了画面はここから個人点・名場面を再計算する

`roomsV2/{CODE}` の親には `.read` を置かない。`meta` / `seats` / `game` / `reveal` は各ノードで共有読取を許可し、`privateRecipes` はホストだけが親読取、本人だけが自UID子ノード読取を持つ。

## 4. 設計上の要点（前作の教訓の組込み先。教訓#1〜7 = 「実装技術知見」§2）

| 教訓 | 実装 |
|---|---|
| #1 read-modify-write競合 | 全状態変更を **Firebase transaction** で原子的に。手元stateは直書きしない |
| #2 権限チェック | `confirmCut`(切り役のみ)・`pickPile`(手番のみ)・`startGame`(ホストのみ)。クライアント入口＋transactionに加え、Firebaseルールで未着席者の `game` 書込みを拒否 |
| #3 秘匿の構造漏洩 | プレイ中は**自UIDの好みしか購読しない**。他人の好み・発火内訳・**個人点も一切**DBに出さない（下記§5 B方式）。全好みを読めるのは採点・終了公開を担うホストだけ |
| #4 幽霊対策 | 12秒ごとの生存報告 + `onDisconnect` + 30秒タイムアウト（`seatActive`） |
| #5 同期鮮度 | Firebase接続状態 +「最終同期 n秒前」を常時表示 |
| #6 リトライ | 全通信を `withRetry`（1回失敗→0.7秒→再試行）で共通化 |
| #7 エラー可視化 | 握りつぶさず、画面上部の赤バナー（`showError`）に表示 |

**採点の権威（resolver）**：最終選択で `phase=await_score` になると、ホストだけが全好みを一度だけ読んで採点し、transaction で `phase=score` にする。ホスト不在中は「進行役の再接続を待っています」と表示して停止し、再接続時に同じ `phase === "await_score"` ガードから再開する。二重採点はtransactionのphaseガードで防止。

**再接続**：Auth永続性を `LOCAL` に設定し、ゲーム識別子・部屋コード・参加種別・最後の席だけをlocalStorageへ保存する。本人確認に保存UIDは使わず、認証確定後の `auth.uid` と `seats` のUIDを毎回照合する。同じブラウザプロファイル・同じ匿名UIDなら開始後も元の席と名前、ホスト権限、自分の好み購読へ復帰する。別プロファイルやストレージ削除後は別UIDになり、開始済みルームへ新規プレイヤーとして途中参加できない。観戦復帰は共有ノードだけを購読し、`privateRecipes` を購読しない。

## 5. 🔑 逆算対策＝B方式（チーム合計のみ公開）

**背景**：個人点を毎儀式正確に公開すると、盤面と照らして「発火した好みの組み合わせ」が数手で一意に絞られ、6儀式でほぼ好みが公開状態になることがシミュレーションで判明した（点数方式をどう変えても、点を揃えても1点固定にしても効果は薄い。真因は「盤面＋正確な個人点」を毎回見ること自体の情報量）。

**対策**：**個人点はDBに一切保存しない**。公開は `roundEval.roundScore`（チーム合計）・`omni`・`rational`・`pct` のみ。各自の点は、各クライアントが**自分の好み＋自分の坩堝からローカル計算**する（`scorePile(ST.myRecipes, myPile)`）。他人の坩堝は「◯◯が受け取った」とだけ表示（点は出さない）。

**効果**（シミュレーション実測）：3人・6儀式後、観測者が対象の手札候補を絞れる数
- 個人点を全公開（旧方式）：約7通り（ほぼ特定）
- **チーム合計のみ（B方式）：約438通り（読めるが特定不可）**

> 【注記 2026-07-15】上記の約7通り／約438通りという数値は、現行実装（V3プール）では再現しない。
> リバランス前の旧プールに対する値だった可能性が高い。現行実装での再計算値と検証手法は
> `sim/witch_privacy/魔女の調合_秘匿性数値再現_結果.md` を参照。定性的結論（チーム合計のみの公開は
> 個人点全公開より秘匿性が高い）は現行実装でも支持される。

読みの手がかりは「相手がどう切ったか」「どの塊を選んだか」という**行動**に一本化される。これは検証の結果、「発火個数のみ公開」等の中間案よりも効果的だった（個数を出しても選択情報と組み合わさるとほぼ特定できてしまうため）。

**終了後は解除**：`reveal` ノードで全好みが公開されるため、終了画面では個人点・名場面ともに表示してよい（プレイ中のみ隠す設計）。

## 6. 好みプール＝V3（発火率再調整済み）

初期プール（隣接・個数固定2）は、切り役が「タイルを並べ替えられず切るだけ」という制約下では、隣接が運ゲー（達成可能率35.7%止まり）、個数が達成困難（44%止まり）という欠陥があった。V3で全型を「腕で狙えば達成可能」75〜100%に再設計済み。

| 型 | 定義 | 点 | 3人達成可能率 |
|---|---|---|---|
| 🤝 同居 pair | XとYが同じ坩堝 | +4 | 79.7% |
| 🔢 個数 cnt2 | Xを2つ以上、またはYを2つ以上 | +5 | 75.1% |
| 1️⃣ 単体 solo | Xをちょうど1つ | +3 | 90.5% |
| ⛔ 二種忌避 abs2 | XとYを両方入れない | +3 | 100% |
| 🚫 忌避 abs | Xを入れない | +2 | 100% |

18枚・各型を上記比率で構成。NPC冴え（合理）：3人86.2%／4人82.1%。称号閾値（大魔女92/熟達82/一人前70/見習い58/駆け出し）は据え置きで機能。

配布制約 `dealConflicts`：「入れたい素材」と「入れたくない素材」が同一人物に配られないよう保証。

## 7. 上級ルール＝秘伝枠（任意・設計論で穴を検証済み）

**内容**：上級モードでは各自の好みが「通常枠2枚＋🌟秘伝枠1枚」になる。秘伝枠は「Xを2つ以上、かつYを入れない」の複合条件・**+10点**・達成困難（3人34%前後、4人50%前後で6種とも均一）。

```
🔥火竜2つ以上+🐍蛇毒なし／🍄影茸2つ以上+🌙満月草なし／✨星屑2つ以上+❄️霜結晶なし
❄️霜結晶2つ以上+🍄影茸なし／🌙満月草2つ以上+✨星屑なし／🐍蛇毒2つ以上+🔥火竜なし
```

**設計検証**（方法論編の観点で全て確認済み）：
- 上位互換でない：EV≈3.4〜5.0が安全枠EV≈3.2〜3.6と拮抗し、真の二択として機能
- 配布バランス：omni天井は通常とほぼ不変（18.1→18.5）
- 達成率の均一性：単一テンプレート（cnt+abs型）に統一したことで3人・4人とも数pt以内に収束（当初は型混在で26〜63%とばらついていたため統一した）
- 難易度上昇を実証：合理NPC冴え 通常83%台→上級72%（3人）／79%→67%（4人）
- omni評価は高難度枠も自動的に織り込むため、評価式の変更は不要だった

`dealAdvanced(n)` が通常枠と秘伝枠の配布制約（矛盾回避＋フォールバック）を担う。

## 8. 工房の記録（表示切替）と名場面

- **工房の記録**：全儀式のミニ盤面を振り返れる。🔀ボタンで「帯の並び順（切り方・読みの主材料）」⇄「魔女順（A→B→C固定・追いやすさ優先）」を切替可能。魔女順では塊の隣接関係が失われるため、切れ目の光る演出は消している。プレイ中・終了後の両方で見られる（プレイ中は個人点非表示、終了後は個人点も表示）。
- **✨ 語り継がれる調合（名場面）**：終了画面で、各自が調合師だった儀式のうち「切り方の質」が最も高かった1回を紹介する。
  - **切り方の質**＝ `bestForCut(band, cuts, players) ÷ omniMax(band, players)`。その切り方に対して全員が最適に選択した場合のチーム点で測るため、**選ぶ側のミスに影響されない**（切り役の切り方そのものを純粋に評価）。omniで正規化するため局面の難易度にも依存しない。
  - 表示は**物語主体・ベストのみ・数字（%）は出さない**：「沈黙のうちに◯◯の望みを見抜き、〈好み〉を叶えた」という叙述のみ。協力ゲームでの個人比較・戦犯化を避けるための意図的な設計（ワーストは一切出さない）。
  - 「叶った好み」はDBに保存せず、終了時に `reveal` の好みから再計算する（B方式の秘匿を維持したまま実現）。

## 9. シミュレーションでは見えない、人間テストで確かめる層

数値・バランス・評価の健全性はシミュレーションで検証済み。オンラインで**実際に人間と無言で回して初めて分かる**のは：

1. **無力感**：最悪の坩堝を取らされる悔しさが、面白さになっているか
2. **吊るし上げ**：切り役が読み違えたときの空気
3. **無言のもどかしさ**：伝えたいのに伝えられない緊張が、心地よいか苦痛か
4. **他人予想の解読感**：回を重ねて仲間の好みが「読めてくる」実感が、実際に起きるか
5. **秘伝枠の手触り**：達成率35%前後という数字が「悔しいけど狙える」に感じるか「無理ゲー」に感じるか
6. **名場面の空気**：個人のベストを称える表示が、協力ゲームの空気を壊さないか

ここが良ければ設計は成功。ダメなら、儀式数・素材数・好みの型の比率・秘伝枠の点数などを調整する（ロジックはすべてシミュレーションで再検証可能）。

## 10. 既知の制限・今後

- 「もう一度」は部屋を出て作り直し（同一部屋での連戦は未実装）
- **公開前セキュリティ完成版（v1.3）**：匿名認証＋6文字コード＋UID別 `privateRecipes`＋着席者だけの `game` 書込み＋終了後ホストだけの `reveal` を実装。ルール自動テストは `tests/witch_rules.test.mjs`。
- **ホストは全好みを知りうる**：配布・採点・終了公開を担うゲームマスターとして信頼する。ホストからの完全秘匿は対象外。
- **ホスト不在中は採点が停止する**：`await_score` のまま待機し、ホスト再接続後に一度だけ再開する。
- **再接続は同一プロファイル・同一匿名UIDが条件**：別プロファイル、サイトデータ削除後、認証状態を引き継がないブラウザでは新UIDとなる。開始後の新UIDは空席へ入れず、観戦だけが可能。
- **招待された着席プレイヤーによる悪意ある直接操作を完全には防いでいない**：未着席者・観戦者は `game` を書けないが、着席者は共有ゲーム状態を書ける信頼境界が残る。クライアントの手番ガードとtransactionは誤操作・競合防止であり、悪意ある着席者への完全防御ではない。
- 上級モード（秘伝枠）はソロ・オンライン双方に実装済み。難易度・値付けは実プレイのフィードバック次第で調整余地あり

### ★ 最終ルールを先に適用してはいけない

本番v1.2は旧 `rooms/{4文字}` を使うため、先に最終版 `witch_rules.json` を適用すると旧部屋が即座に読書き不能になる。v1.3は `roomsV2/{6文字}` だけを使い、旧部屋との互換性はない。

### カットオーバー順序（この順を守る）

| # | 作業 | 本番への影響 |
|---|---|---|
| 0 | コンソール → Authentication → 匿名ログインが有効か再確認 | 無効ならオンライン入口が停止 |
| 1 | ローカルでFirebase Emulatorを起動し、ルール自動テストと3人＋観戦の多窓完走を実施 | 本番影響なし |
| 2 | 誰も遊んでいない時間帯を確保し、進行中の旧 `rooms/` がないことを確認 | 旧部屋は移行しない |
| 3 | v1.3をpushしてPages反映を確認 | この時点ではコンソールの旧ルール次第で `roomsV2` が拒否されるため、新規対戦は開始しない |
| 4 | Realtime Databaseの「ルール」に最終版 `firebase/witch_rules.json` を貼り、公開 | `roomsV2` が有効化、旧 `rooms` は拒否 |
| 5 | 本番で3人＋観戦のスモークテストを行い、他人の `privateRecipes` と観戦者の書込みが Permission denied になることを確認 | 公開確認 |

旧 `rooms` のデータは削除しなくても最終ルールから到達不能になるが、コンソール上には残る。不要なら移行完了後に人間が削除する。旧4文字コードはv1.3では入力できず、旧キャッシュのv1.2クライアントは最終ルール適用後に Permission denied になる。キャッシュ解除と `?v=1.3` 導線はあるが、進行中の旧卓は救済されない。

**運用メモ:** 匿名アカウントは溜まる。自動クリーンアップ（30日で自動削除）は
Firebase Authentication with Identity Platform へのアップグレードが必要なため、現時点では未設定。

## 12. ソロモード検討（NPC方策計測 v1〜v3 → タイプ開示方式への転換・2026-07）

**v1.5でwitch.html本体に実装済み。** ホーム画面「🎭 ひとりで遊ぶ（NPC対戦）」から遊べる。
人間テスト（プロトタイプ・タイプ表示ON・3ゲーム）を事前登録済みkill/keep条件に照らして判定し、
(a)タイプ活用感3/3・(b)退屈な確信0/3・(c)継続意欲「思う」の3条件すべて成立で**KEEP**（実装フェーズへ移行）
と判定された（判定基準は`sim/witch_solo_npc/事前登録_witch_ソロ人間テスト_killkeep条件.md`）。
理論的な背景（方策既知NPCの透明性、行動/評価ノイズの非対称性など）は
`docs/ボードゲーム設計論_理論編.md` §3-13 を参照。

### 経緯（要約）

プレイヤー1＋NPC2のソロモードで「NPCの手札を解読する楽しさ」が成立するかを、
NPC方策のパラメータ（softmax温度による行動ノイズ）を変えながらシミュレーションで検証した
（v1: 15組、v2: A3基準を置換して12組）。いずれも「早期に特定されすぎない」ことと
「最終的にNPCとして機能する（チーム性能が崩壊しない）」ことを両立できず、
事前登録済みの撤退条件が発動。行動ノイズの代わりに評価値ノイズ（気まぐれカードによる
疑似好みバイアス）を使う方式（v3: 8組）に切り替えたが、こちらも「解読の楽しさ」自体は
成立しなかった（詳細・数値は `sim/witch_solo_npc/` 配下の各結果MD参照）。
一方、「タイプの一貫性に応じてNPCが反応する」という機能的価値は3方策族すべてで頑健に
成立したため、design chatは**「NPCの性格タイプを対戦前から開示する」方式**に転換した。

### NPCタイプ3種の固定（v3計測済み組のみから構成・パラメータ変更禁止）

| タイプ | 表示名 | λ | q | w | τ |
|---|---|---|---|---|---|
| 世話焼き | ミラ | 0.65 | 1 | 0.5 | 0.3 |
| 欲張り | ゾラ | 0.8 | 1 | 0.5 | 0.3 |
| 気分屋 | ルゥ | 0.65 | 2 | 1.0 | 0.3 |

このパラメータ組はv3計測（`sim/witch_solo_npc/魔女の調合_ソロNPC方策_計測結果_v3.md`）で
D（能力下限）・B1（読まれる甲斐）を実測済みの組み合わせに限定している。
**値の変更は計測による保証の範囲外に出るため禁止。**

### 確定済みの設計裁定

- **v1.1裁定**：切り役の意思決定にλ（自己利益/チーム利益の混合）は適用しない
  （チーム期待値のみで評価。協力ゲームの切り役の合理行動として妥当と判断）。
  チーム合計を用いた信念更新は、他プレイヤーの得点分布の畳み込み近似を使う
  （joint厳密解ではないが、相対比較目的では判断を歪めないと確認済み）
- **人間プレイヤーの信念更新モデル**：NPCがプレイヤーの手札を推定する際の尤度計算には、
  「一貫プレイヤー」定義（λ=1.0・気まぐれ無し・τ=0.3）を仮定として流用する
  （design chat承認済み。プレイヤー自身は実際にはこの方策に従わないが、
  NPCの意思決定に必要な近似としての位置づけ）

### プロトタイプの位置づけ（人間テストで役目を終えた・参照専用）

`proto/witch_solo_proto.html` は人間テスト用に作った**使い捨てプロトタイプ**。Firebase不使用・
witch.html本体とは無関係の単一HTMLファイル。NPC2体は**異なるタイプから選出する制約**があり
（同一タイプ2体は不可）、「タイプ表示」トグルで性格情報の表示/非表示を切り替えられる
（人間テストでの比較用。本体には表示トグルは搭載していない＝常時表示）。NPC方策・推定エンジンは
Python版（`sim/witch_solo_npc/`）からJS移植し、固定シード3ケース・6儀式フルの同値性検証で
完全一致を確認済み（`proto/verify_final.log`）。KEEP判定後は下記のwitch.html本体実装に
役目を引き継いだため、**削除はしないが以後の変更は行わない**（参照専用）。詳細は `proto/README.md`。
