## 1. 基盤設計の原則(イベントソーシング・権威・同期)

> **前提(第1作で実証):** React(または素のJS)ベースの**単一HTMLファイル**でシミュレーターを作る手法は、
> 対話ベースで盤面ゲームのデジタルプロトタイプを高速に作るのに有効。ルール変更のたびに
> 実際に手を動かして試せる状態を保てることが、設計判断のスピードに大きく寄与する。


### 基盤設計
- **「アクション=イベントの列」設計(イベントソーシング的)** を最初に固める。
  → ログ・リプレイ・Undo・アニメーション・セーブ・オンライン同期が**すべて同じ土台**に乗る。
- 状態そのものを同期せず、`games/{gameId}/events` にアクションを追記 → 各クライアントが同じイベント列から状態を再生。
- **「イベントを適用して次の状態を返す純粋関数」を独立**させる(オンライン非依存でテスト可/ルール変更が容易)。
- 構成は「ゲームロジック / Firebase同期 / UI描画」を分離(部分修正の指示が明確になり破壊的変更が減る)。

### Firebase特有の要点
- **権威(source of truth)をどこに置くか**を最初に決める。
  - Security Rules で「自分の手番の正当なアクションだけ書ける」よう制限、または
  - **Cloud Functions をサーバー権威**にし、クライアントは申請 → 関数が検証して確定イベントを書く。
- **非公開情報(手札・山札)は最大の落とし穴。** 全状態をクライアントに配ると開発者ツールから丸見え=発見/正体隠匿系が即死。
  → 各プレイヤーには見てよい情報だけ配信。シャッフル結果はサーバー(Functions)で管理。非公開情報が絡むならサーバー権威は事実上必須。

### リアルタイム同期の注意
- onSnapshot は非同期に届くため、演出中に次状態が来て競合しうる。
  → イベントを即反映せず**アニメーションキューに積んで順次処理**。
- 楽観的更新(自分のアクションは即ローカル反映、サーバー確定で整合)でプレイフィールが軽くなるが、**非公開情報が絡む部分では避ける**。

### この構成で作りやすいUI改善(実装しやすさ順)
1. 盤面ハイライト付きログ / 状態の二層表示(通常は要約、ホバーで詳細)
2. インタラクションのプレビュー(ホバーで結果を半透明表示 = イベントの仮適用→破棄)
3. イベント起点のアニメーション(状態diffではなくイベントから発火)
4. 有効手のフィルタリング(「違法手を弾く」までに留め「良い手を勧める」には踏み込まない)

## 2. 共有状態(オンライン対戦)設計の教訓 ― 『針の上の女神たち』(第1作)より

Firebase／`window.storage`のような**シンプルなキーバリュー型の永続ストレージだけを使って、複数端末がリアルタイムに同じゲーム状態を共有する**仕組みを作る過程で得た、実装パターンの知見。技術基盤（Firebase移行そのもの）とは別に、「共有状態の設計」そのものに関する教訓としてまとめる。

---

### 2-1. 「最後に書き込んだクライアントが勝つ」問題（read-modify-write競合）

### 起きたこと
参加者リストのような共有データを更新する処理を、**各クライアントが手元にキャッシュしている（古いかもしれない）データを元に上書き保存**する実装にしていたところ、2人が同時に参加・操作すると、**片方の変更がもう片方の書き込みで消えてしまう**不具合が起きた。

### 教訓・パターン
共有データを変更する処理は、**変更の直前に必ず最新のデータを取得し直してから、それを元に上書きする**。手元の状態（Reactのstateなど）を信用して直接書き込んではいけない。

```
悪い例：ローカルのstateを直接書き込む
  nextList = localPlayers.map(...)
  save(nextList)

良い例：保存直前に必ず最新を取得
  latest = await fetchLatest()
  nextList = latest.map(...)
  save(nextList)
```

この手のバグは、**1人でテストしている時には絶対に再現しない**（同時アクセスが起きないため）。複数端末での同時操作を意図的に再現するテストが必要になる。

---

### 2-2. 「見えなくする」だけでは「操作できなくする」ことにはならない

### 起きたこと
役割ごとに見える情報を制限する実装は早い段階でできていたが、**「その役割の人にしかできないはずの操作」を、実は誰でも実行できてしまう**状態が長く残っていた（カード担当以外でもカードを割り当てられる、サイコロ担当以外でも出目を選べる、等）。UIの見た目上は「自分には関係なさそうな画面」に見えても、ボタンやドラッグ操作自体は機能してしまっていた。

### 教訓・パターン
共有状態を操作する**全ての関数の入り口で、「今この操作をしているのは、正当な権限を持つ人か」を明示的にチェックする**。UIを見せる/隠すだけでなく、**実際に状態を変更する関数自体にガードを入れる**（例：`if(!canActAs("card")) return;` を関数の最初の行に必ず入れる）。

見た目を制限しただけで満足せず、「この機能を持つボタンや要素は、他の全端末から見て本当に無害か」を、実装が増えるたびに横断的に洗い出す作業が必要だった。

---

### 2-3. 秘密情報は「見た目を隠す」だけでなく「構造で漏らさない」

「プロセス検証知見」§9(ルール説明)の原則2-8と関連するが、共有状態システム特有の側面として：**シークレットなデータも、結局は全クライアントに配信されている**（表示を制限しているだけ）ため、UIの些細な作り込みの差（要素のサイズ、有無、配置）から、本来隠したいはずの情報が透けてしまうリスクが常にある。「値を隠す」実装をしたら、**その値に依存する他の見た目の要素が無いか**、逆算的に確認する習慣が必要。

---

### 2-4. 「幽霊」対策は、性善説では機能しない

### 起きたこと
タブを閉じる・電波が切れるなど、明示的な「退出」操作を経ずに離脱するプレイヤーは避けられない。これを放置すると、**役割を持ったまま応答しない参加者が残り続け、他の誰もその役割を選び直せず、ゲームが進行不能になる**リスクがあった。

### 教訓・パターン
- 各クライアントが**一定間隔（今回は12秒）で「生存報告」を書き込み続ける**仕組みを入れる
- 一定時間（今回は30秒）反応がない参加者は、**他の全クライアント側で自動的に「いないもの」として扱う**（表示から外す、役割の必須人数カウントから除外する）
- 明示的な退出処理だけでなく、**タイムアウトによる自動的な離脱検知**を必ずセットで用意する

---

### 2-5. 「同期しているはずなのに、本当に同期しているか分からない不安」への対処

ポーリング方式（数秒おきに取得しにいく）でリアルタイム性を模擬する場合、**ユーザーからは「今の状態が最新なのか、通信が止まっているだけなのか」が区別できない**。「同期済み」という表示だけでは不十分で、**「最終更新：n秒前」のような、鮮度が経過時間で分かる表示**を足すことで、不安をかなり軽減できた。

---

### 2-6. 一時的な通信エラーは、自動リトライで多くが解決する

Firebase・Claudeストレージいずれでも、**プラットフォーム側の一時的な不調によるエラーは珍しくない**。全ての通信処理を、「1回失敗したら少し待って1回だけ再試行する」という薄いラッパーで包んだだけで、体感の安定性がかなり向上した。エラーハンドリングを個別の処理ごとに書くのではなく、**通信処理そのものを共通化した上で、リトライを1箇所に実装する**のが効率的だった。

```
withRetry(fn, retries=1, delayMs=700) {
  try { return await fn() }
  catch(e) {
    if (retries<=0) throw e
    await sleep(delayMs)
    return withRetry(fn, retries-1, delayMs)
  }
}
```

---

### 2-7. エラーを握りつぶさない

`catch(e){}` のように**エラーを無言で握りつぶす実装は、「原因不明の無反応」という、最もデバッグしづらい不具合を生む**。今回、「ボタンを押しても何も起きない」という報告の実態は、`window.storage` が使えない環境で例外が発生し、それが握りつぶされていただけだった。**catchした例外は、最低限ユーザーに見える形で表示する**（トーストやエラーメッセージ欄など）。これだけで「原因不明」の状態を大幅に減らせる。

---

### 2-8. まとめ（次回への適用チェックリスト）

1. 共有データの更新は、必ず「直前に最新を取得→それを元に更新」の順で行う
2. 状態を変更する全ての関数に、権限チェックを入れる（UIを隠すだけでは不十分）
3. 秘密情報は、値だけでなく「見た目の構造」からの漏洩も確認する
4. 生存報告＋タイムアウトによる、自動的な離脱検知を用意する
5. 同期の鮮度（最終更新からの経過時間）をユーザーに見せる
6. 通信処理は共通化し、リトライを1箇所に実装する
7. エラーは握りつぶさず、必ずユーザーに見える形で表示する

## 3. Firebase・秘匿・状態管理の追加知見

> **【理論編との接続】** 秘匿情報の実装で繰り返し問題になる「意図しない漏洩」(見た目のサイズ・
> メーターの位置・処理順序・手札枚数)は、理論編・第4部§3-8 **「意図しない影」** として一般化されている。
> **秘匿すべき情報は「送らない」。「送るが隠す」は秘匿ではない。**

### 3-1. Firebase 実装の追加知見(第3作『魔女の調合』・§2の7教訓に加えて)

### 秘匿情報は「UIで隠す」だけでは不十分
UIで他人の好みを描画しなくても、**DBに保存されていれば開発者ツールで見える**。
本当の秘匿には：
- **そもそもDBに書かない**（今回：個人点をDBに保存せず、各クライアントがローカル計算）
- または匿名認証＋セキュリティルールで読取制限

> **設計原則：秘匿すべき情報は「送らない」。「送るが隠す」は秘匿ではない。**

### 派生データは「終了時に再計算」で秘匿を保てる
「その儀式で叶った好み」は、プレイ中に保存すると逆算材料になる。
→ **保存せず、終了時（好み公開後）に再計算する**。
これで秘匿を保ったまま、終了画面での豊かな振り返り（名場面・個人点）が実現できた。

> **パターン：秘匿情報から導かれる派生データは、秘匿解除後に再計算する。**

### Firebase のセキュリティルール警告
`{ "rules": { ".read": true, ".write": true } }` は「DB全体が誰でも読み書き可能」という警告が来る。
**最小対処**（コード改修なし）：
```json
{
  "rules": {
    ".read": false, ".write": false,
    "rooms": {
      "$code": {
        ".read": true, ".write": true,
        "meta": { "host": {".validate": "newData.isString() && newData.val().length <= 40"}, ... },
        "seats": { "$seat": { ".validate": "$seat.matches(/^[0-3]$/)", ... } },
        "$other": { ".validate": false }
      }
    }
  }
}
```
- ルート直下を false にし、`rooms/{コード}` 配下だけ許可 → DB全体の無制限開放を塞ぐ
- 形と文字数を検証 → ゴミ・巨大データを拒否
- **ただし認証を使わないので「誰でも読み書き可能」自体は残り、警告は消えない可能性がある**（正直に伝えるべき）
- 本格対処には匿名認証（Anonymous Auth）が必要（下記）。ただし「uid ベースの読取制限」まで踏み込めるかは構造次第

### 公開向け対処＝秘匿情報はUID別パスへ分ける（第3作 v1.3 で実施）

v1.2では匿名認証、席所有権、ホストだけの配布までは実装したが、`rooms/$code` 親の `.read` により `recipes` 全体を読めた。匿名認証はUIDを識別するだけで、部屋参加者を自動的に信頼する仕組みではない。不特定多数が訪れる公開（記事経由など）には、データ構造と読取境界の再設計が必要だった。

> **秘匿情報は席番号ではなく認証UIDをキーにし、本人の子パスだけを購読させる。親ノードに共有 `.read` を置かない。**

- `privateRecipes/{uid}` は本人とホストだけが読む。通常プレイヤーは自UIDのパスだけを購読する
- ホスト以外のresolver代行を廃止し、ホスト不在中は採点を待機する。可用性より他プレイヤーからの秘匿を優先する明示的な交換条件
- 観戦者は `meta` / `seats` / `game` / `reveal` だけを購読し、プレイ中の好み・個人点・発火状況を受け取らない
- 終了後はホストが `privateRecipes` を席番号へ対応させた `reveal/{seat}` を作り、従来の答え合わせ表示を復元する
- 合言コードは6文字へ伸ばして推測耐性を上げるが、秘匿の主防御はコードではなくUID単位のルールである

実装上の注意（同じ罠を踏まないために）：
- **匿名認証の uid はブラウザ側で永続化される**ため、localStorage の乱数 uid を置き換えても再入室・席復帰は保てる。逆に旧 uid を残すと認証と紐づかない uid が混ざるので、参照箇所を全て auth.uid に一本化する
- 「誰でも進行できる」設計（§4.9-2：進行役切断で卓を止めない）とルールは衝突しやすい。**進行系の書き込み（phase 遷移）をホスト限定にすると、非ホストが最終ラウンドを閉じる通常フローが壊れる** ―― 「着席プレイヤー（いずれかの席の uid が auth.uid に一致）なら可」が落とし所
- transaction はノード全体を書き戻すため、深い `.validate` と衝突しやすい。実際の書込み形をEmulatorテストで通し、余分なフィールドと範囲外値は明示的に拒否する
- ホスト権限が要る操作（席詰め・手札配布）は「ホスト切断中は開始できない」という妥協を生む。許容できるか設計側に確認してから縛る

### Emulatorと匿名UID再接続は一組で検証する

- 実ゲームHTMLのEmulator接続は、明示クエリとlocalhost系hostnameの両方でガードする。本番ドメインでクエリだけを付けてもローカルへ接続させない
- compat SDKでは初期化直後、最初の認証監視・サインイン・DB操作より前にAuth EmulatorとDatabase Emulatorへ接続する。画面にもローカル接続中であることを明示する
- Auth永続性を `LOCAL` に設定して初期認証状態を待ち、`currentUser` が無い場合だけ匿名サインインする。ページ読込みごとの無条件サインインはUID復帰を壊す
- localStorageへUIDを本人証明として保存しない。部屋コード・参加種別・最後の席を復帰の手掛かりにし、本人判定は必ず現在の `auth.uid` とDB上の席UIDで行う
- 同じUIDの既存席は開始後も再利用するが、新UIDを途中の空席へ入れない。ホストが `await_score` へ戻った場合もtransactionのphase/roundガードを通して一度だけ採点する
- 複数窓試験は認証保存領域を分離し、切断復帰だけは同じブラウザコンテキストを再利用する。観戦コンテキストは復帰時も秘匿パスを購読しない

### ホスティング先の選択（実務）
- **Netlify Drop**：手軽だが、**再アップロードで別URLになる**＋デプロイ回数の無料枠に上限（超えるとブロックされる）
- **GitHub Pages**：**URLが固定**（更新しても同じURL）、デプロイ回数の制限が実質なし、更新履歴が残る → **反復開発に最適**
  - ファイル名を `index.html` にする必要がある
  - Public リポジトリが必要（Firebase の apiKey はクライアント公開前提なので問題ない）

### 3-2. 第2作『握るか、逃げるか』で新たに得られた知見

- **Claudeアーティファクトの`window.storage`は、公開後に更新されない・真っ白になるという既知の不具合が再発した**。本ファイル§4に記録されていた症状と完全に一致し、**過去の記録が実際に役立った**。この移行手順は今後も再利用価値が高い
- **秘匿情報（ほころびトークン）は`window.storage`の非共有領域（`shared: false`）に置くことで、多人数間の秘密情報を自然に実装できる**：各プレイヤーのブラウザ/アカウントに紐づくローカル領域に置くだけで、他プレイヤーからは物理的に見えない設計になる。検証イベントなど「発覚した結果だけを公開領域に書き込む」形にすれば、秘匿→公開の流れも綺麗に実装できる
- **JSON経由の状態複製（`JSON.parse(JSON.stringify(state))`）は関数を保持できない**：NPCの戦略を関数ごとオブジェクトに持たせていたところ、複製のたびに関数が消えるバグが発生した。**戦略やロジックは「文字列キー」で持たせ、複製されないマップ（定数）から都度引き直す設計にする**のが安全
- **ラウンド解決（誰がどの処理を担当するか）は、「最新状態を再取得→未処理なら処理→書き込む」というガード付きの楽観的ロックで十分機能する**：完全な排他制御がなくても、小規模な友人間プレイでは実用上問題にならなかった

## 4. アーティファクト → Firebase 移行リファレンス

Claudeアーティファクト(`window.storage`)で作ったアプリを、外部ホスティング+Firebaseの
スタンドアロン版に移行する実践済み手順。第1作で確立し、第2作で同じ症状が再発した際に
**この記録がそのまま役立った**(再利用価値が実証済み)。

Claudeのアーティファクト上で動くWebアプリ（ゲームなど）が、オンラインでの複数人同期機能（部屋を作って共有する、など）を必要とする場合の、実践済みの移行手順。`window.storage`という同じ形のAPIをそのまま使えるようにする「差し替え層（シム）」を使うことで、**アプリ本体のロジックには一切手を加えずに**、通信の土台だけをFirebaseに置き換えられる。

---

### 4-1. この移行が必要になるサイン（症状）

以下のいずれかが起きたら、この移行を検討する：

- ブラウザのコンソールに `Failed to execute 'postMessage' on 'DOMWindow': The target origin provided ('https://www.claudeusercontent.com') does not match the recipient window's origin` というエラーが出て、公開済みのアーティファクトが真っ白になる、または更新されない
- `window.storage` の操作で `Storage get failed: Internal server error while processing action` のようなエラーが出る
- 参加者から「Claudeへのログインを求められた」という報告がある（永続ストレージを使う操作には、閲覧専用の公開アーティファクトであっても認証が必要になる場合がある）
- 新規に公開しようとすると「このコンテンツは既に共有されています」と表示される

これらは基本的に**コード側の不具合ではなく、Anthropicのプラットフォーム側の問題**であることが多い。自分のコードを疑う前に、まずこの移行で解決するかを試す価値がある。

---

### 4-2. 移行の考え方

1. アプリのロジックは `window.storage.get/set/delete/list` という4つのメソッドだけを通信に使うように統一しておく（すでにそうなっているなら、この後の作業は差し替え層の追加だけで完結する）
2. Claudeのアーティファクトとしてではなく、**外部の無料ホスティング（GitHub Pages、Netlify Dropなど）に置く、独立したスタンドアロンHTMLファイルにする**
3. `window.storage` と全く同じ形のAPIを、Firebase Realtime Database（複数人で共有するデータ）＋ ブラウザの `localStorage`（個人のデータ、例えば「自分のID」など）で実装した「差し替え層」を追加する
4. アカウント登録・ログインは一切不要。リンクを開くだけで誰でも参加できる状態になる

---

### 4-3. Firebaseセットアップ手順

1. https://console.firebase.google.com/ にアクセスし、無料プロジェクトを新規作成
2. 左メニューの「Realtime Database」を作成（テストモードでOK。ロケーションは近い地域を選ぶ）
3. 「ルール」タブを以下のように設定して公開（**身内の短期テスト用の簡易設定**。長期運用や機密情報を扱う場合は要見直し）：
   ```json
   {
     "rules": {
       ".read": true,
       ".write": true
     }
   }
   ```
4. 「プロジェクト設定 → 全般 → マイアプリ」で **ウェブアプリを追加**する
5. SDKの設定画面が出たら、**「Config」を選択**する（npm・CDNではなく）。`import`文などが混ざらない、シンプルな `const firebaseConfig = {...}` の形だけが表示される
6. 表示された設定オブジェクトの中身を、下記テンプレートの `FIREBASE_CONFIG` にそのまま貼り付ける（**変数名を `firebaseConfig` のまま貼ってしまうと動かないので要注意。必ず `FIREBASE_CONFIG` に統一する**）
7. 完成したHTMLファイルを、GitHub PagesかNetlify Dropにアップロードする
   - Netlifyの場合、**ファイル名を `index.html` にリネームしてからアップロード**しないと「ページが見つかりません」となる
   - 既存サイトを更新する場合は、Netlify管理画面の「Deploys」タブから再アップロードする（新しくNetlify Dropに置くと、別サイト・別URLになってしまうので注意）

---

### 4-4. 差し替え層（シム）の実装テンプレート

以下をHTMLファイルの `<head>` 内、アプリ本体のスクリプトより**前**に挿入する。

```html
<script src="https://www.gstatic.com/firebasejs/10.12.2/firebase-app-compat.js"></script>
<script src="https://www.gstatic.com/firebasejs/10.12.2/firebase-database-compat.js"></script>
<script>
  // ▼▼▼ ここに、Firebaseコンソールで発行された設定をそのまま貼り付ける（Configタブの内容） ▼▼▼
  const FIREBASE_CONFIG = {
    apiKey: "YOUR_API_KEY",
    authDomain: "YOUR_PROJECT.firebaseapp.com",
    databaseURL: "https://YOUR_PROJECT-default-rtdb.firebaseio.com",
    projectId: "YOUR_PROJECT",
    storageBucket: "YOUR_PROJECT.appspot.com",
    messagingSenderId: "YOUR_SENDER_ID",
    appId: "YOUR_APP_ID"
  };
  // ▲▲▲ ここまで ▲▲▲

  firebase.initializeApp(FIREBASE_CONFIG);
  const _db = firebase.database();

  // window.storage と同じ形のAPIを、Firebase Realtime Database（共有データ）と
  // localStorage（個人データ）で実装するシム。アプリ本体のコードは一切変更不要。
  window.storage = {
    async get(key, shared){
      if(!shared){
        const v = window.localStorage.getItem("bw_"+key);
        return v===null ? null : {key, value:v, shared:false};
      }
      const snap = await _db.ref("shared/"+key).get();
      if(!snap.exists()) return null;
      return {key, value: snap.val(), shared:true};
    },
    async set(key, value, shared){
      if(!shared){
        window.localStorage.setItem("bw_"+key, value);
        return {key, value, shared:false};
      }
      await _db.ref("shared/"+key).set(value);
      return {key, value, shared:true};
    },
    async delete(key, shared){
      if(!shared){
        window.localStorage.removeItem("bw_"+key);
        return {key, deleted:true, shared:false};
      }
      await _db.ref("shared/"+key).remove();
      return {key, deleted:true, shared:true};
    },
    async list(prefix, shared){
      if(!shared){
        const keys = [];
        for(let i=0;i<window.localStorage.length;i++){
          const k = window.localStorage.key(i);
          if(k.startsWith("bw_")){
            const realKey = k.slice(3);
            if(!prefix || realKey.startsWith(prefix)) keys.push(realKey);
          }
        }
        return {keys, prefix, shared:false};
      }
      const snap = await _db.ref("shared").get();
      if(!snap.exists()) return {keys:[], prefix, shared:true};
      const allKeys = Object.keys(snap.val());
      const keys = allKeys.filter(k=>!prefix || k.startsWith(prefix));
      return {keys, prefix, shared:true};
    }
  };
</script>
```

### この実装のポイント

- `shared=true` で呼ばれた場合はFirebase（全員で共有）、`shared=false`（省略時含む）の場合はlocalStorage（その端末だけの個人データ）に振り分けている
- キーに `.` `#` `$` `[` `]` `/` を含む場合はFirebase側でエラーになる（今回のアプリでは `room:CODE:state` のようなコロン区切りのキーのみ使っていたため問題なかった）。もし将来的にこれらの文字を含むキーを使うなら、サニタイズ処理を追加すること
- 通信失敗時の自動リトライは、この層ではなく**アプリ側のロジックで実装するのがおすすめ**（1回失敗したら0.7秒待って再試行、など。プラットフォーム側の一時的な不調を吸収できる）

### 4-6. 運用上の注意点

## 4.5. 静的ホスティング運用の罠（GitHub Pages / Netlify）

第4作『影の競り』の公開時に実際に踏んだ罠。**どれもコードのバグではなく、運用・環境の問題**であり、
「自分のコードを疑う前に、ここを確認する」チェックリストとして残す。

### 4.5-1. 【最重要】ブラウザキャッシュで「更新したのに反映されない」

**症状:** ファイルを更新してアップロードしたのに、URLを開くと古いまま。
**シークレットウィンドウでは新しい版が見える**（＝ファイルは正しく更新されている）。

**その場しのぎ:** スーパーリロード（Windows: `Ctrl+Shift+R` / Mac: `Cmd+Shift+R`）

> **だが、これは根本解決にならない。**
> **仲間に配ったURLでも同じことが起きる。「更新したよ」と伝えても、相手は古い版を開いたまま。**
> **そして毎回「スーパーリロードして」と言わなければならない。**

**構造的対処①：キャッシュ無効化メタ**（全HTMLの `<head>` に）

```html
<meta http-equiv="Cache-Control" content="no-cache, no-store, must-revalidate">
<meta http-equiv="Pragma" content="no-cache">
<meta http-equiv="Expires" content="0">
```

単一HTMLファイル（数十KB〜100KB）なら、毎回読み直しても体感差はない。**開発中はこれが正解。**

### 4.5-2. 【オンライン対戦では致命的】版の食い違いが「原因不明の不具合」になる

**キャッシュ対処だけでは、"既に古い版を開いている仲間" を救えない。**

> **あなたが新版、相手が旧版 → 通信の形（スキーマ）が食い違う → 原因不明のバグ。**
> **これは最もデバッグしづらい不具合の一つ。**

**構造的対処②：版番号をDBに記録し、食い違いを自動検出する**

```js
const VER = "1.0";                       // ★更新のたびに上げる

// 部屋を作るとき
meta: { ..., ver: VER }

// 参加者側の描画で
if (DB.meta.ver && DB.meta.ver !== VER) {
  showBanner(`⚠️ 版が食い違っています ―― この卓は v${DB.meta.ver}、あなたのページは v${VER}。
    ページを再読み込みしてください（Ctrl+Shift+R）。`);
}
```

**さらに、画面の隅に版番号を常時表示**しておくと、口頭でも食い違いを即座に確認できる。

> **原則：オンライン対戦を作るなら、"版番号をDBに書いて突き合わせる" を最初から入れる。**
> **後から入れると、その時点で古い版を持っている人には警告が出せない（＝入れ得ない）。**

### 4.5-3. GitHub Pages が `index.html` を無視して README を表示する

**症状:** `https://user.github.io/repo/` を開くと、`index.html` ではなく `README.md` が表示される。

**原因:** GitHub Pages が **Jekyll** として動いている。

**対処:** リポジトリのルートに **`.nojekyll`**（先頭にドット・拡張子なし・**中身は空**）を置く。
素のHTMLサイトに Jekyll は不要なので、無効化して構わない。1〜2分で再ビルドされる。

**併せて確認:** `Settings → Pages` の **Source = `Deploy from a branch`** / **Branch = `main` / `(root)`**。

### 4.5-4. `index.html` をリネームしてはいけない

`index.html` は「そのフォルダを開いたときに暗黙で表示されるファイル」という**特別な名前**。
リネームすると **`https://user.github.io/repo/` が 404 になり、配布済みのリンク・ブックマークが全部切れる。**

**正しい手順（一瞬もリンクを切らない）:**
1. 今の `index.html` を**コピーして `witch.html` として追加**（この時点で両方が存在＝安全）
2. `witch.html` が動くことを確認
3. **`index.html` の中身を、ゲーム一覧のトップページに置き換える**（Edit → 全選択 → ペースト）
4. 新作を `seri.html` として追加

**結果:** `index.html` = 入口ページ / `witch.html` / `seri.html` = 各ゲーム。
**古いURLを踏んだ人も、一覧から辿り着ける。何も壊れない。**

### 4.5-5. 【前提】Firebase プロジェクトは、作品ごとに分ける

作品ごとにルート名やコード長を分けても、**同じFirebaseプロジェクトではセキュリティルールと障害範囲を共有する**。『魔女の調合』v1.3は `roomsV2/{6文字コード}`、『影の競り』は別プロジェクトを使う。DBを分ける原則は維持する。

- **推奨：作品ごとに新規プロジェクトを作る**（無料枠で複数作れる。作業は `FIREBASE_CONFIG` の貼り替えだけ）
- **⚠️「マイアプリ → アプリを追加」では解決しない。**アプリを追加しても `databaseURL` は同じ＝DBは共有のまま。
  **分けたいのはDBであり、DBはプロジェクトに紐づく。**
- 確認方法：`FIREBASE_CONFIG` の **`databaseURL` が作品ごとに違うこと**

### 4.5-7. ホスティングの選択が、開発速度を左右する

## 4.9. 統合・リファクタで踏んだ落とし穴（第3作『魔女の調合』）

**「統合したつもりで漏れていた」「動いているように見えて壊れていた」類の失敗の記録。**
**どれもプレイヤーからのフィードバックで発覚した。自分では気づけなかったのが共通点。**

### 4.9-1. 「統合した」は「全機能が移った」を意味しない ― 移植漏れ

ローカル検証版 → オンライン版へ統合したとき、**複数の機能が静かに消えていた**（ルール説明ページ全体／トップのSVGイラスト／上級モードのオンライン実装）。

**なぜ気づけなかったか:**
- 統合作業では「ロジックが動くか」に注意が向き、**装飾・補助機能の有無をチェックしない**
- ソロには実装したので「実装した」と記憶が上書きされ、**オンライン側の配線を忘れた**
- **構文チェックは通る**（関数が存在しないのではなく、**呼ばれていないだけ**）

**対策：統合時に「機能インベントリ」を作って照合する**

```
[ ] ゲームロジック（採点・配布・評価）
[ ] ルール説明・ヘルプ・ツールチップ
[ ] イラスト・装飾・アニメーション
[ ] 効果音
[ ] 各モード（対戦/ソロ/観戦）すべてで、上記が動くか
[ ] 新機能を追加したら、全モードに配線したか  ← ★最も漏れる
```

> **教訓：新機能を片方のモードに実装したら、その場で全モードへの配線を確認する。「あとで入れる」は「入れない」になる。**

**機械的な検出法:**
```bash
grep -c 'function dealAdvanced' file.html   # → 1（定義あり）
grep -c 'dealAdvanced(' file.html           # → 1（定義のみ＝呼び出しゼロ！）
```
**定義数と呼び出し数を比較すると、配線漏れが機械的に見つかる。**

### 4.9-2. 【原則】権限は「不正を防ぐため」に付ける。「進行を独占するため」ではない

当初「次のラウンドへ」をホストのみ押せる設計にしていた（権限管理の定石に従った）。**これは悪手だった:**
- **ホストが離席・切断すると、全員が待たされて進行不能**
- 進行を独占する理由がそもそも無い
- 「早く押して」と催促する空気が生まれる（**奉行問題の温床**）

| | 権限が必要か |
|---|---|
| **他人になりすます操作**（他人の手番で選ぶ、他人の秘匿情報を書く） | **必要** |
| **誰がやっても同じ結果になり、秘匿情報を必要としない操作**（次へ進む） | **不要** |
| **他人の秘匿情報を読む必要がある操作**（クライアント採点） | **信頼された役割に限定** |

**対策：進行は誰でもできるようにし、二重実行は transaction のガードで防ぐ。**

```javascript
async function nextRound(){
  // 権限チェックなし。誰でも押せる
  await db.ref(path).transaction(cur => {
    if(!cur || cur.phase !== "score") return;   // ★これで二重実行は防がれる
    return nextState;
  });
}
```

ただし『魔女の調合』v1.3の採点は全員の `privateRecipes` を読む必要があるため例外で、ゲームマスターとして信頼するホストだけが実行する。非ホストへresolverを代行させると好みの読取権限を全着席者へ広げることになる。ホスト不在中は `await_score` のまま「進行役の再接続を待っています」と表示し、再接続後にtransactionのphaseガードで一度だけ再開する。

### 4.9-3. 観戦モードは「作る」より「何を見せるか」が本体

観戦モードを実装したが、最初は**盤面しか出しておらず、実質使い物にならなかった。**

> **原則：観戦モードの価値は「全部見える」ことではなく、「異常が一目で分かる」こと。**

| 観戦者が本当に見たいもの | なぜ要るか |
|---|---|
| **各席の接続状態**（在席/切断・最終生存報告からの秒数） | **★必須。**「止まってる」のか「切断してる」のか判別できないと、何も分からない |
| **今誰の番か**（フェーズ＋何番目か） | 進行が正常か、固まっているかの判断 |
| **獲得・確保の状況** | 進行状況をリアルタイムで追える |
| **共有盤面とチーム結果** | 公開運用ではプレイ中の秘匿情報・個人点・発火状況を観戦者へ送らない |
| **履歴** | 「さっき何が起きたか」の遡り |
| **接続鮮度**（最終同期n秒前） | 観戦者自身のデータが古くないかの確認 |

> **接続状態がないと「バグで止まった」のか「誰かが切断した」のか区別できず、デバッグにならない。**

### 4.9-4. リファクタで壊れる典型パターン（3種）

**パターン1：関数の抽出時に、変数宣言を巻き込んで消す**
置換のアンカーを誤り、`let localCuts = []` ごと消滅。**ReferenceError にならず、暗黙のグローバルとして動くこともあり、発見が遅れる。**
→ **対策：置換後に `grep -c 'let localCuts'` で宣言数を確認する（1であるべき）。**

**パターン2：状態遷移の初期化を1つ忘れる**
```javascript
function nextRound(){
  round++; cutter = (cutter+1) % n; band = makeBand(n);
  cuts = null; assign = {}; chooseIdx = 0;
  // phase = "cut";  ← ★これを忘れて、儀式が進まないバグに
}
```
→ **発見方法：ヘッドレスで通しゲームを回し、「N ラウンド完走するか」を自動検証する。**

**パターン3：型を追加したのに、一部の分岐で未対応**
新しい型を追加し、採点関数には足したが、参照関数が旧プールしか見ておらず `undefined` に。
→ **対策：型を追加したら、その型に触る全関数をリストアップして潰す。**

### 4.9-5. 【原則】テストしている成果物が、出荷する成果物でなければならない

「witch.html を正規表現で加工して witch_stub.html を作り、それでテストする」という手法を採ろうとした。
これは §4.9-1「統合したつもりで漏れていた」と**同じ構造の罠**である。

> **正規表現の当たり方が変われば、テスト対象は静かにズレる。**
> **加工版が通っても、出荷版が通った保証はどこにもない。**

**正しい解:** 多窓テストは、**本番と同じ認証経路のまま**「通常ウィンドウ／シークレットウィンドウ／別ブラウザ」で行う。
ブラウザコンテキストごとにストレージが分離されるため、匿名認証が自然に別 uid を発行する。
加工も分岐も不要で、**出荷する経路そのものをテストできる。**

| 窓 | 匿名 uid |
|---|---|
| 通常ウィンドウ | uid-A |
| シークレットウィンドウ | uid-B（ストレージ分離） |
| 別ブラウザ（Edge 等） | uid-C |

## 5. UI/UX設計の原則

### 5-1. 原則:判断の「補助」は足すほど良い、「肩代わり」は楽しさを殺す

判断の**補助**は足すほど良い。判断の**肩代わり**(最善手サジェスト・自動プレイ)は楽しさを殺す。この境界が最重要。

- **履歴・ログ** — 直前アクションを盤面ハイライトで一瞬表示。捨て札もいつでも確認可。ダウンタイム後の復帰を助ける。
- **状態の可視化** — 得点差・残リソース・勝利条件までの距離を暗算させない。ただし「事実は見せるが最善手は教えない」。
- **フィードバック(ジュース)** — 効果音・アニメで感覚(Sensation)を補う。頻繁な操作ほど演出は短く軽く。
- **操作の摩擦を減らす** — 定型動作をワンクリック化、スナップ。意味のない手作業を削る。
- **取り消し・確認(Undo)** — 探索的プレイを後押し。ただし非公開情報を伴う操作のUndoは不可(巻き戻し悪用対策)。
- **手番・順序の明示** — 「今誰の番か/次は誰か/何をすべきか」を常時表示。

**落とし穴:** 情報を出しすぎ・自動化しすぎると盤面を「読む」時間が消え思考が浅くなる。何を自動化し、何をあえて手作業(=思考の時間)として残すかは主軸の楽しさで決める。

### 5-1b. 【実バグ】`confirm()` / `alert()` / `prompt()` は、埋め込み環境でブロックされる

第4作で「新規ボタンを押しても何も起きない」という不具合が出た。原因は:

> **サンドボックス化された iframe（`allow-modals` なし）では、`window.confirm()` が**
> **ダイアログを出さずに、黙って `false` を返す。**

`if (confirm('やり直す？')) { reset(); }` → **確認も出ず、リセットもされず、エラーも出ない。**
**「ボタンが反応しない」不具合の、典型的な原因のひとつ。**

**対処：ネイティブダイアログを使わず、UI内で完結させる。**

| 操作の重さ | 実装 |
|---|---|
| **軽い**（やり直し等・押し間違えても損が小さい） | **確認を出さず、即実行**（`confirm` を消すだけ） |
| **重い**（退室・他人に影響する） | **2段階クリック** ―― 1回目で文言が「本当に？（もう一度押す）」に変わり、色も変える。数秒で自動キャンセル |

```js
let _ask=false, _t=null;
function askLeave(){
  if(_ask){ clearTimeout(_t); _ask=false; leaveRoom(); return; }   // 2回目 → 実行
  _ask=true; render();                                              // 1回目 → 確認状態へ
  _t=setTimeout(()=>{_ask=false;render();},4000);                   // 4秒で自動キャンセル
}
```

**併せて発見した関連バグ:** 状態を `null` にした後、**残っていた `setTimeout(npcTurn)` が発火してクラッシュ**する。
**非同期処理の入口には必ず `if(!S) return;` のガードを入れる。**

### 5-1c. 【実バグ】全体 `innerHTML` の再描画は、モーダルのスクロール位置を破壊する

**症状:** ルール説明を読んでいる最中に再描画が走ると、**スクロールが先頭に戻る。**

**原因:** `render()` が `#app` を丸ごと `innerHTML` で書き換えており、その中にモーダルが含まれていた。

**対処：モーダルを、再描画の対象外のDOMに分離する。**

```html
<div id="app"></div>
<div id="modal"></div>   <!-- ★ render() は触らない -->
```

```js
function render(){
  app.innerHTML = ...;    // #app だけ
  syncModal();            // モーダルは「開閉時のみ」書き換える
}
function syncModal(){
  if(help && !_open){ modal.innerHTML = helpHTML(); _open = true; }
  else if(!help && _open){ modal.innerHTML = ""; _open = false; }
  // ★開いている間は一切触らない = scrollTop が維持される
}
```

> **原則：「開いている間、DOMが不変であること」がスクロール保持の条件。**
> 個別に `scrollTop` を保存・復元するより、**そもそも書き換えない構造**にする方が確実。

**⚠️ この不具合は「別の症状」として報告される。**
第3作でのプレイヤーからの報告は **「遊び方のスクロールが、全員で同期してる」** だった。
実際には同期などしていない。**他人の操作でDBが更新される → 再描画 → 自分のスクロールが先頭に戻る**
――それが「他人に合わせて動いている」ように見えていた。
**開発中は一人で試すので、再描画が起きず、絶対に気づけない。**

**同じ根を持つ問題（すべて同時に直る）:**
- 入力欄の文字が、再描画で消える／**日本語変換（IME）が「あ」だけ残して壊れる**（§5-1d）
- ドロップダウンの選択が戻る
- チェックボックスが外れる

> **一般原則：リアルタイム更新される領域と、ユーザーの操作状態を持つ領域**
> **（スクロール位置・入力中の文字・開閉状態・選択状態）は、DOMを分離する。**
> **分離できないものは、再描画で失われるUI状態をJS変数に退避する。**

### 5-2. 致命的だった実装ミス(第2作・要注意・再発防止)

- **このアーティファクト環境では、Tailwindの任意値クラス（`bg-[#0B0E11]`のような角括弧記法）はコンパイルされない**。背景・文字色・枠線のほとんどが表示されず、「画面が真っ暗で何も反応しない」という重大な不具合を引き起こした。**次回は最初から、カスタム色は全てインラインstyleで指定する**（`className="bg-[#hex]"`を一切使わない）ことを徹底する
- **危険度メーターに「正確な位置を示す白い線」を実装してしまい、隠しているはずの正確な数値を視覚的に漏らしていた**。**「情報を隠す」設計をした要素については、見た目の演出（アニメーション、位置、サイズなど）が精度の高い情報を副次的に漏らしていないか、都度確認する**（「プロセス検証知見」§9・原則5「秘匿UIは見た目の構造まで確認する」の再確認）

### 6-1. 効果音：Web Audio の合成音だけで完結させる

音声ファイル不要。**単一HTMLのまま音が出せる**ので、配布が楽で読み込み遅延もない。

```javascript
let audioCtx=null, muted=false;
// AudioContext は「ユーザー操作後」でないと started にならない。毎回 resume を試みる
function ac(){
  if(!audioCtx){ try{ audioCtx=new (window.AudioContext||window.webkitAudioContext)(); }catch(e){} }
  if(audioCtx && audioCtx.state==="suspended"){ audioCtx.resume().catch(()=>{}); }
  return audioCtx;
}
// 汎用トーン：周波数・長さ・波形・音量・遅延・スライド先
function tone(freq,dur,type,vol,delay,slideTo){
  const c=ac(); if(!c||muted) return;
  const t0=c.currentTime+(delay||0);
  const o=c.createOscillator(), g=c.createGain();
  o.type=type||"sine"; o.frequency.setValueAtTime(freq,t0);
  if(slideTo) o.frequency.exponentialRampToValueAtTime(slideTo,t0+dur);
  // ★ゲイン包絡：立ち上がりを12msにすると「プチッ」というクリックノイズが消える
  g.gain.setValueAtTime(0,t0);
  g.gain.linearRampToValueAtTime(vol||.14, t0+.012);
  g.gain.exponentialRampToValueAtTime(.0008, t0+dur);  // 指数減衰＝自然な余韻
  o.connect(g); g.connect(c.destination);
  o.start(t0); o.stop(t0+dur+.03);
}
const SFX={
  cut(){  tone(1400,.12,"triangle",.10,0,320); tone(220,.08,"sine",.06,0); }, // 高→低スライドで「切る」感
  pick(){ tone(360,.09,"sine",.12,0); tone(540,.10,"sine",.07,.02); },        // 二音重ねで「取る」感
  score(){[660,880,1100,1320].forEach((f,i)=>tone(f,.5,"sine",.06,i*.06)); }, // 上昇アルペジオ＝達成
  good(){ [523,659,784,1047].forEach((f,i)=>tone(f,.6,"triangle",.07,i*.08)); }, // 高得点時の追加ファンファーレ
  soft(){ tone(300,.06,"sine",.07,0); },   // 軽い操作（刃を置く等）
  click(){tone(520,.05,"triangle",.09,0); tone(780,.04,"sine",.05,.01); }, // 汎用クリック
  yourTurn(){ tone(784,.16,"sine",.11,0); tone(1047,.22,"sine",.10,.12); }, // 手番通知：澄んだ二音
};
```

**音作りのコツ（実践で効いたもの）:**
- **立ち上がり12ms・指数減衰**：これをやらないと「プチッ」とノイズが乗る。最重要
- **音量は控えめ（.05〜.14）**：頻繁に鳴る音ほど小さく。うるさいと即ミュートされる
- **意味と音色を対応させる**：切る＝高→低スライド、取る＝二音重ね、達成＝上昇アルペジオ
- **段階的な報酬**：普通の採点音（score）＋高得点時だけ追加のファンファーレ（good）で「やった感」に差をつける

```javascript

```

### 6-3. 手番通知音：オンラインでは「待ち」が長いので必須

自分の番になった**瞬間だけ**鳴らす。鳴らしすぎると邪魔。

```javascript
let _sndPrev={key:"", myTurn:false};
function soundOnGameTransition(){
  const g=ST.game; if(!g) return;
  const order=chooseOrder(g.cutter,g.n);
  const myTurn = (g.phase==="cut" && ST.mySeat===g.cutter)
              || (g.phase==="choose" && order[g.chooseIdx]===ST.mySeat);
  // 局面が変わった瞬間だけ採点音（同期のたびに鳴らさない）
  const key=`${g.round}:${g.phase}:${g.chooseIdx}`;
  if(key!==_sndPrev.key){
    if(g.phase==="score" && g.roundEval){ ac(); SFX.score(); }
    _sndPrev.key=key;
  }
  // 「待ち → 自分の番」に切り替わった瞬間だけ通知
  if(myTurn && !_sndPrev.myTurn){ ac(); SFX.yourTurn(); }
  _sndPrev.myTurn=myTurn;
}
```
> **原則：リアルタイム同期では「前回の状態と比較」して、変化の瞬間だけ演出する。**
> 状態を購読するたびに鳴らすと、再描画のたびに音が鳴って地獄になる。

### 6-13. ソロ（1人回し）モードは最初に作るべきだった

オンライン対戦をテストするには、本来3〜4人が必要で、毎回集めるのは非現実的。
**1人で全席を操作できるモードがあれば、ロジックの検証が一瞬で終わる。**

- Firebase不要で動く（ネットワーク層の問題と切り分けられる）
- 全情報を表示（デバッグに最適）
- **採点・評価・プールのロジックを本番と共有する**（ここが重要）

> **⚠️ 失敗した点：最初「ローカル検証版」と「オンライン版」を別ファイルで作ってしまい、
> 片方を直すともう片方が古くなる二重管理地獄に陥った。**
> 途中で**単一ファイルに統合**し、ソロ/対戦/観戦がロジックを共有する形にして解決。

> **原則：検証モードは、本番と同じコードを使う「モード」として実装する。別ファイルにしない。**

### 6-18. UX改善の優先順位（今回の実感）

| 4 | **効果音**（Web Audio 合成） | 小 | 大（手触りが激変する） |

| 6 | **観戦モード** | 中 | 大（開発・運用で効く） |
| 7 | **確定前プレビュー** | 小 | 中（不安を消す） |
| 8 | **履歴の表示切替** | 中 | 中 |
| 9 | **アニメーション** | 小 | 中（2箇所に絞る） |

## 7. Windows + Claude Code 開発環境の罠

### 7-0. この開発環境の前提

| 項目 | 状態 |
|---|---|
| OS | Windows（PowerShell 5.1） |
| Python | **3.10.6（導入済み）** |
| git | 2.42（導入済み） |
| **Node.js / npm** | **未インストール** |
| リポジトリ | 日本語パス配下（`D:\ゲーム\ボドゲ\...`） |

> **スクリプト（シミュレーション・ビルドツール）は Python で書く。Node は入れない。**
> 純粋な乱数シミュレーションに Node である必然性はなく、npm を導入すると
> 依存関係・許可ルール・サプライチェーンの面倒が増えるだけである。

### 7-1. PowerShell 固有の罠

| 罠 | 実体 |
|---|---|
| `Get-Process` の `CommandLine` | **PowerShell 5.1 には存在しない**（7.1+ で追加）。`$null -match ...` は常に偽 → **サイレントに何もしない**。代わりに `Get-CimInstance Win32_Process -Filter "Name='python.exe'"` |
| `Get-Content -Raw` と CRLF | 改行を保持するため、`\n` 決め打ちの正規表現は Windows 保存ファイルで外れる。`\r?\n` を使う |
| `Invoke-WebRequest` の例外 | 4xx/5xx でも例外を投げる。**接続不可と HTTP エラーが同じ catch に落ちる** |
| ポートの生死確認 | HTTP リクエストではなく `Test-NetConnection localhost -Port 8811 -InformationLevel Quiet` を使う |
| プロセスの停止 | `Get-CimInstance Win32_Process \| Where-Object { $_.CommandLine -match 'http\.server' } \| ForEach-Object { Stop-Process -Id $_.ProcessId -Force }` |

### 7-2. Claude Code の許可プロンプトが収束しない ― 4つの構造的原因

`--dangerously-skip-permissions` は**採らない**。本番稼働中の GitHub Pages に直結した公開リポジトリであり、
誤った上書き・force push の代償が大きい。方針は「**allow を厚く、deny を壁に**」。
許可ルールは **deny → ask → allow** の順に評価され、**deny はどのスコープからも上書きできない**。

**① 複合コマンドは、サブコマンドごとに独立照合される**
`&&` `||` `;` `|` `&` 改行 が区切り。`python a.py; python b.py` は2回分の照合が必要。1つでも未許可なら止まる。

**② `cd X && git ...` は、どんな allow を書いても止まる**
`cd` でディレクトリが変わると git hooks が走りうるため、必ずプロンプトが出る。
> **対策：`cd X && git ...` を禁止し、`git -C X ...` を使う。**

**③ Windows は `python` であって `python3` ではない**
`Bash(python3 *)` は `python tools\add_toc.py` にマッチしない。パス区切りも `\`。両形式を書く。

**④ ★PowerShell の一発芸スクリプトは、原理的に allowlist できない**
`$src=Get-Content ...; [regex]::Replace(...); [IO.File]::WriteAllText(...)` のような
**その場限りの文字列**は毎回別物になる。許可するには `PowerShell` 単体（＝全許可）にするしかなく、
それは事実上 bypass と同じ。**個別ルールを積んでも永遠に収束しない。**

> **これは「設定が足りない」問題ではなく、「エージェントの作業スタイル」の問題。**
> **インラインのシェル一発芸でファイルを加工させない。Edit ツールか `tools/` のスクリプトに寄せる。**

### 7-3. settings.json 雛形（`~/.claude/settings.json`）

git / python はプロジェクト横断で使うため、**ユーザーレベル**に置く。

```json
{
  "permissions": {
    "defaultMode": "acceptEdits",
    "allow": [
      "Bash(python tools\\add_toc.py *)",
      "Bash(python tools\\build_docs.py *)",
      "Bash(python tools/add_toc.py *)",
      "Bash(python tools/build_docs.py *)",
      "Bash(git add *)", "Bash(git commit *)",
      "Bash(git checkout *)", "Bash(git switch *)", "Bash(git pull *)",
      "Bash(mkdir *)", "Bash(cp *)", "Bash(mv *)",
      "WebFetch(domain:raw.githubusercontent.com)",
      "WebFetch(domain:github.com)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push *)",
      "Bash(git reset --hard *)",
      "Bash(sudo *)",
      "Read(**/.env*)"
    ]
  }
}
```

- `acceptEdits` = ファイル編集と `mkdir`/`mv`/`cp` を自動承認。**効果が最大。**
- `ls` `cat` `grep` `find` `head` `tail` `diff` `wc` は**元々全モードで無確認**。設定不要。
- JSON 内の `\\` は実コマンド上の `\` 1文字。末尾の ` *`（スペース+`*`）は語境界を強制する。
- **`node` / `npm` の許可行は書かない。** この環境には存在しない。
- **`git push` は deny。** GitHub Pages は即本番。承認ゲートを構造で担保する。

### 7-4. 削減してはいけないもの

**`Remove-Item -Recurse -Force` / `rm -rf` は、プロンプトが出るのが正しい。**
再帰・強制削除を allowlist に入れる＝パス変数がひとつ空になっただけでドライブ直下を消せる権限を常時渡すこと。

> **ここで止まってくれるのは、システムが正しく機能している証拠。**

### 7-5. Claude Codeセッションの非永続性と復旧

- セッションコンテナは毎回新規。新セッションはoriginからしか出発できず、前セッションの
  未push作業は`git fetch`しても存在しない（「全ブランチを検索しても無い」は消失でなく
  未pushの兆候）
- 復旧の第一手はアプリのセッション一覧からのresume。resumeが生きていればブランチ・
  作業ツリーごと復元される
- 節目（凍結コミット・resultsコミット）ごとの作業ブランチpushを指示書の手順に組み込む
  こと（mainへの操作とは区別する）

### 7-6. チャット貼り付けによる書式欠落

- チャットUI経由の長文貼り付けで行頭のMarkdown記法（`##`等）が脱落する事故が実測
  された。受領側での機械的な検出法：既存の同型ファイル（前工程の凍結ファイル等）と
  見出し構造を比較する
- `add_toc.py`は見出しが存在しないと無警告で何もしないため、TOC生成の空振りが書式
  欠落の間接的な検出器になる。逆に言えば「TOCが生成された」ことは見出し復元の完了
  条件として使える
