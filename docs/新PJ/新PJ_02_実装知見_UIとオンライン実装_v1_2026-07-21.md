# 新PJ 実装知見 ―― UI/UX・オンライン実装・開発環境

作成日：2026-07-21 ／ 出所：旧PJ『ボードゲーム制作_実装技術知見』からの抜粋・再編
**前提技術：**単一HTML + 素のJS + Firebase Realtime Database（匿名認証） + GitHub Pages
**再編方針：**一回きり・反転型（謎解き・協力・秘匿観測）で実際に効くものを優先。実バグ記録は全部残した。

---

## 1. 基盤設計 ―― 最初に固める3つ

### 1-1. 「アクション＝イベントの列」設計（イベントソーシング的）

これを最初に固めると、**ログ・リプレイ・Undo・アニメーション・セーブ・オンライン同期がすべて同じ土台に乗る。**

- 状態そのものを同期せず、`games/{gameId}/events` にアクションを追記
- 各クライアントが同じイベント列から状態を再生
- **「イベントを適用して次の状態を返す純粋関数」を独立させる**（オンライン非依存でテスト可・ルール変更が容易）
- 構成は「ゲームロジック / Firebase同期 / UI描画」を分離する

> **新PJでの特別な価値：一回きりのゲームは、後から「あの卓で何が起きたか」を検証したくなる。**
> **イベント列がそのままプレイテストのログになる。**観測枠テンプレートの「詰まり地点の記録方法」が、追加実装なしで埋まる。

### 1-2. 権威（source of truth）をどこに置くか

- Security Rules で「自分の手番の正当なアクションだけ書ける」よう制限する、または
- Cloud Functions をサーバー権威にし、クライアントは申請 → 関数が検証して確定イベントを書く

**非公開情報が絡むなら、サーバー権威は事実上必須。**

### 1-3. 【最重要】非公開情報は「見えなくする」ではなく「配らない」

> **全状態をクライアントに配ると、開発者ツールから丸見え。**
> **発見系・正体隠匿系・謎解き系は、これで即死する。**

**新PJは謎解きなので、これはジャンルの生死に直結する。**答え・仕込み・未観測の手がかりがクライアントに落ちていたら、
ゲームは成立していない。

**実証済みの対処（魔女の調合 v1.3）：**

| 対処 | 内容 |
|---|---|
| **UID別パスへ分ける** | `rooms/{id}/private/{uid}/...` に置き、Security Rules で本人のみ読めるようにする |
| **派生データは終了時に再計算** | 途中は秘密のまま保持し、終了時にサーバー側で確定値を計算して公開する |
| **秘匿判定は「場所ベース」で書く** | 「判定確定済み以外の場所に、秘密の形をした値が1つでも残っていたら漏洩」。値ベース（値がどこかに一致すれば許可）は骨抜きになりやすい |
| **故意注入テストを書く** | 秘密を各所に故意に混入させ、必ず検出されることを実地確認する（→ 存在証明義務） |

**関連する実バグ：**
> 危険度メーターに「正確な位置を示す白い線」を実装してしまい、**隠しているはずの数値を視覚的に漏らしていた。**
> **「情報を隠す」設計をした要素は、見た目の演出（アニメーション・位置・サイズ）が精度の高い情報を副次的に漏らしていないか、都度確認する。**

### 1-4. 秘匿状態がゼロなら、Firebaseは要らない

情報を全部公開する版（プロトタイプ・土台版）を作るときは、
**静的HTML1枚＋通話で画面共有**で機能的に完備する。同期基盤を使わない。

## 2. 共有状態（オンライン）で踏んだ7つの罠

### 2-1. 「最後に書き込んだクライアントが勝つ」（read-modify-write競合）

読んで → 加工して → 書き戻す、を複数クライアントが同時に行うと、片方の更新が消える。
**→ トランザクション、または「追記のみ」の設計にする**（イベントソーシングならこの問題が構造的に起きない）。

### 2-2. 「見えなくする」だけでは「操作できなくする」ことにならない

UIでボタンを隠しても、コンソールから関数は呼べる。**権限はサーバー／ルール側で判定する。**

### 2-3. 「幽霊」（切断したプレイヤー）対策は性善説では機能しない

「退室ボタンを押してもらう」前提は必ず破れる。
**→ `onDisconnect` とハートビート（最終更新時刻）の両方を使い、タイムアウトで自動的に扱う。**

### 2-4. 「同期しているはずなのに、本当に同期しているか分からない不安」

オンラインの最大のストレスは待ちではなく**不確実性**。
**→ 最終同期時刻・接続状態・「相手は待機中」を常時表示する。**

### 2-5. 一時的な通信エラーは、自動リトライで多くが解決する

### 2-6. エラーを握りつぶさない

`catch(e){}` で黙らせると、原因不明の不具合として何時間も溶ける。
**→ 画面上の固定領域にエラーを出す。**

### 2-7. 部屋の掃除は最初から入れる

TTL（例：24時間）＋起動時のクライアント掃除＋roomIndex方式。
**roomsV2を直接列挙するとカスケード読み取りが発生する**ため、インデックスを別に持つ。
（魔女の調合で実施済み。カットオーバー順：コードpush → ルール適用 → 旧ゴミ部屋の手動一掃）

## 3. UI/UX の原則

### 3-1. 【核心原則】判断の「補助」は足すほど良い、「肩代わり」は楽しさを殺す

| 補助（足す） | 肩代わり（足さない） |
|---|---|
| 履歴・ログ（直前アクションを盤面ハイライトで一瞬表示） | 最善手サジェスト |
| 状態の可視化（得点差・残リソース・勝利条件までの距離を暗算させない） | 自動プレイ |
| 効果音・アニメによるフィードバック | 「良い手を勧める」 |
| 定型動作のワンクリック化 | ― |
| **有効手のフィルタリング（違法手を弾くまで）** | ― |
| 手番・順序の明示 | ― |

> **落とし穴：**情報を出しすぎ・自動化しすぎると盤面を「読む」時間が消え、思考が浅くなる。
> **何を自動化し、何をあえて手作業（＝思考の時間）として残すかは、主軸の楽しさで決める。**

**新PJでの読み替え ―― ここが最も難しい判断になる：**
謎解きでは「メモ機能」「手がかり一覧」「自動整理」が親切に見えるが、
**手がかりを並べ替えて眺める行為そのものが、気づきの発生源**である可能性が高い。
**自動整理は、エウレカの発生装置を奪う。**

### 3-2. 【実バグ】`confirm()` / `alert()` / `prompt()` は埋め込み環境でブロックされる

> サンドボックス化された iframe（`allow-modals` なし）では、`window.confirm()` が
> **ダイアログを出さずに、黙って `false` を返す。**

`if (confirm('やり直す？')) { reset(); }` → **確認も出ず、リセットもされず、エラーも出ない。**
**「ボタンが反応しない」不具合の典型的な原因。**

**対処：ネイティブダイアログを使わず、UI内で完結させる。**

| 操作の重さ | 実装 |
|---|---|
| 軽い（押し間違えても損が小さい） | 確認を出さず即実行 |
| 重い（退室・他人に影響する） | **2段階クリック**（1回目で文言と色が変わり、数秒で自動キャンセル） |

```js
let _ask=false, _t=null;
function askLeave(){
  if(_ask){ clearTimeout(_t); _ask=false; leaveRoom(); return; }   // 2回目 → 実行
  _ask=true; render();                                              // 1回目 → 確認状態へ
  _t=setTimeout(()=>{_ask=false;render();},4000);                   // 4秒で自動キャンセル
}
```

**関連バグ：**状態を `null` にした後、残っていた `setTimeout` が発火してクラッシュする。
**非同期処理の入口には必ず `if(!S) return;` のガードを入れる。**

### 3-3. 【実バグ・最重要】全体 `innerHTML` の再描画は、ユーザーの操作状態を破壊する

**症状：**ルール説明を読んでいる最中に再描画が走ると、スクロールが先頭に戻る。

**原因：**`render()` が `#app` を丸ごと `innerHTML` で書き換えており、その中にモーダルが含まれていた。

```html
<div id="app"></div>
<div id="modal"></div>   <!-- ★ render() は触らない -->
```

```js
function render(){
  app.innerHTML = ...;    // #app だけ
  syncModal();            // モーダルは「開閉時のみ」書き換える
}
function syncModal(){
  if(help && !_open){ modal.innerHTML = helpHTML(); _open = true; }
  else if(!help && _open){ modal.innerHTML = ""; _open = false; }
  // ★開いている間は一切触らない = scrollTop が維持される
}
```

> **原則：「開いている間、DOMが不変であること」がスクロール保持の条件。**
> 個別に `scrollTop` を保存・復元するより、**そもそも書き換えない構造**にする方が確実。

**⚠️ この不具合は「別の症状」として報告される。**
実際のプレイヤー報告は **「遊び方のスクロールが、全員で同期してる」** だった。
同期などしていない。**他人の操作でDBが更新される → 再描画 → 自分のスクロールが先頭に戻る**、それだけ。
**開発中は一人で試すので再描画が起きず、絶対に気づけない。**

**同じ根を持つ問題（すべて同時に直る）：**
- 入力欄の文字が再描画で消える／**日本語変換（IME）が壊れる**
- ドロップダウンの選択が戻る／チェックボックスが外れる

> **一般原則：リアルタイム更新される領域と、ユーザーの操作状態を持つ領域**
> **（スクロール位置・入力中の文字・開閉状態・選択状態）は、DOMを分離する。**

**新PJでは特に重い：**謎解きは「長文を読む」「メモを書く」「一覧を眺める」が中心操作になる。
**この不具合が残っていると、ゲームが遊べない。**設計段階からDOM分離を前提にする。

### 3-4. Tailwindの任意値クラスは使わない

アーティファクト環境では `bg-[#0B0E11]` のような角括弧記法がコンパイルされず、
**画面が真っ暗で何も反応しない**という重大な不具合になった。
**カスタム色はすべてインラインstyle、またはCSS変数で指定する。**

## 4. UX実装レシピ（コピペ可）

### 4-1. 効果音：Web Audio の合成音だけで完結させる

音声ファイル不要。**単一HTMLのまま音が出せる**ので配布が楽。

```javascript
let audioCtx=null, muted=false;
// AudioContext は「ユーザー操作後」でないと started にならない。毎回 resume を試みる
function ac(){
  if(!audioCtx){ try{ audioCtx=new (window.AudioContext||window.webkitAudioContext)(); }catch(e){} }
  if(audioCtx && audioCtx.state==="suspended"){ audioCtx.resume().catch(()=>{}); }
  return audioCtx;
}
function tone(freq,dur,type,vol,delay,slideTo){
  const c=ac(); if(!c||muted) return;
  const t0=c.currentTime+(delay||0);
  const o=c.createOscillator(), g=c.createGain();
  o.type=type||"sine"; o.frequency.setValueAtTime(freq,t0);
  if(slideTo) o.frequency.exponentialRampToValueAtTime(slideTo,t0+dur);
  // ★立ち上がり12ms＋指数減衰＝「プチッ」というクリックノイズが消える。最重要
  g.gain.setValueAtTime(0,t0);
  g.gain.linearRampToValueAtTime(vol||.14, t0+.012);
  g.gain.exponentialRampToValueAtTime(.0008, t0+dur);
  o.connect(g); g.connect(c.destination);
  o.start(t0); o.stop(t0+dur+.03);
}
```

**音作りのコツ：**
- 音量は控えめ（.05〜.14）。頻繁に鳴る音ほど小さく
- 意味と音色を対応させる（切る＝高→低スライド、取る＝二音重ね、達成＝上昇アルペジオ）
- 段階的な報酬：通常の採点音＋高得点時だけ追加のファンファーレ

**新PJでの用途：気づき・発見の瞬間に音を当てる価値は高い。**
ただし**「正解しました」を音で教えるのは、検証可能性をシステムが肩代わりすること**になりかねない。
自力で確認できる設計を優先し、音は追認に使う。

### 4-2. 全ボタンに自動でクリック音（イベント委譲）

```javascript
document.addEventListener("click", e=>{
  const b=e.target.closest("button");
  if(b && !b.disabled){ ac(); SFX.click(); }
}, true);   // capture=true で他ハンドラより先に確実に拾う
```
**`!b.disabled` の確認が重要。**無効ボタンで音が鳴ると「押せた？」と誤解させる。

### 4-3. 手番通知音：オンラインでは「待ち」が長いので必須

> **原則：リアルタイム同期では「前回の状態と比較」して、変化の瞬間だけ演出する。**
> 状態を購読するたびに鳴らすと、再描画のたびに音が鳴って地獄になる。

### 4-4. ミュート切替は必ず用意する（アイコンが状態を表す。解除時に小さく鳴らす）

### 4-5. 色設計：CSS変数で一元管理し、意味と色を対応させる

## 8. 自動テスト設計

- **推理系ゲームの整合性不変条件**：矛盾する状態が生成されないことを機械的に検査する
- **秘匿判定は故意注入テストで検証する**（§1-3）
- **アサーション数は検出力の代理変数にならない。**件数を完了条件にせず、
  **「発火例と非発火例の両方を提示せよ」**を完了条件にする

**新PJでの特別な用途：作問した謎が「解ける」ことの機械的検証。**
手がかりから答えへの導出経路が実際に存在するか、
**総当たりでない経路が存在するか**を、テスターを消費せずに検査できる可能性がある。
これは机上プレフライトの自動化にあたる。**早い段階で検討する価値が高い。**
