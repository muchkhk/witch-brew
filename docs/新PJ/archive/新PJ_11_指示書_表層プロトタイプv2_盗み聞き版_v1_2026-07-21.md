# 【Claude Code 指示書】表層プロトタイプ v2 ―― 交換台・盗み聞き版

発行日：2026-07-21 ／ 発行元：設計チャット5（Opus）
宛先：Claude Code
関連：新PJ_10（v1 指示書）／報告_表層プロトタイプ_交換台_v1／新PJ_09（F-7・F-8）／実装知見（02）

## 【目的】

v1 は「作業に感じる」で kill 文言に触れたが、**診断の結果、表層の死ではなく検査の欠陥**と判定した。

> v1 は「加入者に属性を持たせない／通話の長さは接続前に不明」としたため、
> **接続先を選ぶ材料が構造的にゼロ**だった。設計知見 §1 の「可読性」が最初から欠けており、
> 最適行動は「一番待っている人から繋ぐ」に固定される。これは判断ではなくディスパッチ規則である。
> 加えて、報告 §3 の「60%で追いかけ呼び出し」は予見不能な帰結であり、帰結型ではなく類型22（博打型）だった。

v2 は判断材料の供給源を1つだけ足して、**「誰を繋ぎ、誰を待たせるか」に判断が立つかどうか**を再検査する。

**判定に使う言葉（v1 から1つ追加）：**

| 出たら死んでいる | 検出しているもの |
|---|---|
| 「どっちでもいい」「作業に感じる」「何も考えずに押している」 | 類型21 無差別型／設計知見 §1 |
| **「聞かない方が早い」** | **死因1-5 市場不成立／類型23 逆インセンティブ** |
| 「運次第」「読みようがない」 | 類型22 博打型 |

**これは「面白い」の証明ではない。**設立メモ §3-1 の通り、ソロ／プロトタイプは
**ネガティブフィルタ（死んでいることの確認）にしか使えない。**

---

## 【背景】

- 表層は電話交換局（新PJ_09 F-8）。プレイヤーは交換手で、加入者からの呼び出しを繋ぐ
- v1 の成果物 `proto/switchboard_proto.html` は土台として使える。**作り直さず、そこから改造してよい**
- v2 は**情報層をゼロにしない。**「盗み聞き」だけを載せる（本番の観測動詞）
- ただし**本番の核（非同一性の仕掛け）は絶対に載せない**（後述の禁止事項）
- 秘匿情報は存在しない（1人用）ため、**Firebase は不要。単一の静的HTML 1枚のまま**（実装知見 §1-4）
- 1人で動かせること

## 【やること】

### 1. 成果物

- **単一HTMLファイル 1枚。**外部ファイル依存なし（CSS・JS はインライン）
- ファイル名：`switchboard_proto_v2.html`（v1 は残す。上書きしない）
- 1人で最初から最後まで通せること（1セッション 5〜10分程度）

### 2. v1 から変更する点（本指示書の本体）

#### 2-1. プラグの用途を2つにする ―― 耳側に挿す

- プラグ1対は、**「繋ぐ」か「聞く」のどちらか一方**にしか使えない
- **聞く**：通話中の1件を選んで耳側に挿すと、その通話の会話が**1行ずつ、数秒おきに**流れる。
  会話が終わるか、プレイヤーが抜くまで、そのプラグは占有される
- 会話の表示先は記録欄とは**別の領域**（「聴取欄」）。記録欄と混ぜない
- 抜いたプラグは即座に空く

> **これが F-6（観測と経路指定が単一の有限プールを共有する）の実物化である。**
> 聞くことは、繋げないことと同義でなければならない。**別枠を作らないこと。**

#### 2-2. 帰結を決定的にする ―― ランダム追いかけ呼び出しの廃止

- v1 の「通話終了後 60% の確率で追いかけ呼び出し」を**削除する**
- 代わりに、**加入者間に固定の関係を持たせる**（データとして事前に定義）。
  例：Aは Bの雇い主／CはDに金を借りている／EとFは同じ家に住んでいる 等
- 「誰と誰を繋いだか」「誰を待たせたか」に対して、**関係にもとづく決定的な帰結**が起きる。
  同じ状況なら同じことが起きる（乱数で分岐させない）

#### 2-3. 関係は表に出さず、会話からしか知れない

- **加入者一覧に関係を表示しない。**名簿は名前だけ（v1 と同じ）
- 関係は**盗み聞きした会話の中に断片的に現れる**
- したがって：聞いた人だけが帰結を予測でき、聞かなかった人は予測できない

> **2-1〜2-3 の3点セットが v2 の本体である。**
> 1つでも欠けると v1 と同じ「材料のない選択」に戻り、検査が成立しない。

#### 2-4. 放棄の扱い

- v1 の「放棄された発信者は以後二度と呼ばない」は**維持する**（痛みの源。死因1-3 生命感）
- ただし、**その加入者が離れそうであることは、会話から予見できる**ようにする
  （例：「もう一度だけかけてみる」といった発話が事前に流れる）
- 予見不能な恒久ペナルティにしないこと

### 3. 会話データについて（重要）

- 会話は**8〜12名の加入者と、それらの間の関係**から成る小さな世界として、**この指示書の実行者が作成してよい**
- 分量の目安：加入者10名／関係6〜10本／会話パターン 20〜30本程度
- **文面は使い捨てである。**本番の謎には一切流用しない（流用すると初見が焼ける）。
  したがって凝りすぎなくてよい。**判断材料として機能する最低限**でよい
- **良い／悪いの評価語を出さない**（F-7 帰結型）。帰結は出来事の文だけで表す

### 4. v1 から維持する点

- 呼び出しは時間経過でキューに追加される（ランダム間隔）
- 通話には長さがある。**接続前には分からない。ただし聞けば見当がつく**ようにする
- 待ち時間が一定を超えると放棄される
- セッションは固定時間で終了。終了時に記録欄の全文を表示
- 記録欄は追記のみ・上が新しい

### 5. 数値パラメータ

**すべて画面上で、実行中に変更できること**（v1 と同じ方針）。

プラグ本数／呼び出しの発生間隔／放棄までの待ち時間／通話の長短の比率／セッション長／
**会話1行あたりの表示間隔**（新規）

### 6. 実装上の必須事項（実装知見 02 より。省略不可・v1 から継続）

- **`confirm()` / `alert()` / `prompt()` を使わない**（§3-2）。確認が要る操作は2段階クリック
- **全体 `innerHTML` の再描画をしない**（§3-3）。
  リアルタイム更新される領域（キュー・パネル）と、スクロール位置を持つ領域（**記録欄・聴取欄**）の
  **DOM を分離する。**追記のみで既存行を書き換えない
- **Tailwind の任意値クラスを使わない**（§3-4）。色は CSS 変数で一元管理
- **エラーを握りつぶさない**（§2-6）。画面上の固定領域に表示
- 非同期処理の入口に必ずガードを入れる（§3-2 関連バグ）
- **画面に VER 定数のバッジを出す**（§5）
- 効果音は任意。付ける場合は Web Audio の合成音のみ（§4-1）。ミュート切替必須

## 【触ってはいけないもの】

- **Firebase・オンライン同期・認証を一切実装しない**
- **本番の核を一切入れない。**具体的には：
  - 同じ名前が2つある／別の局が存在する、といった仕掛け
  - 不一致・非同一・確約・署名にあたる機構
  - **同じ加入者を両端に差す操作**（本番の核。入れると初見が焼ける）
- **メモ機能・呼び出し一覧の自動整理・並べ替え・フィルタを実装しない**（handoff 04 §1-1 凍結済み）
- **聞いた内容を自動で整理・要約・一覧化しない。**聴取欄は流れるログのみ。
  （実装知見 §3-1：並べ替えて眺める行為そのものがエウレカの発生源であり、自動整理はそれを奪う）
- **関係図・相関図を表示しない。**関係は会話からしか知れないこと
- **スコア・評価・「上手い／下手」の表示を作らない**（F-7）
- v1 の `switchboard_proto.html` を書き換えない・削除しない。既存 `index.html` をリネームしない
