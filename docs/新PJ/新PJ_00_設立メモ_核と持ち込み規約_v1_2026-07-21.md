## 0. なぜ分けたか（1分で読む用）

旧プロジェクトのゲート体系（X0〜S0、閉鎖済み超系譜、禁止動詞）は、
**「対人・繰り返し遊べる・隠し情報の読み合いゲーム」の連続失敗から作られたフィルタ**である。

狙っている体験はそこに属していない。同じ棚に置いた結果、
**核となる体験が3回続けて設計から脱落した**（候補6の埋葬 → handoff §6での降格 → 土台版で消滅）。

> **機序：フィルタを生成器として使うと、出力はフィルタの通過実績に引き寄せられる。**
> **落とすための道具で作ろうとすると、既に通ったものの近傍しか出てこない。**

旧PJの通過実績はほぼ全部が「系統的な資源管理型」だった。だから毎回そこへ引き戻された。
**これは偶然ではなく機序なので、棚を分けない限り4回目が起きる。**

## 1. 核となる体験（凍結。降格させない）

```
ある情報を得たことで、前提だと思っていたことがひっくり返り、
今まで見てきた情報の見え方が変わる。
これが一度ではなく、種類を変えて繰り返し起きる。
各反転は不可逆であり、卓は一回きりである。
```

> **改訂記録（2026-07-22）：**旧文は「不可逆で、一回きり。」だった（archiveに逐語保存：
> `archive/新PJ_00_核凍結文_旧版_2026-07-21.md`）。**判断種別：作家判断（ユーザー・
> 2026-07-22。改訂許可）＋文案は机上判断（Opus）。**
>
> **注記：**「一回きり」には *卓が一回きり*（＝初見が希少資源。§3の運用全体がこれに依存する）と
> *反転が一度きり* の2つが混在していた。**改訂したのは後者のみであり、前者は一切変更されていない。**

### 1-1. この一文の扱い（旧PJからの最重要の変更点）

| | 旧PJ | 新PJ |
|---|---|---|
| 反転体験の地位 | `DESIRED_SCENE` ＝ **仮説。PASS証拠にしてはならない**装飾 | **設計の起点。これが無い案は、他が全部良くても不採用** |
| 設計の起点 | 中心決断（資源葛藤） | **反転体験** |
| 「反転のために中心決断を変更する」 | 禁止 | **許可**（そもそも反転が先） |

旧PJの「護り3本」（反転をPASS証拠にしない／起点にしない／そのために中心決断を変えない）は、
**新PJでは全部無効**である。あれは反転を装飾扱いするための拘束だった。

### 1-2. ただし、旧PJが正しく指摘していた3つの脅威は生きている

これらは反転を降格させるためではなく、**反転を実際に成立させるために**必要な検査項目である。

| # | 脅威 | 内容 |
|---|---|---|
| 1 | **反転を味わうのは1人だけ問題** | 多人数で誰かが先に気づいて口に出した瞬間、残りの体験は「なるほどね」に劣化する。期待値が人数分の1になる |
| 2 | **リズムの不一致** | 反転の発火時点で**未確定の行動が残っていなければ**、反転は行動に反映されず、ただの種明かしになる |
| 3 | **発火の保証がない** | 「気づかなくても進める」（詰み回避）は「気づかないまま完走できる」と同義。一回きりの卓で発火しなければ、それで終わり |

**旧PJで到達した対処案（有効。持ち越す）：**

> 有限の観測機会を複数人で分担するため、誰も全体を見ていない。
> **各人の断片を持ち寄った瞬間、全員の観測の意味が同時に変わる。**

これは脅威1への正しい対処である。ただし**未解決の危険が1つ残っている**：

> 「持ち寄れば分かる」は**単なる情報合成＝帳簿作業**に堕ちる。
> 断片A＋断片B＝答え、なら反転ではなくパズルの組み立て。
> **反転が成立するのは、合流によって「既に見たもの」の意味が変わるときだけ。**

## 2. 持ち込むもの・置いていくもの（規約）

### 2-1. 無条件で持ち込む（プロセス規律）

| 項目 | 内容 |
|---|---|
| **表層単体の面白さ** | 情報層を外した表層が単体で面白くなければ kill。範囲論に勝っても独立に残る |
| **事前登録の厳守** | kill条件は実施前に凍結。結果を見てから閾値・基準を動かさない。救済には新たな正当化が必要 |
| **存在証明義務** | kill条件は「発火例（故意に壊すと必ず落ちる）」と「非発火例（正常なら必ず通る）」を両方構築してから登録する。件数は検出力の代理変数にならない |
| **構造診断 vs 症状パッチ** | 新要素を足すたび「核を強めるか／症状を抑えるだけか」を宣言。パッチが3つ溜まったら実装を止めて再診断 |
| **死因を2件記録して埋葬** | パッチで延命しない。埋葬したら新しい核から出発する |
| **判断種別の記録** | kill/keepには「実プレイ判定か机上判断か」を必ず残す。机上分類は信頼できない |
| **違和感を軽視しない** | 「何か変」「どっちでもいい」「作業に感じる」は構造的問題の兆候として扱う |
| **設計順序** | 主軸の楽しさ → プレイヤー間の構造 → 核となる判断 → メカニクス → テーマ → 人数 → 数値 → UI → 実装。上流が未確定のまま下流を作り込まない |
| **役割分離** | 設計チャットはコード・シミュレーションを書かない。実装は指示書でClaude Codeへ。下流が上流を検査する経路としても機能する |
| **指示書の受け渡し確認** | 「発行済み」と「受領済み」は別。実行側が内容を確認したことをチャット上で確認してから完了扱いにする |
| **凍結テキストはファイルで渡す** | チャット貼付ではMarkdown書式が無音で落ちる |
| **未pushコミットは存在しない** | コンテナは非永続。push＝作業の存在確認 |
| **有限テスター資源下のkill運用（方法論v1）** | **これは元々この体験のために書かれた文書。最重要の持ち込み品**（→ §3） |

### 2-2. 置いていく（旧PJに残す）

| 項目 | 置いていく理由 |
|---|---|
| `X0/X1/B0/B1/I0/I1/S0` のゲート一式 | 対人読み合いゲームのために作られたフィルタ |
| **B0「情報を全部外してもゲームか」** | 一回きりの反転体験では**情報が商品そのもの**。外すと何も残らないのは欠陥ではなくジャンルの定義。この衝突は既に3回起きている |
| **I1「同じ参加者に情報ありで再テスト」** | 気づきは使い捨て。原理的に実施不能 |
| 閉鎖済み超系譜 C・E・G・H | すべて「相手の内面や隠れた値を読む」系。新PJの体験はそこに属さない |
| 3サイクルの禁止動詞（読む／解読する／見抜く／当てる） | 同上。**対システムの解読は元々この禁止の趣旨に含まれない**（旧PJで判定済み） |
| 連続KILL横断監査 §8 の階段（情報OFF土台 → 人間5〜10回 → …） | B0と同じ理由 |
| 候補1〜6、F1〜F23 | **復活禁止は継続。**記録としての保存のみ可、部品としての再利用は禁止 |

### 2-3. 条件つきで持ち込む（歯止めを差し替える）

**`INFORMATION_FIRST_DESIGN = PROHIBITED`（情報機構を先に作って目的を後付けするな）**

旧PJでは禁止事項だったが、一回きりの体験ゲームは本質的に情報が先である
（脱出ゲーム、Return of the Obra Dinn、Outer Wilds、Her Story ―― すべてそう）。

**新PJでは、この禁止を次に差し替える：**

> **情報機構を先に作ってよい。ただし「その情報を知ったプレイヤーが、次に何をするか」が空欄のまま先へ進まない。**

候補6が実際に死んだのはここである。設計文書自身が「終わり方の不在」「遊びが空」と記録していた。
情報が先だったこと自体ではなく、**知った後の行動が空欄だったこと**が死因の実体だった。

## 3. 検証の運用（一回きりゲームの最大の難所）

### 3-1. 制約（旧PJから継続。棚を分けても消えない）

- テスターは身内3〜4人。**本番＝テスト。修正して再テストする相手はいない**
- 各人の「初見」は一度しか使えない。**初見は最も希少な資源**
- ソロ／NPC／シミュレーションは**ネガティブフィルタのみ**。「面白かった」は証拠にならない。「死んでいる」ことの確認にしか使えない

### 3-2. 二段構え

```
机上プレフライト（人間不要）
  → ここで死んでいれば、テスターを1人も消費せずに埋葬できる
        ↓ 生存
有限テスター資源（人間必須・1人1回）
  → 事前配分に従って順次投入
```

**机上プレフライトを厚くするほど、テスター消費が最終判定に集中できる。**
このジャンルでは「安く殺す」の"安さ"は、**テスター人数をどれだけ節約できるかで測る。**

### 3-3. 机上プレフライトの検査項目

| # | 検査対象 | 死んでいる状態 |
|---|---|---|
| 1 | 総当たり耐性 | 気づきの選択肢が少なく、総当たりで通過できる |
| 2 | 検証可能性 | 気づいた瞬間、その場で真偽を確認できない |
| 3 | 誤答の破壊性 | 間違えると不可逆に状態が壊れ、以後遊べない |
| 4 | 冗長経路 | 1本道で、詰んだら詰みっぱなし |
| 5 | 鍵バイパス | 気づかなくても突破できる裏口がある |
| 6 | **反転の発火タイミング** | 反転時点で未確定の行動が残っていない＝ただの種明かし（§1-2 脅威2） |
| 7 | **合流の質** | 断片A＋断片B＝答え。既に見たものの**意味が変わらない**＝帳簿作業（§1-2） |
| 8 | **冗長経路と発火保証の両立** | 相反しやすい。**ここが主戦場**（§1-2 脅威3） |

### 3-4. 一発kill条件（1回の観測で発火する定性条件）

統計が組めないので、量的閾値ではなく定性条件を使う。**発火例・非発火例を両方書いてから凍結する。**

| kill候補 | 発火例（死んでいる） | 非発火例（生きている） |
|---|---|---|
| **エウレカの不在** ★最重要 | 正解に辿り着いても「なぜ合っているか」を説明できない | 正解の理由を、手がかりに遡って自分の言葉で説明できる |
| 総当たり検出 | 謎を読まずに総当たりで通過し、本人が「別に考えてない」と言う | 総当たりも可能だが、自発的に手がかりを読みに行く |
| 1本道詰み | ある1点で完全停止し、他へ迂回する動きが一切出ない | 詰まった後、自発的に別の手がかりを探しに戻る |
| 外部依存の即時化 | 数分で「攻略見た方が早くない？」と口に出す | 詰まっても、まだ自力で試す構えが続く |

> **エウレカの不在が最重要。**正解率だけ見ると「解けた」と誤判定するが、
> **理由を説明できない正解は事故であり、このジャンルの核（気づきそのものが報酬）が発火していない。**

### 3-5. 観測枠テンプレート（人間セッション前に埋める）

```
□ 確保できるテスター人数（N）：
□ 配分（1人目＝導線確認／中盤＝気づきの分布／最後＝通し判定）：
□ 各枠で測る対象（1つに絞る。欲張らない）：
□ ヒント介入の方針（どこまで自力で見せるか）：
□ 打ち切り基準（何分／何回の失敗で次へ進めるか）：
□ 適用するkill条件（§3-4から選択。発火例・非発火例を確認済みか）：
□ 詰まり地点の記録方法：
```

> **事前登録すべきは「閾値」ではなく「誰に何を聞くか」。**
> 配分を先に決めずにテストへ入ると、詰まった1人目への対応がその場の判断に流され、
> 「何を測ったか分からないテスト」になる。

## 4. 未解決の懸念（設立時点で判明しているもの）

### 懸念1：このジャンルの質は、機構ではなく作家性から出る

Obra Dinn も Outer Wilds も Her Story も、面白さの源泉は**「よく書けた謎と、よく仕込まれた嘘」**である。
旧PJで積み上げた道具（kill条件・シミュレーション・事前登録）は**系統的なゲームのための道具**であり、
作家性コンテンツにはほとんど効かない。

**→ 対応方針：ユーザーは「よく書けた謎を書く作業に時間を使う意思がある」と表明済み（2026-07-21）。**
この前提が崩れたら、新PJ自体を再検討する。

### 懸念2：反転体験を届ける既存作は、ほぼ全部が1人用

多人数で「前提がひっくり返る」を成立させた例は、マーダーミステリー（旧PJで除外済み）、
レガシー系（Pandemic Legacy）、脱出ゲーム箱 ―― とかなり限られる。
**§1-2 の「誰も全体を見ていない／断片を持ち寄る」は、この問題への現時点で最良の対処案。**

### 懸念3：一回きり・不可逆と、検証は原理的に相性が悪い

観測は1卓分しかない。外したら修正して試す相手がいない。**§3 の運用はこの制約のためにある。**

### 懸念4：作問コストが償却できない

一回きりなので、作った謎は1卓で消費される。
**同じ骨格で謎を差し替えられる構造**（作問器・カードプール量産）を早い段階で検討しないと、
1作あたりのコストが釣り合わなくなる可能性がある。旧PJでは伝達核v1.0で同種の対処（毎試合8枚配布・秘密は有効軸のみ）を採っている。

## 5. 新PJ最初の作業

1. 核となる体験（§1）を**降格させずに**確認し、凍結する
2. その体験を成立させる最小の器を、**反転を起点として**設計する（旧PJのように既存ゲームから中心決断を借りてこない）
3. 机上プレフライト（§3-3）を通す
4. 一発kill条件（§3-4）を、発火例・非発火例つきで凍結
5. 実装指示書 → Claude Code

## 6. 運用（旧PJから継続）

- 設計チャットはコード・スクリプト・シミュレーションを書かない。実装・ファイル操作・リリース・MD管理は
  Claude Code へ指示書形式（【目的】【背景】【やること】【触ってはいけないもの】【完了条件】【報告してほしいこと】）で委譲
- **モデル：**核心診断・kill/keep判断・構造診断は Opus ／ 整合性検査・網羅表・MD整形・指示書の定型起草は Sonnet 5
- **記号を使うときは初出で日本語を併記する**（旧PJで、判断者が読めない記号で運用していた失敗があった。
  「統制は判断者が読める場所に置かれた時点で有効になる」という自らの知見に違反していた）
- チャット肥大時は警告する
- 各チャット末に handoff メモを生成する
