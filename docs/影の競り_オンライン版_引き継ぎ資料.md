## 1. 現在の公開状態

- 本番公開中の旧版とv2候補は互換性がない。
- v2候補の作業ブランチは `fix/seri-v2-security`。
- P0緊急封鎖は完了済み。v2候補を `boadgame2` へdeployしてはならない。
- 公開時はPages反映と `firebase/seri_v2_rules.json` のdeployを同日に連続して行う。
- `firebase.json` の既定Rulesは『魔女の調合』用であり、『影の競り』v2のdeployには専用configを使う。

## 2. 入口と正式ルール

- **solo・C1**：人間1人＋NPC3人、4面、8ラウンド、初期資金34。Firebase AuthとRealtime Databaseを使わない。
- **オンライン標準**：3人は3面・6ラウンド・資金30、4人は3面・8ラウンド・資金34。
- **オンラインC1**：3人は3面・6ラウンド・資金28、4人は4面・8ラウンド・資金34。
- **観戦**：共有情報だけを購読し、プレイ中のprivateと操作UIを持たない。終了後はreveal、順位、履歴を表示する。
- **series**：廃止。C2/C3などの実験モードはv2候補へ追加しない。

ルームコードは6文字。旧4文字コードと旧 `rooms` は使用しない。

## 3. solo・C1の互換仕様

soloとC1の正本はP0前のコミット `9c99921e5e139a29acf40a5e85b4053038c9b59f`。

- 面の値は各面 `-2..6`。
- C1の個人価値は「全4面の合計＋本人の盲点面をさらに2回加算」。盲点面は実質3倍。
- 開始席は `round % 4`。入札は1刻みで、最後に残った者が落札価格を支払う。
- 得点は残金＋落札した原石の個人価値。資金のある参加者が2人未満、または8ラウンド終了で終局。
- 同点は席順で安定して表示する。
- NPCは本人の盲点真値を判断へ使用せず、公開入札と見えている値から判断する。
- 旧正本に存在したsolo途中保存を、v2専用localStorageキーで維持する。onlineのUID・ルーム・privateとは混在させない。

## 4. v2データ構造

```text
seriRoomsV2/{6文字コード}/
  meta       hostUid, phase, n, rounds, money, nc, rule, version, createdAt
  seats      {seat}: uid, name, lastSeen
  game       round, phase, price, curSeat, starter, live, lastActionId
  money      {seat}: number
  actions    {seat}: id, type, round, price, createdAt
  private    {uid}: ownerUid, seat, lens, see, hostだけlots
  history    {round}: winnerSeat, price, values, createdAt
  reveal     lenses, createdAt
```

`totals`、`fired`、`series` は保存しない。C1の `see/{round}` はカンマ区切りで、本人の盲点1面だけを `x` とし、それ以外を数値で持つ。

## 5. 認証・権限・信頼境界

- Firebase匿名認証の `auth.uid` を席所有者IDにする。localStorageの自己申告値を権限に使わない。
- プレイヤーは自UIDの `private/{uid}` だけを読める。ホストは進行・採点のため全privateを読める信頼主体。
- 一般プレイヤーは自席 `actions/{seat}` だけを書き、game・history・revealを直接変更しない。
- 観戦者と非着席者はgame/actions/seats/privateへ書き込めない。
- Rulesは許可フィールド、型、文字数、値域、人数別設定、席所有、ホスト権限を検証する。
- ホスト不在中は採点と進行を停止し、同一匿名UIDの再接続後に再開する。
- 匿名認証は本人確認ではなく、ブラウザプロファイル単位の識別である。

## 6. 表示安全性

- DB値、プレイヤー名、NPC名、履歴は `createTextNode`／`textContent` 相当の安全なDOM APIで表示する。
- `innerHTML` へユーザー入力やDB値を連結しない。
- 観戦者はプレイ中の盲点、個人価値、他人のprivateを表示しない。
- 本番URLで `?emulator=1` を付けても、localhost／127.0.0.1以外ではEmulatorへ接続しない。

## 7. テストとローカル起動

```powershell
npm ci
npm run test:seri-v2-static
npm run test:seri-solo-c1
npm run test:seri-v2-rules
```

ブラウザ受入は同一オリジンに対し、独立BrowserContextまたは独立プロファイルを使用する。各環境の匿名UIDが異なることを確認する。Rules受入は本番と同じ `boadgame2-default-rtdb` namespaceになるよう、`npm run emulators:seri-v2` を使用する。

2026-07-14のローカル受入では、solo 8ラウンド、標準3人＋観戦6ラウンド、C1 4人＋観戦8ラウンド、ホスト／プレイヤー再接続、Permission denied、保存型XSS代表入力を確認済み。これは本番公開の承認ではない。

## 8. 公開手順の境界

1. 作業ツリーと対象コミットを固定し、全自動テストを再実行する。
2. 明示承認後にmainへ統合し、pushする。
3. Pagesでv2反映を確認する。
4. 対象Firebaseプロジェクトが `boadgame2` であることを再確認する。
5. 匿名認証を確認し、`firebase/seri_v2_rules.json` を変更せずdeployする。
6. 独立UIDで本番スモークテストを行う。

公開前にpush、main統合、Rules deploy、本番DB書込みを行わない。

## 9. 既知の制限

- ホストは全原石と全privateを読める。
- ホスト不在中は採点とラウンド進行が停止する。
- Rulesで許可された本人actionsの範囲まで、悪意ある着席プレイヤーの全操作を排除するものではない。
- v2は旧 `rooms`、4文字コード、旧キャッシュと非互換。
- GitHub管理画面のSecurity設定は、管理者ログイン状態で別途確認する。

## 10. 読み構造の検証結果と次の段階（2026-07）

観測位置の公開方法A（現行非公開）・B・B-lite・B-historyを比較し、標準4人卓を中心に600,000ゲームを検証した。続いて読み利益の原因分解90,000ゲームと、私的価値倍率×位置公開の2×2要因検証80,000ゲームを実施した。

- Bは推定精度を改善したが、局所利益・最終得点・勝率へ十分変換されなかった。
- B-liteは平均1.86個目で全員の位置が判明し、ほぼ早期B化した。
- B-historyで全位置に十分な履歴が揃ったゲームは44.4%で、情報が遅く弱かった。
- B2、B3、A2は採用しない。現行A3を維持する。
- 位置公開には小さな局所改善があったが、十分な利益や2倍時に特に強くなる交互作用はなかった。
- 以前の**現行A混合卓ではS2が勝率29.36%・平均得点41.94で最高**だったが、今回のA3では再現しなかった。履歴読みの優位は実装・対戦構成に敏感で、頑健性未確認。
- 2倍化は勝率だけでなく、平均得点・落札利益の低下、過払い率上昇、私的価値余剰の縮小も併記する。
- 負のシグナルから悪石を避ける経路は、人間テスト対象の低リスクな仮説として残る。

次は現行A3・標準4人・同じメンバーで3～5戦する人間プレイテスト。詳細は観測位置公開、読み利益変換、私的価値倍率、負のシグナル読みの各報告書を参照する。
