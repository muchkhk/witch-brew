# 『魔女の調合』秘匿性数値の再現 — 結果報告（2026-07-15）

対象: `docs/` 内の知見MD3ファイルに登場する『魔女の調合』の秘匿情報リーク数値
（旧方式で6儀式後に約7通り／現方式(B方式)で約438通り／選択行動のみで816→35通り 等）。

リポジトリのどこにも存在しないことが判明した（欠損リスト1）。本ディレクトリは、その根拠を
現行実装（V3プール・2026-07時点の `witch.html`）から独立に再構成する試みの記録である。

**witch.html は一切変更していない（読み取り専用で構造を読み取っただけ）。数値が食い違っても
実装を「直す」対応はしていない——記事・知見側が事実に合わせる。**

## 1. 定義の再構成（仮定）

### 1-1. 前提となるゲーム構造（witch.html から読み取り）

- 好みプール = 18枚（`POOL`）。1人の手札 = そこから3枚。候補空間 = C(18,3) = **816通り**。
- 3人プレイ・6儀式（`rounds = n*2`）が実測条件（`docs/ボードゲーム制作_プロセス検証知見.md`に明記）。
- 各儀式で「誰がどの坩堝(materials)を受け取ったか」は、着席者・観戦者を問わず**常に公開**されている
  （`game.history[].contents`/`piles`/`owners` は `game` ノードの一部で、`.read` は着席者・観戦者共通に許可されている。`reveal`のような終了後限定の制限は無い）。
  これは旧方式・現方式のどちらでも変わらない。**変わるのは個人の得点が見えるかどうかだけ。**
- witch.html の `SOLO` モード実装 (`soloScore()`) は今も `totals`（個人点の配列）を history に含めている。
  これは「旧方式」＝個人点を公開する版の直接的な実装上の裏付けである
  （ソロは秘匿不要なので、全部見せる旧仕様のデータ形のまま残っている）。

### 1-2. 「候補」の定義

観測対象を **特定の1人（席0）** に固定する（席は対称なので一般性を失わない）。
「6儀式後の候補数」= 「観測者が公開情報だけから、対象の816通りの手札のうちどれが本当の手札でも
矛盾しないか」を数えた数 —— 全列挙によって、観測される情報系列を再現できる手札の個数。

これは `docs/ボードゲーム設計論_理論編.md` §3-4 の「N回観測後に、秘匿情報の候補が何通りに絞れるかを
シミュレーションする」という記述と一致する。

### 1-3. 公開方式ごとの「観測者が知っていること」（複数解釈が生じた箇所を含む）

| 方式 | 観測者が知っている情報 |
|---|---|
| 個人点を全公開（旧方式） | 毎儀式の (a)対象が受け取った坩堝の中身 (b)対象の**正確な個人得点** |
| 発火した個数のみ公開 | 毎儀式の (a)坩堝の中身 (b)対象の**発火したレシピの枚数**（0〜3） |
| チーム合計のみ公開（現行B方式） | 毎儀式の (a)全員の坩堝の中身 (b)**チーム合計点のみ**（個人点は非公開） |
| 選択行動のみ | 毎儀式、対象が**手番者だった回**に限り、どの坩堝を選んだか（点数・個数は一切非公開） |
| 全好み1点固定 | 「個人点を全公開」と同じ観測だが、得点計算を全レシピ pt=1 に差し替えたもの |

**「チーム合計のみ公開」の定義で、実装前に誤りかけた点（重要）:**
最初、「他プレイヤーの手札は観測者にとって既知（真の値）」と仮定してチーム合計から対象の個人点を
逆算する実装を考えたが、これは**誤り**だと気づいた。他プレイヤーの手札が既知なら、チーム合計から
引き算するだけで対象の正確な個人点が求まってしまい、「個人点を全公開」と数学的に同一になり、
438通りという大きな差が説明できなくなる（実際438は7よりずっと大きいので、他プレイヤーの手札の
不確実性こそが秘匿を守っている）。
正しい定義は「観測者は他プレイヤーの手札も知らない」という前提で、**対象の候補手札Hについて、
他の(n-1)人に何らかの手札を割り当てれば全6儀式のチーム合計を再現できるか**（＝存在すればHは候補）
というjoint（同時）な整合性判定である。本スクリプトはn=3（他2人）についてこれを816×816の
総当たり（numpyで高速化）で厳密に解いている。

**「選択行動のみ」で仮定した観測者の推論ルール:**
対象が「自分の手札から見て一番得点が高い坩堝を、先読みなしで貪欲に選ぶ」という自己利益最大化
ポリシーで動いていると観測者が知っている、という前提を置いた（§2参照）。この前提がなければ
「選択」は手札の情報を一切含まない（ランダム選択なら何も漏れない）。妥当性確認として、
実際の行動生成もランダムにした対照実験を別途行った（下記）。

## 2. ゲームの行動モデルに関する仮定（重要・複数解釈の余地あり）

**witch.html には、対戦を自動で進行させるNPCの意思決定ロジックは実装されていない。**
`rationalReach`/`seqChoose`/`omniMax`/`bestForCut` は、実際のプレイ（人間が手番を操作する）が
終わった**後**に、resolver（ホスト）が「全知の計画者ならどこまで到達できたか」を採点するために
呼ぶ事後評価関数であり（`resolveScore()`内でのみ呼ばれる）、対戦中の意思決定には使われていない。
オンライン対戦・ソロ検証モードのどちらも人間が全手番を操作する設計である。

したがって「6儀式のゲームを何百回もシミュレートする」ためには、行動を決めるモデルが別途必要になる。
本スクリプトは以下を採用した（このモデル選択が、再計算値がMD記載値と一致しない主要因の一つになりうる）:

- **切り役**：帯の切り方をランダムに選ぶ（cutSetsから一様乱数）。切り役は自分の坩堝を選べず
  余りを受け取るため、切り方で自分の得点を直接操作する動機をモデル化するのは投機的すぎると判断した。
- **手番者**：自分の手札だけを見て、残っている坩堝のうち自分の scorePile が最大になるものを選ぶ
  （先読みなし・貪欲な自己利益最大化）。同点は坩堝インデックスが小さい方。

## 3. 再計算値とMD記載値の対照表

シード: `MASTER_SEED = 20260715`（スクリプト内で固定）。3人・6儀式。全列挙は候補フィルタ部分
（816候補 × 観測系列との整合判定）に限る。ゲームのトラジェクトリ生成（帯・切り方・手番の選択）は
§2の仮定に基づく乱数シミュレーションであり、多数回の独立試行の平均で近似している
（全列挙は不可能な規模のため。詳細は§2）。

| # | 項目 | MD記載値 | 再計算 平均 / 中央値（n試行） | 一致 |
|---|---|---|---|---|
| 1 | 個人点を全公開 | 約7通り | **4.55 / 2**（n=3000） | **不一致**（オーダーは近いが、中央値は約1/3.5） |
| 2 | 発火した個数のみ公開 | 約2通り | **10.98 / 7**（n=3000） | **不一致・向きが逆転**（MDでは全公開より狭いはずが、再計算では全公開より広い） |
| 3 | チーム合計のみ公開 | 約438通り | **17.69 / 10**（n=100） | **大きく不一致**（約25〜44倍小さい） |
| 4 | 選択行動のみ（816→35） | 35通り | **117.72 / 94**（n=3000） | **不一致**（約3〜4倍大きい） |
| 5 | 全好み1点固定 | 約16通り | **10.98 / 7**（n=3000、#2と数学的に同一値） | **不一致**（§5-3参照） |
| 6（対照） | 選択行動・ランダム手番者 | （MD記載なし・自作の妥当性確認用） | 23.84 / 7（n=1000） | — |

（詳細な分布 min/max/stdev は `results.json` を参照）

### 3-1. それでも残る定性的な結論

数値は軒並み不一致だが、**方式間の大小関係（定性的な結論）の一部は再現できた**：
「チーム合計のみ公開」（平均17.69）は「個人点を全公開」（平均4.55）より**約3.9倍広く**、
B方式が旧方式より秘匿を守るという記事の核心的な主張の**方向性**は本スクリプトでも支持された。
ただし倍率は記事の「7→438＝約63倍」とは大きく異なり（本スクリプトでは約3.9倍）、
**「読めるが特定不可」と言えるほどの余地（438通り）が実際にあるかは、この再現では確認できなかった**
（17.69通りは「読めるが特定不可」と呼ぶには心もとない数字である）。

### 3-2. fired_count の逆転について

MDは「発火した個数のみ公開」の方が「個人点を全公開」より**狭い**（約2 < 約7、より特定に近い）と
報告しているが、本スクリプトでは**逆**（10.98 > 4.55、個数のみの方が広い＝特定しにくい）になった。
直感的には「正確な点数」は「発火した個数」より真に多くの情報を含む
（点数はどのレシピが発火したかまで区別しうるが、個数はそれを潰す）ため、
全公開の方が個数のみより狭くなるのが自然であり、**本スクリプトの結果の向きの方が理論的に妥当**だと
考えられる。MDの「約2」がどのような前提で計算されたのかは、現存する資料からは再構成できなかった。

## 4. 再現できなかった／定義を再構成できなかった数値

- **点数を型ごとに揃える → 6R後8通り**：MDのこの実験は「型ごとに点数がバラバラだった旧プール」に
  対する改善実験だと推測される。しかし現行 `POOL`（V3）は既に型ごとに点数が統一されている
  （pair=4, cnt2=5, solo=3, abs2=3, abs=2 のいずれも型内で完全に同一）。つまり現行実装で
  「型ごとに揃える」を行っても**何も変化しない（全公開と同一の結果になるはず）**。
  MDが指す旧プールの具体的な点数配分がリポジトリ中に見当たらないため、この数値は再現不能と判断した。
- **粗い4段階表示（低/中/高/負）→ 6R後18通り**：「負」の得点バケットが存在する前提だが、
  現行 `scoreRecipe` は全レシピ型が0点かpt(正)のいずれかを返す設計であり、**負の得点は発生しない**。
  これもMDが指す仕様が現行実装より古い（負の得点を持つレシピ型が過去に存在した可能性がある）と
  判断し、4段階の境界を恣意的に決めてまで再現するのは不誠実と考え、見送った。

## 5. witch.htmlの実装とMDの記述が食い違っていた箇所

1. **NPCの自動意思決定ロジックが実装に存在しない**（§2参照）。MDの「実測」という言葉は、
   本リポジトリの外（別のスクリプトや、削除された過去のツール）で行われたシミュレーションを
   指していると考えられる。再現に必要な行動モデルの詳細（切り方の決め方・手番の決め方）は
   MDのどこにも明記されておらず、本スクリプトはゼロから設計し直した（§2で仮定として明記）。
2. **「型ごとに揃える」「粗い4段階（負を含む）」が現行POOLの構造と整合しない**（§4参照）。
   V3プール（現行）へのリバランス前の、旧い点数体系を前提にした実験だと考えられる。
3. **「発火した個数のみ公開」と「全好み1点固定」は、現行POOLでは数学的に同一の指標になる**
   （§3-2参照）。現行 `scoreRecipe` はレシピ型を問わず「条件を満たせば pt 点、
   満たさなければ 0点」の二値なので、全レシピの pt を1に統一した合計点＝ちょうど発火した
   レシピの枚数になる。したがって「発火した個数のみ公開」(MD:約2通り)と
   「全好み1点固定」(MD:約16通り)が**別々の数値として**MDに記載されていること自体が、
   MDの想定するレシピ・得点体系が現行実装と異なっていたことを示す状況証拠だと考えられる
   （本スクリプトでは両者とも同じ10.98/7という結果になった）。

## 6. 実行方法

```bash
cd sim/witch_privacy
python witch_secrecy_repro.py
# results.json に平均・中央値・分布のサマリが出力される
```

依存: Python 3.10, numpy（`team_total_joint` の高速化にのみ使用）。
決定論性: `MASTER_SEED = 20260715` から派生した `random.Random` のみを使用。同一環境なら再実行しても
`results.json` の数値は完全に一致する（実際に確認済み）。
