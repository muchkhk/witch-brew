## 1. プロジェクトの目的

本プロジェクトは、**単に動作するゲームを作ることではなく、人間同士で繰り返し遊びたくなるオリジナルボードゲームを設計・検証・公開すること**を目的とする。

同時に、成功だけでなく失敗の過程も記録し、次の作品で再利用できる設計知見・検証手法・実装技術として蓄積する。

成果物は主に次の2種類である。

1. 単一HTMLを中心とした、ブラウザで遊べるゲーム
2. 設計・検証・実装の知見をまとめた文書群

## 2. 最上位原則

### 2-1. 「動く」より「面白い」を優先する

技術的に実装可能であっても、ゲームの核を弱める機能は追加しない。

完成度の高いUI、豊富な機能、複雑なNPC、厳密な数値調整は、それ自体を目的にしない。すべてはプレイヤーの体験に奉仕しなければならない。

### 2-2. プレイヤーに意味のある選択を残す

本プロジェクトの設計上の背骨は、次の一文である。

> **意味のある選択を、リプレイに耐える形で、時間を通じて途切れさせず、テーマ上の意味を伴わせる。**

UIは判断を補助してよいが、最善手の提示、判断の自動化、推論の肩代わりをしてはならない。

### 2-3. 理屈より実際の体験を優先する

シミュレーション上の均衡、期待値の接近、NPCの勝率は、ゲームが面白いことの証明ではない。

特に心理戦・読み合い・疑心暗鬼を核とする作品では、人間同士の実プレイを最終判定とする。

### 2-4. 失敗を隠さず、死因を残す

中止した案も削除せず、何が機能し、何が死因だったかを記録する。

失敗記録は未完成品ではなく、次の設計判断を改善するための正式な成果物である。

## 3. 設計判断の規律

### 3-1. 設計順序

原則として、次の順で決める。

1. 主軸となる楽しさ
2. プレイヤー間の構造
3. 核となる判断
4. メカニクス
5. テーマとの整合
6. 対応人数
7. 数値
8. UI
9. 実装

上流が未確定のまま下流を作り込まない。

### 3-2. 核と対症療法を分ける

新しい要素を追加するときは、必ず次のどちらかに分類する。

- **核を強める要素**
- **症状だけを抑える対症療法**

対症療法が3つ以上積み重なった場合、追加実装を止め、土台や核に構造的な問題がないか再診断する。

### 3-3. 違和感を軽視しない

「何か変」「どちらでもいい」「作業に感じる」「見づらい」といった感覚的な報告は、構造的問題の兆候として扱う。

対策を実装する前に、可能な限り測定・再現・比較を行う。

### 3-4. 設計中の作品を勝手に実装しない

設計が確定していない作品は、正式な実装指示が出るまでコード化しない。

探索中の案、保留案、中止案を、実装担当の判断で復活・拡張してはならない。

## 4. 実装判断の規律

### 4-1. 単一HTMLを基本とする

各ゲームは、ソロ検証・オンライン対戦・観戦・ルール説明を可能な限り同じHTMLに統合し、二重管理を避ける。

ただし、検証用の独立スクリプトはゲーム本体と分離してよい。

### 4-2. 秘匿情報は「隠す」のではなく「送らない」

本来見えてはいけない情報を全クライアントへ配信し、UIだけで隠す設計は禁止する。

完全な秘匿が技術上困難な場合は、限界を明記し、身内プレイ前提などの運用条件を隠さない。

### 4-3. 権限は不正防止のために使う

操作権限は、正当でない操作を防ぐために設ける。

ゲーム進行を不必要にホストへ集中させない。誰が押しても安全な操作は、原子的更新やフェーズガードで二重実行を防ぎ、全員が進行できる設計を優先する。

### 4-4. エラーと同期状態を見せる

通信エラーを握りつぶさない。

オンライン対戦では、接続状態・最終同期・版不一致・操作失敗をプレイヤーが認識できるようにする。

## 5. 情報源の優先順位

仕様や記述が食い違った場合は、次の順で正誤を判定する。

1. **現行の実装コード**
2. **`PROJECT.md`**
3. **`CLAUDE.md`**
4. **`docs/*.md`**
5. **検証スクリプトと結果**
6. **`docs.html`**
7. **現在地メモ・導入手順・その他補助資料**

ただし、コードが設計意図と矛盾している場合、コードを無条件に正当化してはならない。  
「現在どう動くか」はコードを正とし、「どうあるべきか」は本憲章・`CLAUDE.md`・正式設計文書を照合して判断する。

`docs.html` は生成物であり、仕様の原本ではない。

## 6. 正式名称とファイルの役割

### 6-1. 正式名称

以下を正式名称とする。

| 正式名称 | 旧名称 |
|---|---|
| `seri.html` | `kage_no_seri.html` |
| `witch.html` | `witch_online.html` |
| `firebase_rules.json` | `firebase_rules_seri.json` |

旧名称は古い資料を読むための互換情報としてのみ扱い、新しい文章・コード・指示書では使用しない。

### 6-2. 主なファイル

| ファイル | 役割 |
|---|---|
| `PROJECT.md` | プロジェクトの憲章・最上位判断原則 |
| `CLAUDE.md` | 開発規律・現在地・作業手順 |
| `index.html` | 公開ゲーム一覧・各作品への入口 |
| `seri.html` | 『影の競り』本体 |
| `witch.html` | 『魔女の調合』本体 |
| `firebase_rules.json` | 『影の競り』用Firebaseルール |
| `docs/*.md` | 設計・検証・実装知見の原本 |
| `docs.html` | `docs/*.md` から生成される閲覧用ページ |
| `tools/add_toc.py` | Markdown目次の生成・更新 |
| `tools/build_docs.py` | `docs/*.md` から `docs.html` を生成 |
| `現在地メモ_新チャット用.md` | 新しい会話へ現在地を渡す補助資料 |

### 6-3. 文書生成

`docs.html` を直接編集してはならない。

```text
docs/*.md
  ↓ tools/add_toc.py
目次更新
  ↓ tools/build_docs.py
docs.html生成
```

設計・実装上の学びは、該当する原本MDへその場で書き戻す。「あとで書く」は原則として認めない。

## 7. 設計・実装・検証の役割分担

### 7-1. ChatGPT／設計チャット

主に次を担当する。

- ゲームの核、構造、体験、テーマの検討
- 既存知見との照合
- 懸念の先回り
- 対症療法の検出
- プレイテスト結果の診断
- 実装前の仕様確定
- Claude Codeへ渡す指示書の作成

設計チャットは、依頼された案を無条件に肯定せず、面白さを損なう懸念があれば先に指摘する。

### 7-2. Claude Code／実装担当

主に次を担当する。

- 確定仕様の実装
- バグ修正
- リファクタリング
- テスト・検索・差分確認
- 文書と実装の整合確認
- 版番号と公開物の更新
- コミット準備

設計が未確定の部分を独自判断で作り込まない。必要な設計判断は設計側へ戻す。

### 7-3. シミュレーション

主に次を確認する。

- 支配戦略
- 死に選択肢
- 数値破綻
- 発生率
- 極端な不公平
- 仮説間の比較

次の判定には使用しない。

- 人間が面白いと感じるか
- 心理戦が成立するか
- ブラフが感情的に機能するか
- 再戦したくなるか

### 7-4. 協働の作業場所

複数のAI（Claude Code・ChatGPT Codex 等）が関わる場合も、**単一の正式なリポジトリ**で作業する。

- 日付付きの一時フォルダなどへ別クローンを作り、それを継続的な作業元にしない。
- 並行して別ブランチを編集する必要があるときは、無関係な複製を作らず **Git worktree** を使う。
- 同一のワーキングツリーを、複数のAIが同時に編集しない。

（このPC固有の正式リポジトリのパスは `CLAUDE.md §10` に記す。）

## 8. 変更とリリースの原則

### 8-1. 変更は関連箇所を一組で扱う

ゲームHTMLを変更した場合、必要に応じて次を同時に更新・確認する。

- `VER`
- `PAGE_VER`
- 画面上の版表示
- `index.html` のリンククエリ `?v=`
- Firebase設定
- キャッシュ無効化
- `.nojekyll`
- `CLAUDE.md`
- 該当する引き継ぎMD
- `現在地メモ_新チャット用.md`
- `docs.html`

### 8-2. 必須確認

変更後は、少なくとも次を確認する。

- ソロ
- オンライン対戦
- 観戦
- ルール説明
- 複数人でのみ起きる状態遷移
- リロード・再接続
- 版不一致
- 終了・再戦・初期化
- エラー表示
- モバイル表示

必要に応じてヘッドレス試験、複数端末試験、モンテカルロ試験を行う。

### 8-3. 文書整合

次の語を検索し、古い記述が残っていないか確認する。

```text
未実装
未対応
未検証
TODO
旧ファイル名
旧バージョン
```

Markdownを更新した場合は、次を実行する。

```bash
python3 tools/add_toc.py docs/
python3 tools/build_docs.py
python3 tools/add_toc.py --check docs/
python3 tools/build_docs.py --check
```

### 8-4. Push承認

変更内容、検証結果、懸念、差分を先に提示する。

ユーザーの明示的な承認なしにリモートへpushしない。  
強制push、履歴破壊、無断の大規模削除は禁止する。

---

## 9. 禁止事項

次を原則禁止とする。

- ゲームの面白さを確認せず、機能の多さを完成度とみなす
- 設計中の作品を勝手に実装する
- 対症療法を無制限に積み重ねる
- NPCや数値試験だけで「面白い」と結論づける
- 秘匿情報を全員へ送ってUIだけで隠す
- 権限確認をUI表示だけに任せる
- 通信エラーを握りつぶす
- ブラウザ標準の `alert()`、`confirm()`、`prompt()` に依存する
- 全体 `innerHTML` 再描画で入力・スクロール・フォーカスを壊す
- 進行操作を理由なくホスト専用にする
- `docs.html` を直接編集する
- 古いファイル名を新規資料で使用する
- Firebaseプロジェクトを作品間で共用する
- 承認なしでpushする
- 強制push、`git reset --hard`、無断の履歴破壊を行う

## 10. 現在の主要作品

### 10-1. 『魔女の調合』

正式ファイルは `witch.html`。

秘密の好みを持つプレイヤーたちが、無言で素材を分割・選択する協力ゲーム。  
オンライン対戦、ソロ検証、観戦、ルール説明を統合する。

Firebaseプロジェクトは他作品と分離する。

### 10-2. 『影の競り ― 盲目の宝石商たち』

正式ファイルは `seri.html`。

核は次の判断である。

> **「あいつが高く入れた。あいつには何が見えている？　追うか、降りるか。」**

情報非対称、競り、私的価値、ウィナーズ・カースを用いた読み合いのゲームであり、プロジェクトで初めて人間プレイ上の「面白い」に到達した作品として扱う。

現在は、機能追加より人間同士の継続的な実プレイ検証を優先する。

## 11. この文書の変更

`PROJECT.md` は最上位の入口文書であるため、頻繁な作業ログや個別バグを記録しない。

変更対象は、次のようなプロジェクト全体に関わる決定に限る。

- 目的
- 最上位の設計思想
- 情報源の優先順位
- 正式名称
- 役割分担
- リリース規律
- 禁止事項

個別作品の仕様、現在の実装状況、既知の不具合は `CLAUDE.md` または各引き継ぎMDへ記録する。

この文書と他の資料が矛盾した場合は、矛盾を放置せず、その場でどちらを更新すべきか判定する。
