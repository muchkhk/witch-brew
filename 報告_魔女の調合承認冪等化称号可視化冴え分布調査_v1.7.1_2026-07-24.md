## 1. Aで採った旧UID特定方法・ルール変更

### 旧UID特定方法：`prevUid`フィールドを追加した

指示書が示唆した通り、`takeoverRequests/{uid}`スキーマに`prevUid`（申請時点でその席に
座っていたUID）を追加した。理由：席のUIDは承認の1段目（transaction）で既に新UIDへ
差し替わるため、2段目（好み移行）が中断された状態から回復しようとすると、「移行元の
UIDが何だったか」を示す情報が席データ自体からは失われている。申請ノードに記録しておく
以外に、この情報を安全に復元する手段がなかった。

`prevUid`は申請者（`submitTakeoverRequest`）が、選んだ席の**申請時点での現在のUID**を
`TAKEOVER.seats[seat].uid`から読み取って記録する。ホストが検証するのではなく申請者自身が
記録する設計にしたのは、申請時点でその情報を持っているのは申請者のクライアントだけ
（ホストは後から`ST.seats`を見るが、承認が実際に押される頃には既に変わっている可能性が
ある）だからである。

### Firebaseルール変更（追加のみ）

```diff
-            ".validate": "newData.hasChildren(['seat','name','ts'])",
+            ".validate": "newData.hasChildren(['seat','name','ts','prevUid'])",
             "seat": { ".validate": "newData.isNumber() && newData.val() % 1 === 0 && newData.val() >= 0 && newData.val() <= 3" },
             "name": { ".validate": "newData.isString() && newData.val().length <= 24" },
             "ts": { ".validate": "newData.isNumber()" },
+            "prevUid": { ".validate": "newData.isString() && newData.val().length <= 128" },
             "status": { ".validate": "newData.isString() && (newData.val() === 'pending' || newData.val() === 'rejected')" },
```

既存の`seats`・`privateRecipes`・`takeoverRequests`の読取・書込制限は一切変更していない
（`prevUid`フィールドの追加のみ）。承認の冪等化ロジック自体（回復パス判定・好み移行の
やり直し）は、既存のホスト権限（`seats`親・`privateRecipes`親の書込権限）の範囲内で完結
しており、追加のルール緩和は不要だった。

### コンソール適用手順・推奨順序

前回（v1.7）と同じ理由・同じ順序を推奨する：**ルール適用→push**。
`prevUid`は既存フィールドへの追加であり、旧クライアント（v1.7以前）が書く
`takeoverRequests`データには影響しない（旧クライアントは`prevUid`を送らないため、
新ルールの`hasChildren(['seat','name','ts','prevUid'])`を満たせず**書込が拒否される**
点に注意——ただし、これは「ルール先行適用時に旧v1.7クライアントの新規申請が拒否される」
というだけで、それ以外の機能（対戦・観戦・ソロ等）には無影響。v1.7.1のpushとルール適用を
なるべく近接して行うことを推奨する（v1.7との間に長い空白を作らない）。

## 2. 回復パスが誤発火しないことの確認

Emulator受入試験で以下を確認した：

1. **正常な初回承認**：`online:false`の席への新規申請→承認、で従来どおり席transaction→
   好み移行が実行され、正しく完了することを確認（回復パスの分岐条件
   `seatData.uid===uid`は、承認前は必ず偽になるため素通りする）。
2. **中断状態からの回復**：席は既に新UIDへ差し替え済みだが`privateRecipes`が未移行、
   という状態をDB直接操作で再現し、承認ボタンを再度押すと（`seatData.uid===uid`が
   真になるため）席transactionを再実行せず、好み移行のみをやり直して完全な状態へ
   収束することを確認した。
3. **却下・再申請**：`rejectTakeover`・`retryTakeoverChoice`のコードパスは今回変更して
   おらず、影響がないことをコードレビューで確認済み（回復パスの判定は`approveTakeover`
   冒頭にのみ追加しており、却下・再申請の関数は呼び出されない）。

いずれもEmulator上の実操作、または静的テスト（`tests/witch_static.test.mjs`の
v1.7.1-Aテスト）・ルールテスト（`tests/witch_rules.test.mjs`の中断状態回復テスト）で
確認済み。全テスト合格（静的27件・ルール32件）。

### 所見（閾値調整については実装せず、判断材料のみ提示。楽観的フレーミングはしない）

1. **現行の共有閾値（人数・モード共通）のもとでは、上位2称号（熟達82%・大魔女92%）の
   到達率に、人数・モードによる大きな差がある。** 通常モードで≥82%到達率は3人79.4%・

   上級モードではこの差がさらに開く。下位2称号（見習い58・一人前70）はどの条件でも
   ほぼ全員が到達するため、**問題は上位2称号に集中している**。
2. **上級モードは全条件で到達率を大きく下げ、人数が多いほど下げ幅も大きい**
   （平均で3人-7.1pt・4人-9.8pt）。
3. 「4人を3人に揃えるための閾値シフト」は、82・92の境界で通常モード-2.9〜-5.0点、
   上級モード-7.4〜-8.9点と、無視できない大きさだった。
4. **プール公開（v1.7）はこの差を縮めるとは考えにくい。** 読みの精度向上は人数・モードに
   よらず一律に効くと考えられ、既存の構造差を放置したまま通算冴え%全体を底上げする
   方向に働く可能性が高い（＝現行閾値、特に92%が相対的に緩くなる可能性）。
5. **人数別・モード別閾値にすべきかは、この報告だけでは決められない。** 数値上は

   がある。プール公開後の人間プレイテストで実測分布を見てから、このシミュレーション

   触らず閾値側で調整する余地を残す）。

## 4. Bのゲージ表示

「終了画面」（オンラインプレイヤーの`endHTML()`）と「観戦者の終了後表示」
（`spectatorHTML()`の`game.phase==="end"`分岐）の両方に、称号アイコン・称号名の直下へ
横ゲージを追加した。

**構成**：
- 駆け出し（0-58%）・見習いを抜けし者（58-70%）・一人前の魔女（70-82%）・
  熟達の魔女（82-92%）・大魔女（92-100%）の5帯を、幅を境界値の差に比例させた
  横帯として描画（例：駆け出し帯は0〜58なので全体の58%の幅）。
- 各帯の色は駆け出し（暗い紫がかった灰色）→大魔女（蝋燭色）へ向かって明るくなる
  グラデーション状の配色とし、既存の世界観（蝋燭・月・明朝体）に合わせた。
- 帯の境界（58/70/82/92）に目盛りラベルを表示。
- 自チームの最終冴え%の位置に、蝋燭色の三角マーカーを表示（`title`属性でホバー時に
  「冴え XX%」を表示）。

**閾値のハードコード重複を避ける実装**：既存の`rankFor(pct)`関数が使っていた
`if(pct>=92)...`という分岐チェーンを、`RANKS`という単一の配列（各帯の`min`値・
称号名・アイコン・台詞・ゲージ色を持つ）へ置き換えた。`rankFor()`もゲージ描画関数
`titleGaugeHTML()`も、この`RANKS`だけを参照する。閾値を将来変更する場合は`RANKS`の
`min`値を書き換えるだけでよく、ゲージの帯幅・目盛りラベルは自動的に追随する
（静的テストで、ゲージ描画コード内に58/70/82/92等の数値が再ハードコードされて
いないことを確認済み）。

Emulator上で、冴え86%（熟達の魔女帯）の終了済みゲームを用意し、プレイヤー・観戦者
双方の終了画面でゲージのDOM構造（5帯の幅比率・色・マーカー位置・目盛り）が設計通りに
描画されることを確認した。

## 5. 実装中に気づいた懸念・逸脱・未解決事項

- **既存の「終了画面の%メーターバー」（`.meterwrap`/`.meter`、単色の塗りつぶしバー）は、
  称号ゲージに置き換えた**（両方表示すると同じ情報が重複するため）。この判断は指示書に
  明示されていないため、逸脱として記録する。ラウンド単位の冴え%表示（採点結果画面の
  メーター）は変更していない——あくまで「通算の冴え・称号」の文脈でのみ置き換えた。
- 席引き継ぎの中断状態回復は、「好み移行のみが中断された」ケースを対象にしている。
  「席transaction自体が中断された」（＝seats/{seat}.uidがまだ書き換わっていない）
  ケースは、従来のレースガード付きtransactionがそのまま再試行されるため、追加対応は
  不要（transactionは何度再実行しても安全という既存の設計のまま）。
- R-2のシミュレーションは「合理」NPCのみで実施した（既存の呑気NPC・気分屋NPC等の
  ゆらぎ戦略は、witch.htmlのNSOLO専用エンジンであり、rationalReachが表す「合理チーム」
  とは前提が異なる別物のため、比較対象として加えなかった。指示書の「なければ合理のみで
  よい」に従った）。
- R-2のPythonポートは、`bestAssign`（witch.htmlでは全順列を総当りする実装）を、数学的に
  完全に等価な直接計算（残り坩堝のE_sb総和）に置き換えて高速化した（素朴な移植では
  1200試行×4条件に約9時間かかる見込みだったため）。値は理論上100%同一だが、浮動小数点の
  加算順序が変わるため、ごく稀に最終桁での丸め差がタイブレークを揺らす可能性がある
  （script内コメントに詳細と影響範囲を記載。1000試行以上の集計値には実務上影響しない）。
- R-2結果と既存引用値（特に上級モード）の食い違いは、旧引用値の算出根拠が
  リポジトリのどこにも残っていなかったために生じた「答え合わせ不能」な状況である。
  今回のR-2は算出根拠（スクリプト・シード・試行数）を明記しているため、今後は
  こちらを正とすることを提案する（引き継ぎ資料に注記として反映済み）。
