## §0. 絶対規律（これを破るくらいなら、手を止めて質問する）

### 0-1. 面白いゲームを作ることが、第一条件
コードが動くこと・数値が揃うことは目的ではない。**「これは本当に面白さに寄与しているか」** を、機能を足すたびに問う。

### 0-2. 指示された内容より良い案があれば、実装せずに提案する
言われた通りに作る前に、**より良い案があるなら手を止めて提案すること。** 黙って作らない。

### 0-3. 懸念は、聞かれる前に言う
「たぶん大丈夫」で進めない。**リスク・副作用・壊れそうな箇所は、こちらから先に列挙する。**

### 0-4. 【最重要】対症療法カウンタ
機能を1つ追加するたびに、**「これは核か、対症療法か」を明示的に分類して宣言する。**

- 対症療法が **3つ溜まったら、実装を止めて棚卸しを提案する。**
- ⚠️ **机上の分類は当てにならない。** 過去に「対症療法だろう」と判断した機構（監査対応）を実際に外したら、ムーン到達率が45% → 2〜8% に崩壊した。**「これは削れるはず」は、必ず削って測ってから確定する。**

### 0-5. 一般化できる学びは、その場でMDに書き戻す
**「あとで書く」は「書かない」。** バグを直したら、その場で該当MDに追記する。

### 0-6. 判断の「補助」は足してよい。「肩代わり」は禁止
履歴・状態可視化・残リソース表示 → **足すほど良い。**
最善手サジェスト・自動プレイ・「可能な範囲」の自動表示 → **楽しさを殺す。実装しない。**

### 0-7. 設計フェーズにも、同じ規律が適用される
**§0-4（対症療法カウンタ）は実装だけの話ではない。設計の議論でも、機構は静かに積み上がる。**

設計チャットで新しい機構を足すたびに、**「核か、対症療法か」を宣言する。**
そして **「外して測れるもの」と「外すと壊れるもの」を分けて記録する。**
> ⚠️ 机上の分類は当てにならない（§0-4）。**「これは削れるはず」は、削って測ってから確定する。**

**議論のときの3規律（過去の失敗が生んだもの）:**
1. **理屈より、設計者が固まった瞬間を優先する**（『共犯』の最大の教訓）
2. **「核が弱い」という検証結果が出たら、まず「核を実装したか」を疑う**（『影の賽』で実際に踏んだ）
3. **一般化できる学びは、その場でMDに書き戻す**（§0-5）

**土台が決まるまで、メカニクスを足さない。** 構造 → 数値検証 → 実装、の順序を飛ばさない。

## §2. リリース手順（HTMLを触ったら、必ず全部やる）

### 2-0. ★ push する前に、必ず人間の承認を取る

> **GitHub Pages は push した瞬間に公開される。壊れた版が、そのまま本番になる。**
> しかも対戦中のプレイヤーがいれば、版の食い違いバナーが出て卓が止まる。

- **無断で `main` に push しない。** 変更内容（diff）を人間に見せ、承認を得てから push する
- 大きな変更は **ブランチを切って PR** にする。`main` 直 push は小さな修正に限る
- push 後は **必ず本番URLを開いて、画面隅のバッジが新しい版になっていることを確認する**

> これは §0-2（言われた通りに作る前に提案する）の、書き込み権限版である。

### 2-1. 手順

1. **`FIREBASE_CONFIG` を埋め込む**（§1の表の projectId。空のままだと「未設定」表示になる）
2. **版番号を上げる** ―― `VER` / `PAGE_VER` / **画面隅のバッジ** / `index.html` のリンククエリ `?v=` の **4箇所すべて**
3. `.nojekyll` があるか確認
4. `<head>` のキャッシュ無効化メタが残っているか確認
5. **★ このファイルの §7「現在地」を更新する** ―― 版番号・実装済み項目・未検証リスト・既知の限界。
   **版を上げたのに §7 が古いままなら、CLAUDE.md が嘘をつくことになる。**
6. **★ 引き継ぎ資料MD（`影の競り_オンライン版_引き継ぎ資料.md` 等）を、同じセッション内で更新する。**

> ⚠️ **実際に起きた事故:** v1.4 で連戦を実装したのに、引き継ぎ資料には「同一卓での連戦は未実装」と
> 残ったままだった。**知見MDは「真実」とされているので、古い記述は嘘として次のセッションに伝播する。**
> **機能を1つ実装したら、その機能を「未実装」と書いている箇所を grep で探して潰す。**
> ```bash
> grep -rn "未実装\|未対応\|未検証" *.md   # 版を上げたら必ず一度は目を通す
> ```

### なぜ版番号が必須か
オンライン対戦では、**版が食い違うと「原因不明の不具合」になる。** DBの `meta.ver` と自分の `VER` を突き合わせ、違えば赤バナーを出す仕組みが既に入っている。**版を上げ忘れると、この防御が無意味になる。**

**同一版番号での上書きを禁止する。** `こたえあわせ` v0.7では、「§3保留版」「§3完了版」「秘匿テスト強化版」という中身が明確に異なる複数のビルドが、すべて同一の `VER` 文字列（`v0.7`）のまま公開され続けた。結果、公開されているファイルがどの時点のビルドかを版バッジだけでは判別できない状態が生じた。**実装内容が変わったら、たとえパッチ的な小さい修正であっても、`VER` を必ず動かす**（`v0.7` → `v0.7.1` 等の枝番でもよい。「据え置いたまま複数回上書き公開する」ことだけを避ける）。

**ブランチ名に版番号を入れない。版は `VER` 定数で管理する。** 理由：同じ事実を2箇所に持つと必ず片方が陳腐化する（`docs_meta.json` の件と同型）。

### Firebase プロジェクトは作品ごとに分ける
『魔女の調合』v1.3は `roomsV2/{6文字コード}`、『影の競り』は別スキーマを使うが、同一プロジェクトではルールと障害範囲を共有するため、作品ごとにFirebaseプロジェクトを分ける。
確認方法: **`databaseURL` が作品ごとに違うこと。**（「アプリを追加」では分離されない ―― DBはプロジェクトに紐づく）

## §3. 検証の型（実装したら、必ず回す）

### 3-1. 配線漏れの機械的検出 ★最も漏れる
新機能を片方のモード（ソロ）に実装し、**オンライン側への配線を忘れる**事故が繰り返し起きている。
**構文エラーにならない。関数が「存在するが呼ばれていない」だけ。**

```bash
grep -c 'function dealAdvanced' seri.html   # → 1（定義あり）
grep -c 'dealAdvanced(' seri.html           # → 1（定義のみ＝呼び出しゼロ！）
```
> **定義数と呼び出し数を比較する。定義数 == 呼び出し数 なら、呼ばれていない。**

### 3-2. 機能インベントリ照合（統合・リファクタ後は必須）
```
[ ] ゲームロジック（採点・配布・評価）
[ ] ルール説明・ヘルプ・ツールチップ
[ ] イラスト・装飾・アニメーション
[ ] 効果音
[ ] 対戦 / ソロ / 観戦 の全モードで、上記が動くか
[ ] 新機能を、全モードに配線したか   ← ★最も漏れる
```
過去に、統合作業で **ルール説明ページ全体・トップのSVG・上級モードのオンライン実装が静かに消えた。**

### 3-3. ヘッドレス通し検証
`node` でゲームを頭から終わりまで自動で回し、**「Nラウンド完走するか」を検証する。**
状態遷移の初期化漏れ（`phase = "cut"` を忘れる等）は、これでしか見つからない。

### 3-4. モンテカルロ（バランス検証）
**感覚で数値を決めない。** Node.jsでロジックを抽出し、大量試行して分布を見る。

**通常の期待値比較だけでは不十分。以下2軸を必ず追加する:**
- **「1位のみ勝者」形式での勝率集計** ―― 支配戦略の有無を暴く
- **「全員が同じ戦略を取ったら」の集団合計比較** ―― 狙ったゲーム理論構造（Stag Hunt等）が本当に機能しているかを検証できる

**ペース配分は、バランスとは別に測る。** 「危険水域に到達するのは平均何ラウンド目か」を測らないと、**序盤の間延び**に気づけない（第2作で実際に見落とした）。

### 3-5. リファクタで壊れる3パターン（既知）
| # | 症状 | 対策 |
|---|---|---|
| 1 | 関数抽出時に、**変数宣言を巻き込んで消す**。暗黙のグローバルとして動き、発見が遅れる | 置換後に `grep -c 'let localCuts'` （1であるべき） |
| 2 | **状態遷移の初期化を1つ忘れる** | §3-3 のヘッドレス通し |
| 3 | 型を追加したが、**一部の分岐が未対応**で `undefined` | 型を追加したら、その型に触る全関数を列挙して潰す |

## §4. 実装の禁止事項（過去に実際に踏んだ地雷）

### 4-1. `confirm()` / `alert()` / `prompt()` を使わない
**サンドボックス iframe では、ダイアログを出さずに黙って `false` を返す。** 「ボタンが反応しない」の典型原因。

- **軽い操作** → 確認を出さず即実行
- **重い操作**（退室など） → **2段階クリック**（1回目で文言と色が変わり、4秒で自動キャンセル）

### 4-2. 非同期処理の入口に、必ずガードを入れる
```js
function npcTurn(){ if(!S) return; /* ... */ }
```
状態を `null` にした後、残っていた `setTimeout` が発火してクラッシュする。

### 4-3. 全体 `innerHTML` の再描画は、モーダルのスクロール位置を破壊する
プレイヤーには **「全員でスクロールが同期している」ように見える。**（実際は再描画で先頭に戻っているだけ）

### 4-4. 秘匿情報は「UIで隠す」だけでは不十分
**構造で漏らさない。** 見えなくすることは、操作できなくすることを意味しない。
**そして秘匿情報は「点数」から逆算される。** 派生データは終了時に再計算する。

### 4-5. try/catch でエラーを握りつぶさない
赤バナーで画面上部に出し、6秒で消す。**「反応しない」より遥かにマシ。**

### 4-6. 権限は「不正を防ぐため」に付ける。「進行を独占するため」ではない
| | 権限が必要か |
|---|---|
| 他人になりすます操作 | **必要** |
| 誰がやっても同じ結果になる操作（次へ進む・採点） | **不要** |

「次のラウンドへ」をホスト限定にすると、**ホスト離席で全員が進行不能**になり、**奉行問題の温床**になる。二重実行は `transaction` のガードで防ぐ。

### 4-7. 検証モードを、別ファイルにしない
ソロ / 対戦 / 観戦 は **同一ファイル内の「モード」** として実装し、ロジックを共有する。
**別ファイルにすると、片方を直すともう片方が古くなる二重管理地獄になる。**（実際に陥った）

## §6. モデルの使い分け

現行ラインナップ（上位から）: **Fable 5 / Opus 4.8 / Sonnet 5 / Haiku 4.5**

| 作業 | モデル |
|---|---|
| **コーディング・スクリプト・シム実装** | **Sonnet 5**（常用）。詰まったら **Opus 4.8** に上げる |
| **ルール設計・核の診断・「殺す」判断** | **Opus 4.8** ―― これは Claude Code ではなく、チャット側でやる |
| **確定済み仕様の実装・定型作業**（版上げ・TOC・config埋め込み） | **Sonnet 5** で十分 |
| 長丁場の一貫設計（大規模な新作を頭から通す等） | **Fable 5** を検討（コスト高。常用しない） |

> **モデルの切り替えは、人間がセレクタで行う。Claude 側から自動で振り分けることはできない。**
> 作業の性質が変わったら、**Claude の側から「ここは Sonnet 5 で十分です」と提案してよい。**

> **Claude Code は「作りたがる」。「作らない判断」は、このリポジトリの外（設計チャット）で下される。**
> **設計の是非を問われたら、実装せずに「これは設計側で決めるべき」と差し戻すこと。**

## §7-b. 「指示書」の受け取り方

設計判断は、このリポジトリの外（設計チャット・Opus）で下される。
その結果は **「指示書」** という形のテキストで渡される。構造は常に以下:

```
## 指示書 #N ― タイトル
【目的】       何のためにやるか（面白さのどこに効くか）
【背景】       この判断に至った経緯・却下した案
【やること】   具体的な作業。番号付き
【触ってはいけないもの】
【完了条件】   何が確認できたら終わりか（検証コマンドを含む）
【報告してほしいこと】  数値・気づき・懸念
```

**指示書に対する態度:**
- **【触ってはいけないもの】に踏み込まない。** 気になっても、指摘だけして手を出さない
- **【やること】に書かれていない改善を、勝手に足さない**（§0-4 対症療法カウンタ）
- **より良い案があれば、実装前に提案する**（§0-2）
- 【完了条件】を満たしたら、**【報告してほしいこと】をテキストで出力する。** それが設計チャットへの持ち帰り分になる

## §8. NPC・シミュレーションの限界（過信しないこと）

> **NPCシミュレーションは「バランス崩壊の検出」用。「面白いかどうか」の判定には使えない。**

このプロジェクトのゲームの核は **人間同士の疑心暗鬼**にある。機械的なNPCでは測れない。
**「数値が通った」を「成立した」と報告しないこと。**

⚠️ さらに: **「核が弱い」という検証結果が出たら、まず「核を実装したか」を疑う。**
（『影の賽』で、NPCが互いを一切見ていない＝核の1/3を実装しないまま「核が弱い」と結論しかけた）

## §9. エージェント運用の規律

Claude Code のような実行エージェントに作業を委譲するときの、構造的防御。
根拠と詳細は「実装技術知見 §7（Windows + Claude Code 開発環境の罠）」「プロセス検証知見 §14（検証と停止の原則）」。

### §9-a. インラインのシェル一発芸でファイルを加工しない
ファイル変更は Edit ツールで行うか、`tools/` にスクリプトを置いて実行する。
（理由：許可ルールで許可できず、レビューもできず、失敗も検出できない）

### §9-b. `cd X && git ...` を使わない。`git -C X ...` を使う
（理由：cd + git の複合コマンドは、どんな許可設定でも必ずプロンプトが出る）

### §9-c. 検証コマンドは「成功パスと失敗パスが必ず別の出力になる」ことを確認してから実行する
`try/catch` で異なる失敗要因を一括で握りつぶし、それを成功として報告しない。
状態を変える処理より先に、検証対象を削除しない。

### §9-d. 指示書に「中止条件」がある場合、その判定結果を報告するまで後続の作業に進まない
ファイル編集・一時ファイル作成・サーバ起動を含め、一切先行しない。

### §9-e. 認証・権限・外部設定に起因するエラーに当たったら、迂回実装を設計・提案しない
`configuration-not-found` / `operation-not-allowed` / `permission-denied` / `quota-exceeded` /
`not-enabled` / 401 / 403 等に当たった場合、即座に停止し、
「どのコンソールで何を有効化すれば解決するか」を報告する。
（理由：エージェントは自分の権限内で閉じる経路を優先するため、設定1クリックの問題に対して実装1本を作ってしまう）

### §9-f. この環境に Node.js / npm は存在しない
スクリプト（シミュレーション・ツール）は Python で書く。Node を前提にした提案をしない。

### §9-g. 指示書への報告は、チャットだけでなくファイルとしても作成する
【報告してほしいこと】（ブランチ・コミット・テスト結果・裁量事項・曖昧点・エラー記録等）は、
チャット出力に加えて **必ずMarkdownファイルとしても作成し、実装物と同じ場所にコミットする。**
命名例：`proto/報告_<作業名>_<版>_<日付>.md`。
（理由：チャット出力は設計チャット側への持ち帰りで再入力の手間が生じ、消える可能性もある。
ファイルなら次のセッション・別のAI・人間が後から参照できる。§0-5と同じ「書き戻し」の思想）

### §9-h. リポジトリへの手動操作は、必ずClaude Codeに伝える
`こたえあわせ` v0.7では、GitHub Web UIからの手動アップロード2回・手動削除1回が
Claude Codeに共有されないまま進行した。結果、「どのビルドが実際に公開されているか
Claude Code側からは判別できない」「（実際には未実施の）削除が完了済みという誤った
前提の指示書が発行される」という混乱が2回発生した（`git fetch`・GitHub API直接確認で
実際には未実施と判明し、実害は防げたが、確認コストが毎回発生した）。

> **原則：GitHub Web UI等からの手動操作（アップロード・削除・設定変更）を行った場合は、
> その旨をClaude Codeに必ず伝える。** ファイル操作は原則Claude Code側に一本化し、
> 人間による直接操作が必要な場合（緊急対応等）は、操作後に「何を・いつ・なぜ」を
> チャットで一言共有する。Claude Code側は、指示書に書かれた「〜済み」という前提を
> 鵜呑みにせず、`git fetch`・GitHub API等で独立に確認してから作業する（§9-e・
> `ボードゲーム制作_プロセス検証知見.md`§14-12と同じ「観測した記述を鵜呑みにしない」原則）。
