## 1. P0-1：スマホ再読み込みで入れないバグ

### 特定原因と該当仮説

**仮説A（匿名認証UIDの消失）が根本原因。** Firebase Emulator上で以下を実証した：

1. 部屋を作成し「playing」フェーズまで進める（席・好み・game状態をDB上に用意）。
2. あるプレイヤーの認証セッションを、**部屋メモ（localStorageの`witch_v13_room`）は残したまま**、
   匿名認証だけ新しいUIDへ差し替える（`signOut()`→`signInAnonymously()`）。
3. リロード → `restoreSavedRoom()`が`findSeatByUid()`で自席を見つけられず、
   「前回の部屋にあなたの席がないため、再接続できませんでした。」のエラーは正しく表示される。
4. **その状態から手動で「参加する」に合言コードを再入力しても**、`joinRoom()`が
   `state.meta.phase!=="lobby"`を理由に「その部屋は既にゲーム開始済みです。新しいUIDでは
   途中参加できません。」で弾く。

→ **自動復帰・手動再参加のどちらも成功しない、真の行き止まりであることを確認した。**
これは実装バグというより、**「開始後の新規UIDは参加を拒否する」というセキュリティ設計
（なりすまし防止）の裏面**として最初から存在していた制限で、`docs/魔女の調合_オンライン版_
引き継ぎ資料.md` §10に「開始後の新UIDは空席へ入れず、観戦だけが可能」として既に記載済みだった。
今回の作業は、この既知の制限が実際にどう発火するか・回復不能であることを実証した。

**仮説Aの場合の対応方針（指示書どおり）：修正せず報告のみ。** 設計チャットへ差し戻す。

### 影響しうるブラウザ環境（実機未検証・一般的に知られている条件からの推定）

- LINE / X(Twitter) / Instagram 等のアプリ内ブラウザ（WebViewの再生成でIndexedDBの
  認証セッションが引き継がれないことがある）
- iOSのプライベートブラウズ
- iOSのITP（Intelligent Tracking Prevention）によるスクリプト書き込み可能ストレージの
  分断・期限切れ

**注意**：実機（実際のスマートフォン・実際のアプリ内ブラウザ）では検証していない。
上記はEmulator上で「匿名認証セッションが失われて新UIDになる」状況を人為的に再現した結果からの
推定であり、報告された症状の実機ブラウザ・機種は不明のまま。

### 復帰導線の変更の選択肢（未実装・列挙のみ）

1. **「座席を引き継ぐ」フロー** ―― 合言コード＋名前の一致で、`online:false`が一定時間
   続いている席へ新UIDを差し替え、`privateRecipes`を旧UIDから新UIDへtransactionで
   移し替える。懸念：匿名認証には名前以外の本人確認手段がなく、同じ部屋にいる別プレイヤーが
   名前を知っていればなりすませる。
2. **ホスト媒介の復帰** ―― ホストが「席を空ける」操作をして新規参加者として入り直させる。
   懸念：`privateRecipes`はUID紐づけのため、素朴に空けると好みが失われる。移行ロジックが必要。
3. **現状維持（クライアント側の対策には限界がある）** ―― 既に`Persistence.LOCAL`
   （最も永続的な設定）を使用済み。ストレージ消失はブラウザ／OS側の挙動であり、
   アプリ側の実装だけでは防げない。

### 実装した最小修正（副次的に見つかった無言失敗経路）

机上レビュー中に、`restoreSavedRoom()`の`state.exists===false`（保存された部屋コードが
実際にはDB上に存在しない＝期限切れ・削除済み等）の分岐が、`forgetRoom()`だけ呼んで
`showError()`を呼ばずに終わっていた（無言でホーム画面へ）ことを発見。
Emulatorで存在しない部屋コードを指した状態から再現し、`showError()`追加後にエラーバナーが
正しく表示されることを確認した。

```diff
- if(!state.exists){ forgetRoom(); return; }
+ if(!state.exists){ forgetRoom(); showError("前回の部屋が見つかりませんでした（期限切れ、または削除された可能性があります）。"); return; }
```

## 2. P1-1：スクロール位置リセットの共通対策

### 列挙したスクロールコンテナ

`overflow-x:auto`/`overflow-y:auto`を持つCSSクラスを網羅的にgrepし、以下3種・9箇所を特定：

| コンテナ | data-scroll-id | 出現箇所 |
|---|---|---|
| `.bench`（素材列） | `"bench"` | 観戦・ソロ検証・NPCソロ・オンライン対戦（4箇所） |
| `.miniRow`（切り方プレビュー） | `"cutpreview"` | ソロ検証・NPCソロ・オンライン対戦（3箇所） |
| `.miniRow`（工房の記録・儀式別） | `` `mini-r${round}` ``（儀式番号で一意） | `recordPanel()`（全モード共通・1箇所） |
| `.miniRow`（名場面） | `` `highlight-${seat}` ``（席番号で一意） | `highlightPanel()`（全モード共通・1箇所） |
| `.overlay`（ルール説明モーダル） | ―（対象外） | `toggleHelp()`が`render()`を呼ばず`#helpLayer`のみ書き換えるため、元々このバグの対象外 |

### 適用した共通機構

`render()`内で、`app.innerHTML`を差し替える**直前**に`data-scroll-id`要素の
`scrollLeft`/`scrollTop`を`captureScrollPositions()`で保存し、差し替え**直後**に
`restoreScrollPositions()`で復元する。個別パッチではなく`render()`本体1箇所に集約した
ため、以後スクロール可能な要素を追加する際は`data-scroll-id`を付けるだけで自動的に保護される。

Emulator上で4人戦を模擬し、素材列を横スクロールした状態で他プレイヤー操作による再描画を
発生させ、スクロール位置が保持されることを確認した（`scrollWidth > clientWidth`の
オーバーフロー状態で`render()`を呼んでも`scrollLeft`が変化しないことをJS実行で確認）。

### P2-5：用語集の消失経緯

`git log --all -p -- witch.html`で全履歴を検索したが、`witch.html`に「用語集」という
独立セクションが存在した形跡は**過去に一度もなかった**。つまり「消えた」のではなく、
**そもそも実装されたことがなかった**。既存のツールチップ方式（`term()`/`T`オブジェクト）は
維持しつつ、遊び方モーダルの末尾に独立した「📚 用語集」セクション（坩堝・儀式・魔力・
調合師・好み・冴え・秘伝の7語）を新設した。

### P2-8：観戦終了後表示の検証

Emulator上に、`meta.phase="ended"`・`game.phase="end"`・`reveal`（席番号キー）を
直接投入した部屋を作り、別セッションから観戦した。結果：**全好み・個人点・工房の記録の
冴え%・名場面がすべて正しく表示され、コード修正は不要だった。**

調査中に見つけたのは、検証用シードスクリプト側の誤り（`reveal`をUID文字列でキーづけして
しまい、`spectatorHTML()`が期待する席番号キーと食い違っていた）であり、`witch.html`本体の
バグではない。修正後、正しいデータ形状で再検証して問題なしを確認した。

## 4. R-1：素材別 好み/忌避 カバレッジ集計（報告のみ・変更なし）

V3プール18枚について、素材6種が「入れたい側（pair/cnt2/solo）」「入れたくない側
（abs/abs2）」に何回登場するかを機械的に集計した（`POOL`/`HARD`定義から算出）。

| 素材 | 入れたい側（pair+cnt2+solo） | 入れたくない側（abs+abs2） |
|---|---|---|
| 🌙満月草 | 5 | 0 |
| 🔥火竜の鱗 | 4 | 1 |
| 🍄影茸 | 3 | 2 |
| ✨星屑 | 4 | 1 |
| ❄️霜結晶 | 5 | 0 |
| 🐍蛇毒 | 3 | 2 |

秘伝枠（HARD6枚）を含めると、「集める」役・「避ける」役とも各素材ちょうど1回ずつで均等
（HARDは6種の素材それぞれについて「Xを集めYを避ける」を1枚ずつ用意する設計のため）。

**V3プール本体だけを見ると、🌙満月草と❄️霜結晶は「入れたい」役にしか登場せず「入れたくない」が
0回。逆に🍄影茸と🐍蛇毒は「入れたくない」役が「入れたい」役の半分近くまで比率が上がる**、
という非対称がある。UIへの表示は行わず、採点ロジック・好みプールの構成も変更していない。

## 5. 実装中に気づいた懸念・逸脱・未解決事項

- **P0-1の根本原因（UID消失時の復帰導線）は未解決のまま。** §1の3案はいずれも設計判断
  （なりすましリスクの許容度・`privateRecipes`移行ロジックの是非）が必要で、実装していない。
- **P2-1（コピーボタン）の実クリップボード成功は、この作業環境では確認できなかった。**
  検証に使ったブラウザ自動化ツール（Claude Codeの内蔵Browser pane）が、ユーザー操作由来の
  クリックでも`navigator.clipboard.writeText()`を`NotAllowedError`で拒否し続けた。
  Permissions APIは`"granted"`を返すため、これは自動化環境固有の制約である可能性が高いが、
  **実際のスマートフォン・PCブラウザでの成功は未確認**。次回人間プレイテストで実際に
  コピーが機能するか確認してほしい。
- **この環境にはNode.js/npm/Pythonが実在した**（`CLAUDE.md`§9-fの記載と矛盾）。
  `package.json`のdevDependencies（`firebase`/`firebase-tools`/`@firebase/rules-unit-testing`）が
  未インストールだったため`npm install`を実行し、Emulatorでの実地検証を行った。
  `CLAUDE.md`§9-f・§7-wに、確認済みの事実として追記済み（環境が変わりうる可能性への
  注意書きも添えた）。
- **検証用に一時的に`firebase-admin`を`--no-save`でインストールした**（Auth Emulatorの
  カスタムトークン発行に使用）。`package.json`・`package-lock.json`には記録されておらず、
  `node_modules`は`.gitignore`対象のためコミットには含まれない。検証用の一時スクリプト
  （`_seed_witch_game.mjs`等）は作業完了後に削除し、コミットしていない。
- `package-lock.json`に1件の差分がある（`google-logging-utils`という`extraneous`扱いの
  孤立エントリの削除）。これは`npm install`の副作用であり、今回のdevDependencies導入とは
  無関係の軽微なクリーンアップ。
