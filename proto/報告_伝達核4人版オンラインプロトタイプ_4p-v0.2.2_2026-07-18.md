# 報告：伝達核 4人版 4p-v0.2.2 差分 — 参加フローの席割当バグ修正

**対象指示書:** 指示書：伝達核 4人版 4p-v0.2.2 差分 — 参加フローの席割当バグ修正 — v1（2026-07-18 design chat発行）
**前提VER:** `4p-v0.2.1`（コミット`1dfc38e`）
**成果物VER:** `4p-v0.2.2`

---

## 1. 症状の再確認

指示書§1の症状（本番Firebase・ホスト着席済みの部屋へ参加者が着席しようとしてpermission_deniedで失敗）を、
コードレベルで裏付けた。

## 2. 原因の確定

**指示書§2の仮説どおりで確定。** `joinRoom()`は、参加者の画面（Home画面）に`seats`を購読する
`.on('value')`リスナーが一切存在しない状態（`attach()`は着席確定後にしか呼ばれない）のまま、
`seats/${i}`に対して直接`.transaction(cur => cur || {...})`を実行していた。

Firebase JS SDKの`transaction()`は、対象パスにローカルの同期済みキャッシュが無い場合、初回試行を
`cur=null`という仮定で計算し、その結果を条件付き書き込みとしてサーバーへ送る。今回のケースでは
サーバー側の実データ（ホスト占有）に対して安全ルールが**恒久的に**拒否する条件（既存データのuidが
書き込み者と一致しない）であるため、これは「値の食い違いだから最新値を取り直して再試行」という
Firebase標準の楽観的リトライでは救えない、**ルール起因の確定拒否**であり、`transaction()`のPromiseが
`permission_denied`で reject される。コードにはこの1回の拒否から次の候補席へ進むリトライ構造が無く、
呼び出し元の`safe()`ラッパーがこの例外をそのままエラーバナーへ表示していた。

### エミュレータE2E（v0報告TEST 1）で3人の参加が通った理由（実機で検証）

design chatの推定（「ゼロ遅延でseatsが同期済みだった」）を鵜呑みにせず、**実際に検証した**。
修正前（コミット`1dfc38e`時点）の`transmitcore_4p_v0.html`をそのまま使い、ホスト着席済みの部屋へ
参加者がページ読み込み直後（`seats`購読ゼロの状態）で参加する、同一のE2Eシナリオをローカル
Firebase Emulatorに対して再実行したところ、**修正前のコードでも参加者は正しく席1へ着席し、
permission_deniedは一切再現しなかった。**

これは「テストの実行順序や待機時間の問題」ではなく、**Firebase Database EmulatorとFirebase本番の
`transaction()`実装差そのもの**である可能性が高いと判断する（エミュレータは、ローカルキャッシュの
有無によらず常にその場の最新値を起点に評価しており、本番のような「ローカル未同期時はnull前提で
一度書き込みを試み、サーバーから拒否/訂正を受けて初めて実データを知る」という非同期の往復が発生しない、
または発生してもルール拒否ではなく値訂正として扱われる、等の実装差が考えられる）。
断定はできないが、**エミュレータでの合格は本番での合格を保証しない、という具体的な実例**として
`実装技術知見.md`への一般化を推奨する。

## 3. 修正内容（実装済み）

1. 参加候補ごとに、**transactionの前に必ず`seats`を`.get()`でサーバーから読み直し**、その時点で
   未試行かつ空いている最初の席を選ぶ（純関数`pickNextOpenSeat(seatsSnapshot, candidates, tried)`として
   切り出し、単体テスト可能にした）
2. transactionが不成立（`committed:false`）、または回復可能なpermission_denied（`isRecoverableSeatClaimError`
   で判定）で例外が投げられた場合は、**エラーバナーを出さずに**次の未試行席で再読み取り→再試行する
   （最大4席分。`for`ループの上限を候補数に一致させ、既に試した席は`tried`セットでスキップ）
3. 想定外の種類のエラー（permission_denied以外）は従来どおり呼び出し元へ伝播し、エラーバナー表示は維持
4. 全席が本当に埋まっている場合のみ「その部屋には空席がありません」を投げる（従来から変更なし）

**同型バグの点検（指示書§3-3）：** リポジトリ内で`.transaction()`を使っている箇所は
`joinRoom()`のこの1箇所のみ（`grep -n "\.transaction("`で確認）。他の共有ノードへの書き込みは、
(a) ホストが部屋作成直後に行う初回`seats`書き込み（衝突相手が存在しないため無関係）、
(b) 自分の既存席への`lastSeen`ハートビート更新（`.update()`、自己書込みのみ許可されるため衝突しない）、
(c) 常時同期済みの`.on('value')`購読から得た`ST.seats`の参照読み取り（`hostPerformInitialDeal`等）、
のいずれかであり、**同型の「未同期キャッシュ起因の書き込み」は他に見つからなかった。**

## 4. テスト追加

- **単体：`pickNextOpenSeat`**（`tests/transmit4p_static.test.mjs`に追加）。同期済みスナップショットから
  未試行の最初の空席を返す／全席占有または全候補試行済みでnull（満席）を返す、の5ケースを検証
- **E2E再現/回帰テスト**（`test_4p_seatrace.js`、ローカルFirebase Emulator + Playwright、スクラッチのみ・
  リポジトリ非コミット。既存の生Playwright検証と同じ位置づけ）：ホストが席0に着席済みの部屋へ、
  参加者がページ読み込み直後・`seats`購読ゼロの状態で参加 → **修正後のコードで**エラーバナーを出さず
  席1へ着席することを確認。続けて2人目の参加者も席2へ着席することを確認。
  修正前コードでも（§2の通り）このシナリオ自体はエミュレータ上で再現しなかったため、この意味では
  「バグ再現」ではなく「修正後コードの回帰確認」として機能した
- 既存の静的テスト：**17/17 全通過**（既存16件 + 本差分で追加した1件）
- ルールテスト：**11/11 全通過**（ルール自体は無変更。回帰なし）
