## 4. 指示書が個別に要求する報告事項

### 4-1. 工程0-a：開始ボタン不活性化の全条件（データ来歴つき）

開始ボタンの`disabled`属性は、**唯一の条件式** `present >= 3`（[ui_template.html:1218](proto/kotaeawase/ui_template.html)）で決まる。想定していたほど「散らばって」はいなかった――**指示書の停止条件「0-aの条件が散らばりすぎて列挙不能」には該当しない。**

`present`の来歴：
```
present = this.seatsPresent()
seatsPresent() = PLAYERS.filter(p => s[p] && pr[s[p].uid] === true).length
  s  = this.roomInfo.seats    ← RTDB `rooms/{code}/seats` を watchRoom() 経由で購読・memory反映
  pr = this.roomInfo.presence ← RTDB `rooms/{code}/presence` を watchRoom() 経由で購読・memory反映
```
つまり `present` は **RTDBの2つの独立したノード（`seats`と`presence`）を掛け合わせた値**であり、localStorageは一切関与しない。GMがリロードしても、RTDB購読が復帰すればこの値も復帰する。

もう1つ、ボタンの見た目は活性でも**実際には何も起きない**別経路がある。`startGame()`冒頭のガード：
```js
if (this._starting || this.host) { ...; return; }
```
これは画面には一切現れない**メモリ上のみの状態**（`this._starting`／`this.host`）。ボタンを連打した場合や、何らかの理由で`this.host`が既にセットされている場合、ボタンは押せてもクリックが黙って無視される。これは「不活性（disabled）」ではなく「活性だが無反応」という別種の症状であり、design chatが想定していた「不活性の理由」の範囲外だが、GMから見た体感としては同じ「押しても始まらない」に見えるため、0.5-bの診断ログでは`startGame:noop`イベントとして別途記録し、この経路も後から判別できるようにした。

なお、design chatが例示していたUI文言「4/4」は実装と一致しない。GM自身はカウントに含まれないため、実際の文言は「X/3」（プレイヤー3人のみ）。

### 4-2. 工程0-b：着席データフローの追跡

`Online.doClaim(seat)`（UI側）と`net.js`の`claimSeat()`は**別実装**で、意味も異なる（重複・不整合の疑いとして記録するのみ、今回は統合しない＝指示書の対象外）：
- `doClaim()`：席の占有者がオンラインなら拒否、オフラインなら奪える（実際にオンラインで使われている経路）
- `claimSeat()`（`net.js`）：uidが不一致なら常に拒否。presence を見ない

`watchRoom(db, code, cb)`（[net.js:331](proto/kotaeawase/net.js)）は`seats`・`presence`・`meta`に対して**3本の独立した`on('value')`リスナー**を張っており、コールバックはそれぞれ単独で発火する。**`seats`の更新と`presence`の更新が同時に観測される保証はない。**

`doClaim()`は`seats`への書き込みを`await`するが、その直後に呼ぶ`keepPresence()`は**内部のpresence書き込みをawaitしない**（fire-and-forget）。これはコード上明確なギャップとして確認した。

席が「クレーム後に自然に空へ戻る」経路は**コードベース全体を探しても存在しない**（「退席」関数が無い）。したがって実際のリスクは「席が勝手に空く」ではなく「**席は埋まっているのに presence が false/欠落のまま固着する**」という非対称な壊れ方である。

### 4-3. 工程0-c：GMホスト状態のライフサイクル

`kotaeawase:host:{CODE}`（localStorage）は、**`RoomHost.publish()`の中でのみ**書き込まれる（[net.js:219](proto/kotaeawase/net.js)）。そして`RoomHost`インスタンス自体が存在するのは`startGame()`が成功した後だけ。

つまり：**待機ロビーの全期間、GMのホスト状態はどこにも保存されていない。** これは「症状3（GM復帰時に保存データが見つからない）」を、症状1（GMが`wait`のまま`startGame()`を一度も成功させられていない）の**自然な下流結果として完全に説明する**。2つは独立したバグではなく、1つの根本原因（症状1）から生じる1本の因果関係だった。

### 4-4. 工程0-d：Node.jsによる再現の試行

MockDBを使い、以下4シナリオを試みた。

1. **4人ほぼ同時クレーム**：**再現せず**。MockDBの`.set()`/`.get()`が同期的に近い挙動をするため、着席カウントの一時的な食い違いは発生したが、次のイベントループティックで自己修復し、永続的な不整合は残らなかった。
2. **クレーム→退席→再クレーム**：退席機能自体が存在しないため、このシナリオは文字通りには再現不能（設計上、退席という状態遷移が無い）。
3. **クレーム直後のリロード**：**再現せず**。`doClaim()`の`await`後に`keepPresence()`が呼ばれる順序は保たれており、MockDB環境では presence 書き込みが失敗する要因が無いため、正常に完了した。
4. **GMが3人揃う前にstartGame()を試行**：これは**バグではなく設計通りの動作**であることを確認した（ボタンがdisabledのため、そもそもクリックイベントが発火しない）。

上記1〜4はいずれも「バグを再現できた」とは言えない。**指示書の停止条件「0-dが再現し、原因が特定できたら停止する」には該当しないため、本指示書の範囲（工程0.5/1/2）へ進んだ。**

一方、**再現ではなく構造的な実証**として、`keepPresence()`のpresence書き込みを人為的に失敗させる（実際のネットワーク瞬断を模した）改造版MockDBを用意したところ、`present`カウントが2のまま固着し、自己修復しない状態を作れた。`keepPresence()`の唯一の再試行経路は`.info/connected`の状態**変化**イベントであり、接続がそもそも「切れた」と検知されないまま留まるケース（例：モバイル回線のバックグラウンド遷移、Wi-Fi切り替えの遅延）では、このイベントが発火しない可能性がある。これは実際のインシデントの**再現ではなく**、機構として起こりうることの実証である。

### 4-5. 仮説との整合性

主仮説（着席同期の不整合）・副仮説（GM権威状態の不在）のどちらとより整合するか：

- **副仮説は独立に真であることが確定した**（0-cの通り）。ただしこれは症状3のみを説明し、症状1（開始できない）は説明しない。design chat自身の見立て通り。
- **主仮説の方が症状1に近いが、design chatの想定より一段階具体的な機構**を特定できた：「着席同期がランダムに乱れる」ではなく、「**presence書き込みが（fire-and-forgetのため）失敗・遅延したまま、再試行の唯一の経路（`.info/connected`の変化イベント）が発火しないケースがある**」という、より狭く特定可能な形。
- どちらの仮説も「否定」はされておらず、**両方とも真だが説明範囲が異なる**、というのが今回の調査の結論。

### 4-6. 診断ログのイベント種別一覧（工程0.5-b）

`DiagLog`（[ui_template.html](proto/kotaeawase/ui_template.html)、上限400件のリングバッファ）が記録する12種：

## 5. 停止しなかった自己判断

1. **Node.jsのインストール**：このPCにNode.jsが存在せず、既存テストスイートを実行できなかった。CLAUDE.md §9-eの「認証・権限・外部設定に起因するエラーには迂回実装をしない」に従い、Pythonでのテスト書き直し等の回避策は取らず、チャットで人間に状況を説明し、`winget install OpenJS.NodeJS.LTS`の実行許可を得てから導入した。システム変更を伴う行為のため、実行前に明示的な承認を取った。
2. **`startButton:judge`イベントの重複排除（`_lastWaitLogKey`）**：`renderWait()`は再描画のたびに呼ばれるため、無条件に`DiagLog.push`すると同一状態のログで400件のリングバッファがすぐ埋まり、直前の重要なイベント（`seat:claim`等）が押し出されてしまう。状態（`present`と`reasons`の組）が変化した時だけ記録するようガードを追加した。指示書に明記の無い判断だが、「診断ログの実用性を保つ」という0.5-bの目的に照らして必要と判断した。
3. **「前回の部屋に戻る」の実装方式**：プレイヤー用の専用復帰フロー（GMの`gmResumeCode`画面に相当するもの）を新設せず、既存の`renderJoinForm()`（参加コード入力画面）に前回コードを事前入力する方式にした。理由：プレイヤーには「復帰」と「初回参加」で区別すべき状態の違いが無く（`doClaim()`が既存のオフライン席の引き継ぎに既に対応済み）、専用画面を増やすと対症療法的な複雑化になる（§0-4）。既存の1画面を強化する方が構造として単純。
4. **`copyCode()`のエラー処理**：`navigator.clipboard.writeText()`が失敗した場合（権限拒否・非HTTPS等）、`exportDiagLog()`のようなファイルダウンロードへのフォールバックは実装しなかった。部屋コードは画面に常時表示されているため、コピーに失敗しても目視で読めば済み、フォールバックを足すほどの必要性がないと判断した（対症療法カウンタ§0-4を1つ節約）。
5. **CRLF正規化**：`build.py`実行直後の`kotaeawase.html`にCRLF（2,598箇所）が混入していた。既存の`kotaeawase.html`はLFであり`.gitattributes`にも例外指定が無いため、リポジトリの既存慣習に合わせてLFへ正規化した。原因（新規インストールしたNode.js側の改行コード扱いの違いの可能性）は未調査。
