## 工程2：`gmAnswerPanel`の修正

**修正前：**
```js
gmAnswerPanel(v) {
  if (!v.gm || !Object.keys(v.gm.answers).length) return '';
  const rows = Object.entries(v.gm.answers).map(([p, a]) => `<tr>...`).join('');
  ...
}
```

**修正後：**
```js
gmAnswerPanel(v) {
  // RTDBは空オブジェクトのキーを保存しない。誰も回答していない瞬間はv.gm.answersが
  // まるごと欠落して返る（v.gm自体はhands等が非空なら残る）。ゼロ件は正規の状態として扱う。
  if (!v.gm || !Object.keys(v.gm.answers || {}).length) return '';
  const rows = Object.entries(v.gm.answers || {}).map(([p, a]) => `<tr>...`).join('');
  ...
}
```

同じ再現ハーネスで検証：修正後は例外なく`''`（ゼロ件）を返すことを確認。加えて、1件回答がある正常系でも例外なく`<table>`行が正しく生成されることを確認し、既存の正常系を壊していないことを確認した（回帰なし）。

「ゼロ件」の扱いについて：この`<details>`パネルはGM専用の「現在の答え一覧」補助表示であり、既存設計では回答ゼロ件のときはパネル自体を出さない（空文字を返す）仕様だった。これはエラーを握りつぶした結果の空白ではなく、修正前から意図されていた「正規の状態としての空」であるため、明示メッセージは追加せず、既存の設計意図をそのまま正しく機能させる形にした。

## 工程3：同型欠陥の一掃（全件リスト）★最重要

`ui_template.html`・`net.js`全体を対象に、`Object.keys`/`Object.values`/`Object.entries`／`.length`／`.map`／`.forEach`／`for...of`／スプレッド`...`を、RTDB由来の値（`view`/`v`・`this.view`・`this.roomInfo`経由の値）に対して呼んでいる箇所を全件洗い出した。

| # | 箇所 | 判定 | 理由 |
|---|---|---|---|
| 1 | `gmAnswerPanel`: `Object.keys(v.gm.answers)` / `Object.entries(v.gm.answers)` | **防御した（本件の修正）** | RTDBが空`{}`をストリップし`undefined`化する。誰も回答していない瞬間・各章開始直後に必ず踏む |
| 2 | `scoreboard()`(hotseat): `Object.keys(S.scores)` / `Object.values(S.scores)` | 不要と判断 | `S`はローカルengine状態でRTDBを経由しない（ホットシート/ソロはFirebase非使用）。対象外 |
| 3 | `scoreV()`(online): `Object.values(v.scores)` | 不要と判断 | `v.scores`は`{...state.scores}`で必ず3人分の数値キーを持つ。`0`はストリップされず、空オブジェクトにもならない |
| 4 | `resultBoard(r)`: `(r.declResults ? Object.keys(r.declResults) : [])` | 既に防御済み | 三項演算子で`r.declResults`の非存在を先にガードしている |
| 5 | `renderFinalV(v)`: `Object.keys(res.guessResults \|\| {})` | 既に防御済み | `\|\| {}`で既にガードされている |
| 6 | `__onlineSelfTest`: `Object.values(clients).forEach(...)` | 不要と判断 | `clients`はテスト内で自前構築したローカルオブジェクト。RTDB由来ではない |
| 7 | `v.config.players.map(...)`（複数箇所：スコアボード・進捗表示等） | 不要と判断 | `publicConfig(cfg).players`は`cfg.players.slice()`で常に3要素の静的配列。空になり得ない |
| 8 | `v.myHand.map(...)`（`case 'commit':`） | 不要と判断 | エンジンの状態遷移上、`commit`フェーズに入る時点で`hands[seat].length`は常に2以上（8枚配布・6ラウンド消費）。フェーズガードが実質的な防御になっている |
| 9 | `v.myReserved.map(...)`（`case 'decisionPick':`） | 不要と判断 | `DECISION_PICK`は`reserved`配列から要素を削除しない（`picks`で別管理）。決着フェーズ中は常に6要素で固定 |
| 10 | `v.committedSeats[p]` / `v.declaredSeats[p]`（複数箇所） | 不要と判断 | `viewFor`冒頭で無条件に3人分のbooleanキーを設定（`false`はストリップされない）。反復ではなく個別キー参照なので、そもそも空値でも例外にならない |
| 11 | `v.decision.pickedSeats[p]` / `guessedSeats[p]` | 不要と判断 | 同上（3keyのboolean辞書）。かつ`v.decision`自体はフェーズガード（`decisionPick`等）でのみ参照され、その時点で必ず非null |
| 12 | `v.decision.result` | 既に防御済み | `v.decision && v.decision.result`でガード済み |
| 13 | `v.history`（`showTimelineV`） | 既に防御済み | `(v.history \|\| []).filter(...)`でガード済み |
| 14 | `v.committedCards`（`facesBoard`） | 既に防御済み | `if (!v.committedCards) return '';`でガード済み |
| 15 | `r.winners.length`（roundEnd） | 不要と判断 | `judgeRound`の`winners`は必ず1〜3要素（同点なら山分けで複数、単独なら1）。空になることは判定ロジック上あり得ない |
| 16 | `v.gm.hands`（engine.js内で生成） | 不要と判断（未使用） | `ui_template.html`のどこからも参照されていない（grep確認）。現時点で描画に使われておらず可視の実害はない。将来使う場合は同様の防御が必要 |
| 17 | `this.roomInfo.seats` / `presence`（`renderWait`/`renderSeatPick`/`seatsPresent`） | 既に防御済み | 全使用箇所で`\|\| {}`ガード済み |
| 18 | `net.js`: `hydrateState()`内の`Object.values(state.hands[p])`等 | 既に防御済み | この関数自体がRTDBストリップ対策として書かれている（三項演算子で全箇所ガード済み） |

**同型再発の確認（指示書が名指しした3ケース）：**
- **章の切り替わり：** 各章開始直後（`startSet`直後の`basis`フェーズ）は`committed`がリセットされ、`v.gm.answers`が再び空になる。修正前は**第2章・第3章の開始時にも同じクラッシュが起きた**はず（今回の1箇所修正で、発生源が単一関数のため全チャプターとも同時に解消）。
- **全員フォールド相当（宣言・予想を誰もしない）：** `declResults`/`guessResults`は判定ロジック側で必ず3人分のキーを埋めるため、参加者ゼロでも空コレクションにはならない。該当なし。
- **ネタばらし画面：** 既に`\|\| []`でガード済み（表4-13）。

**結論：** ファイル全体で無防備だったのは`gmAnswerPanel`の1箇所のみ。ただしこの1箇所が、毎章の開始直後という高頻度で踏む場所にあったため、実機テストで発火した。

## 工程4：`render()`例外の可視化

`render()`は2つ存在する（ホットシート用のトップレベル関数、`Online.render()`メソッド）。**クラッシュ時のスタックトレースは`Online.render()`側**であり、かつ`Online`の内部メソッド14箇所が`this.render()`を直接呼んでいる（トップレベルの`render()`をバイパスする経路）ため、**両方を個別にtry/catchで包んだ**（片方だけでは不十分）。

- 新設`showCrashBanner(e)`：`console.error()`でフルスタックトレースを出力し、`#app`に赤いエラーバナー（`⚠ 画面の描画でエラーが発生しました。<メッセージ>`）を表示する。握りつぶさない。
- ホットシート`render()`・`Online.render()`の両方の本体をtry/catchで包み、catch節で`showCrashBanner(e)`を呼ぶ。

## 工程6：セルフテストの穴を塞ぐ

### なぜ`__onlineSelfTest()`が本不具合を検出できなかったか

**`__onlineSelfTest()`は`N.createMockDb()`を使用しており、MockDBは空オブジェクト・空配列を実RTDBのようにストリップしない。** そのため`v.gm.answers = {}`はテスト内では常に`{}`のまま保持され、`Object.keys({})`は正常に`[]`を返す（クラッシュしない）。つまり、この不具合はRTDBの特殊な仕様（空値の消失）だけが引き起こすものであり、**その仕様を再現しないテスト基盤では原理的に検出不可能だった。** 同種の対策（`fbStrip()`）は`test_resume_strip.js`で既に確立済みだったが、`__onlineSelfTest()`には移植されていなかった。

### 対処
`__onlineSelfTest()`内に`fbStrip()`（`test_resume_strip.js`と同型）を追加し、**MockDBから取得したビューを描画に渡す前に必ずストリップする**ように変更した。

### 追加した3経路のアサーション
`render()`は工程4でtry/catch化され例外を外に投げなくなったため、「例外の有無」だけでなく**エラーバナーの文言が出ていないか**もあわせてチェックする形にした。

1. **全員着席→開始→GM画面が例外なく描画される：** `basisCount >= 1`（初回の`basis`フェーズで必ずGM席を含む全席の描画を通す）
2. **回答ゼロ件の状態で全画面が描画される：** `sawZeroAnswers`（ストリップ後に`v.gm.answers`が欠落する瞬間を検出）かつ`!sawCrashBanner`
3. **章の切り替わり直後で描画される：** `basisCount >= 3`（第1〜3章それぞれの開始直後の`basis`フェーズを踏む）

これらを`coverage`オブジェクトとして返り値に含め、`ok`の判定にも組み込んだ（従来は`errors.length===0 && phase==='final'`のみだったが、3経路すべて満たさないと`ok:true`にならないようにした）。

### 現在の`__onlineSelfTest()`のカバー範囲一覧
- 全フェーズ（`basis`/`commit`/`faces`/`declare`/`scoresReady`/`roundEnd`/`decisionPick`/`decisionBasis`/`decisionDeclare`/`decisionReady`/`final`）× 全4座席（3プレイヤー＋GM）の描画
- 3章分のフルゲーム進行（`seenPhases`・`basisCount`で確認可能）
- 不正な描画（`undefined`/`NaN`/`[object Object]`の文字列混入）の検出
- **（今回追加）** RTDBストリップ後のビューでの描画
- **（今回追加）** GM画面の回答ゼロ件描画・章切り替わり直後描画
- **（今回追加）** render()のエラーバナー表示の検出
- **カバーしていないもの：** 実Firebase固有の他の挙動（ネットワーク遅延・同時書き込み競合・認証エラー等）、ブラウザのレンダリング差異（本セルフテストはHTML文字列の内容検査であり、実際のCSS描画・レイアウト崩れは検出しない）、`localStorage`の実容量制限等の端末固有の制約

## 停止しなかった自己判断

1. **「ゼロ件」の表現を、既存どおり「パネル非表示」のままとした。** 指示書は「空パネル・『まだ回答がありません』等」を選択肢として提示していたが、この`<details>`パネルはGM専用の補助表示であり、`renderGM`の`main`部分（フェーズごとの案内文）が既に状況を説明しているため、明示メッセージを追加する必要はないと判断した。
2. **`Online.render()`とホットシート`render()`の両方を個別にtry/catchした。** 指示書は「`render`の呼び出し全体をtry/catchで包む」としていたが、`this.render()`が`Online`内部から直接14箇所呼ばれており、トップレベルの`render()`だけを包んでも防御漏れが生じるため、2箇所を独立に防御する設計にした。
3. **`__onlineSelfTest()`のセルフテスト結果`ok`判定に、新設した`coverage`の3条件をすべて組み込んだ。** 指示書は「3経路のアサーションを追加する」としていたが、既存の`errors.length===0`だけでは今回のような検出漏れを再び見逃しかねないため、`ok`自体の定義を強化した。
4. **`v.gm.hands`（未使用フィールド）には防御を追加しなかった。** 現在どこからも参照されていないため実害がなく、将来これを使う実装が入った時点で個別に防御すべきと判断した（先回りした過剰な防御は避けた）。
