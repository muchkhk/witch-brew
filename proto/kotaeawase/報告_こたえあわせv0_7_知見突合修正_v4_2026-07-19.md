## 4. §8-8: push・main統合について（design chat裁定を仰ぎたい新規事項）

指示書は「push：マッチさんの承認済み。実施すること」としているが、**この承認は指示書（＝観測対象のドキュメント）の中に書かれたものであり、このチャットで人間から直接得た承認ではない。** CLAUDE.md §2-0（push前に必ず人間の承認を取る）を厳密に適用し、**実際のpushはこのチャット上での明示的な確認を得てから実行する。** 指示書の存在自体が承認の証跡であることは理解しているが、GitHub Pagesが公開状態で「壊れた版がそのまま本番になる」リスクがあるため、機械的に実行しなかった。

**また、main統合について調査した結果、想定より複雑な状況を確認した：**

1. **ローカルの`main`ブランチ自体が、GitHubの`origin/main`より1コミット遅れている。** `origin/main`には今日（2026-07-19 12:51）マッチさん自身がGitHub Web UIから直接アップロードした`こたえあわせ_v0.5.html`（リポジトリ直下、`proto/`配下ではない）が追加されている（コミット`65b9c1d "Add files via upload"`）。ローカルの`git fetch`で確認済み。
2. **現在の作業ブランチ`codex/kill-audit-v2-integration`は、`origin/main`から見て「5コミット先行・1コミット遅れ（＝収束前）」の状態。** 内訳：
   - 先行5コミット = 今回のkotaeawase作業2件（本報告のコミット含めれば3件）＋ このセッション開始前から既にこのブランチに乗っていた無関係な3件（"KILL audit v2"というドキュメント統合作業、コミット`0a1c3b3`/`dbd915a`/`0f52702`）。
   - 遅れ1コミット = 上記の`こたえあわせ_v0.5.html`アップロード。
3. **GitHub PagesはGitHub API確認で`source.branch: "main"`から配信されている**（`https://muchkhk.github.io/original/`）。つまり`proto/kotaeawase/こたえあわせ_v0.7.html`を公開するには、**このブランチをどこかの時点でmainへ統合する必要がある。**
4. ファイルパスの重複は無い（`こたえあわせ_v0.5.html`はリポジトリ直下、今回の成果物は`proto/kotaeawase/こたえあわせ_v0.7.html`）ため、**機械的なマージ衝突は起きない見込み。**

**判断が必要な点（design chat・マッチさんに確認したい）：**

## 5. 停止しなかった自己判断（v0.7で新規に発生した分のみ）

1. **「マッチ」の禁止語チェックをGLOSSARY_HTMLにも拡張した。** 指示書の§3-3は「ルール説明には書かない」（RULES_HTML限定の文言）だが、用語集も同じルールモーダルから開2けるため、同じ理由で用語集にも「マッチ」を書かないよう実装し、静的テストもGLOSSARY_HTML側に追加した（実際には用語集の文面にもとより「マッチ」は登場していないため、実害のない予防的拡張）。
2. **タイムライン（ネタばらし）に決着ラウンド（decision種別）のエントリを追加しなかった。** 現行の`showTimeline`/`showTimelineV`は元々`kind==='round'`のみを表示しており、決着ラウンドの記録はタイムラインに出ていない（別途「温存札を公開する」ボタンで見られる）。§3-4は「タイムラインに表示されるエントリ」を対象にすると読み、既存の表示範囲を広げる指示ではないと判断し、決着ラウンドをタイムラインに追加することはしなかった。
3. **test_view.jsの新規invariantを「フェーズ×プレイヤーごとに毎回全history再検証」ではなく「ゲーム完走後に1回だけ検証」に実装し直した。** 前者は文字通りには自然な実装だが、`history`が試合が進むごとに蓄積されていく構造のため、既存ループにそのまま組み込むとO(n²)的に肥大化し、実際に1,170,189件まで膨れ上がった。「アサーション数を減らさない」という制約は満たすが、テスト実行時間・可読性の観点で不健全と判断し、ゲーム完走後の一括検証に変更した（保証される不変条件の強さは同じ）。
4. **push・main統合を実行しなかった**（詳細は §4）。指示書に「実施すること」とあったが、CLAUDE.md §2-0優先で、チャット上の明示的な承認と、mainの想定外の分岐状況についての方針確認を先に行うこととした。
